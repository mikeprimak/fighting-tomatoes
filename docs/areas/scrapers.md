## Backfill (retroactive results) architecture (2026-04-28)
All 5 native live parsers (`ufcLiveParser`, `bkfcLiveParser`, `oneFCLiveParser`, `oktagonLiveParser`, `matchroomLiveParser`) accept an optional third `BackfillOptions` parameter (defined in `liveTrackerConfig.ts`). Defaults preserve existing live-tracker behavior; the backfill orchestrator (`scripts/backfillResults.ts`) opts in. Flags:
- `nullOnlyResults` — only write winner/method/round/time when DB value is NULL.
- `skipCancellationCheck` — bypass the CANCELLED↔UPCOMING reconciliation pass.
- `skipNotifications` — suppress next-fight push notifications.
- `skipStaleLiveReset` — bypass live-tracker self-healing status downgrades (BKFC's stale-LIVE pass, Oktagon's lifecycle-completed-no-winner reset).
- `completionMethodOverride` — when set (e.g. `'backfill-ufc'`), stamp `Fight.completionMethod` + `Fight.completedAt = now()` on COMPLETED transitions.

Per-org backfill wrappers live alongside each parser as `services/backfill<Org>Results.ts`. Each one re-uses the existing live scraper + parser (single source of truth per org). Wrapper invocation styles vary:
- **UFC, BKFC** spawn the JS scraper as a child node process and read its JSON output (existing pattern from the live trackers — `scrapeLiveEvent.js`, `scrapeBKFCLiveEvent.js`).
- **ONE FC, Oktagon, Matchroom** instantiate the TS scraper class in-process (`new XxxLiveScraper(input).scrape()`).
- **Oktagon takes a slug, not a URL** — extracted from `event.ufcUrl` via `events/([^/?]+)` regex; the wrapper carries this same extraction.

The orchestrator runs daily at 10:00 UTC via `.github/workflows/results-backfill.yml`, 1hr after `tapology-backfill.yml` so the two don't fight for Render DB connections or Puppeteer memory. `BACKFILL_ORGS` env var (CSV) narrows a manual dispatch to specific orgs for smoke-testing.

Tapology family (PFL/RIZIN/BKFC-via-Tapology/Zuffa/KC/DBX/Top Rank/Golden Boy/Gold Star/The Ring/Matchroom-via-Tapology/MVP) stays on the older `backfillTapologyResults.ts` + `tapology-backfill.yml` path.

## GitHub Actions Workflows
- `ufc-scraper.yml` — daily UFC data scrape
- `ufc-live-tracker.yml` — live event tracking during UFC events
- `pfl-scraper.yml` — daily PFL data scrape (22:00 UTC)
- `tapology-backfill.yml` — scheduled backfill that re-runs `backfillTapologyResults.ts` against recently-completed Tapology-backed events to fill in winner/method/round results that the live tracker missed (e.g. results posted after the 12h live window closed). Accepts `window_days` input.

  ```bash
    -d '{"ref":"main"}'
  ```

## Key Gotchas
- **UFC scraper** MUST run with `TZ=America/New_York` — UFC.com adapts times to viewer timezone via client-side JS. GitHub Actions runners are UTC, so without this the parser gets wrong times.
- **PFL scraper** must only scrape *upcoming* events, not the full historical archive. `pflmma.com/all-seasons` renders both upcoming and past events in the same DOM via two tab panels (`#nav-upcoming` and `#nav-past`). Step 1 scopes the event-link `querySelectorAll` to `#nav-upcoming` — do NOT use the whole `document` or you'll scrape ~130 past events back to 2018 and time out before the import step runs. See `2026-04-09.md` for the debug log.
- **PFL per-event date** must come from the **calendar share links** on the event page (`dates=20260411T180000`, `dtstart=`, `st=`), NOT from `DateTime.fromISO(...)` in page scripts. The DateTime.fromISO script is the site-wide countdown timer in the header, which shows the *next featured PFL event across all events* — so every event page's script scrape returns the same date. Using the calendar-URL extraction in Step 2 (`eventDateFromCalendar`) gives a reliable per-event date. Upcoming events show no year on the list page (`Fri, Apr 10`), so list-view date parsing is unreliable for upcoming events anyway. If the calendar fallback breaks, every event will get the parser's `2099-01-01` placeholder and disappear from the mobile "upcoming" tab. See `2026-04-09.md` for the debug log.
- **PFL event start time** is currently still scraped from `DateTime.fromISO(...)` and will be wrong for non-featured events (same countdown-pollution problem as above). Not fixed yet — promote the `"Main Card: X:XX PM ET"` text extraction above the ISO scrape when you get to it.
- **RAF scraper** determines event status from the per-event page's `div.past-event-tag` (`isPastEvent`), not from gallery card buttons. An earlier version keyed off "buy tickets" button presence, which wrongly marked future events as Complete whenever tickets weren't yet on sale (e.g. RAF 09 May 30 2026). Fallback when the event page scrape fails is date-based with a 24-hour buffer — don't reintroduce button-presence heuristics. Also note: `rafDataParser.ts:225` is write-once on `eventStatus`, so re-running the scraper will NOT correct a historically wrong status; those have to be fixed at the DB level. See `2026-04-09.md`.
- **Tapology live tracker** (`services/tapologyLiveParser.ts` + `services/tapologyLiveScraper.ts`) now has two non-obvious behaviors added 2026-04-10:
  1. **On-the-fly fight creation**: if a scraped Tapology fight has no matching DB fight (fighter-name match with several fallback strategies), the parser creates both fighters via `findOrCreateFighter` and the fight row in place. `orderOnCard` comes from Tapology's card position (`boutOrder`, extracted from `#boutCompactNumber{boutId}`) — opener = 1, main event = N. Falls back to `max(orderOnCard) + 1` if that slot is taken. This is how we recovered from the PFL daily scraper missing Mohamed vs Samuel on the Pretoria card.
  2. **Idempotent completion**: the completion block is no longer gated on `fightStatus !== 'COMPLETED'`. It compares each result field (winner/method/round/time) against the DB value and writes any difference. This lets the tracker backfill a result onto a fight that the lifecycle job already marked complete without data (the premature-completion case).
  - **No Contest / Draw**: Tapology rendered as `<span class="uppercase">Ends in a No Contest, Unintentional Eye Poke</span>`. `normalizeMethod` checks `"no contest"` and `"draw"` **before** the KO/TKO branch (re-ordered 2026-04-10 because "No Contest, Accidental Knockdown" could otherwise misclassify). For NC/Draw the parser sets `winner` to the sentinel string `'nc'` / `'draw'` per the `Fight.winner` schema comment, NOT null — the mobile UI renders the blue NC badge based on this value.
  - **DRAW/NC convention extended to non-Tapology live parsers (2026-04-28)**: `bkfcLiveParser.ts`, `matchroomLiveParser.ts`, `oktagonLiveParser.ts`, and `oneFCLiveParser.ts` now also encode draw → `winner='draw'` and no-contest → `winner='nc'` when the scraper produces a method but no winner side. Same sentinel-string contract as Tapology. Web `CompletedFightCard.tsx` and `FightDetailClient.tsx` render gold DRAW + blue NO CONTEST badges off this. UFC live parser was already correct via its own path. Historical rows backfilled by `src/scripts/backfillDrawsNcs.ts` against Render (52 draws + 70 NCs filled). See `2026-04-28.md`.
- **Tapology scraper output directory convention**: the output dir in each `scrape*Tapology.js` MUST match the input dir in the corresponding `*DataParser.ts`, or the import step ENOENTs on `latest-events.json` every run — scrape logs success, the CI job exits 1. Convention across the codebase is **no hyphens / concatenated lowercase** (`goldenboy`, `toprank`, `dirtyboxing` family — though Dirty Boxing historically uses `dirty-boxing` and both sides agree). Admin panel key (`admin.html`), image dirs (`/images/events/<promo>/`), and R2 paths all follow the no-hyphen form. Golden Boy and Top Rank were silently failing daily because the scrapers wrote to `golden-boy` / `top-rank` while the parsers read from `goldenboy` / `toprank` — fixed 2026-04-10. See `2026-04-10.md`.
- **Event-lookup `findFirst` in Tapology parsers MUST be tightly scoped**. The pattern `OR: [{ ufcUrl }, { name }, { name: { contains: '<promotion>' } }]` is dangerous because Prisma's `findFirst` is unordered — the third clause will cross-match unrelated events in the same promotion, and the subsequent `update` will try to overwrite that wrong row's `ufcUrl` with a URL another row already legitimately owns → `P2002 Unique constraint failed on (ufcUrl)`. Correct pattern (used by `ufcDataParser.ts` and now `dirtyBoxingDataParser.ts`): match on `ufcUrl` first, fall back to `(promotion, exactName)` or `(promotion, dateWindow)`, and wrap the `update` in a `try/catch` that retries **without setting `ufcUrl`** on P2002 so legacy duplicate rows don't block the sync. Dirty Boxing was crashing every daily run on DBX 5 because of this — fixed 2026-04-10.
- **The PFL / BKFC / RIZIN / Zuffa / Karate Combat / Dirty Boxing live trackers all run on the Hetzner VPS** (`scraper-service` systemd unit, port 3009), **not GitHub Actions**. `eventLifecycle.ts` calls the VPS first via `VPS_SCRAPER_URL` and only falls back to `tapology-live-tracker.yml` if the VPS is unreachable. **The VPS does not auto-deploy from `main`** — fixes to `tapologyLiveScraper.ts`/`tapologyLiveParser.ts`/`scraperService.ts` require SSH + `bash /opt/scraper-service/packages/backend/vps-update.sh` (does `git pull && pnpm install && pnpm build && systemctl restart scraper-service`). If you push a tracker fix and it doesn't show up, this is almost certainly why. See `2026-04-10.md`.
- **Tapology daily scrapers MUST extract fighter links scoped to each `<li class="border-b">` bout row, NOT page-wide.** The naive `document.querySelectorAll('a[href*="/fightcenter/fighters/"]')` then pair-by-index pattern pulls fighter links from sidebar widgets and related-event panels too. On every Tapology event page this injects ~25 unrelated boxing fights (Tyson Fury/Makhmudov, Conor Benn/Prograis, Frazer Clarke/Huni, etc.) into the scraped card. The correct pattern — used by `tapologyLiveScraper.ts` and now all 7 `scrape*Tapology.js` daily scrapers — is to iterate `document.querySelectorAll('li.border-b, li[class*="border-b"]')`, skip any `<li>` inside `nav/header/footer/aside`, then query fighter links only within each `<li>`. See `2026-04-11.md`.
- **Data parser cancellation-detection signatures must be fighter-ID-based OR fully diacritic-normalized.** Two distinct bugs bit today:
  1. **Last-name-only sigs collide and mismatch.** The old pattern `[normalizeName(name).split(/\s+/).pop(), ...].sort().join('|')` breaks when two fighters share a last name (the Blaze brothers at RAF 08 → Jason Nolf vs Marcus Blaze escaped cancellation because the scraper had Nolf vs Joey Blaze producing the same `blaze|nolf` sig). It also breaks when `lastName` contains multiple words (DB has `"Bo Mi Re Shin"` as lastName, scrape produces `shin` as last-word — no match, so Baumgardner vs Shin got stuck CANCELLED). Fixed by switching `mvpDataParser`/`rafDataParser`/`dirtyBoxingDataParser`/`karateCombatDataParser`/`zuffaBoxingDataParser` to `[fighter1Id, fighter2Id].sort().join('|')` which is globally unique and immune to either failure mode.
  2. **Full-name sigs must strip diacritics on BOTH sides.** Fighters are stored with diacritics stripped (`scrapeAllOktagonData.js` calls `stripDiacritics(firstName)` and `stripDiacritics(lastName)` before writing), but `scrapedFightPairs` was built from the raw scraped names. Czech/Slovak/Polish fighters never matched: DB had `"lucia szabova|leidiane fernandes"`, scrape built `"lucia szabová|leidiane fernandes"`. The Oktagon parser's `normName` helper (`stripDiacritics + lowercase + trim`) applied to both sides is the fix. **Any parser that still builds name-pair sigs for cancellation (`topRankDataParser`, `oneFCDataParser`, `matchroomDataParser`, `pflDataParser`, `rizinDataParser`, `goldStarDataParser`, `goldenBoyDataParser`, `bkfcDataParser`, `ufcDataParser`) has this same latent bug** and should be audited in a follow-up sweep. See `2026-04-11.md`.
- **Oktagon `__NEXT_DATA__` has TWO queries containing `'fightCard'`** — match the query key strictly. Oktagon's dehydrated React Query state now exposes both `['events', 'fightCard', <id>]` (the real fight-card data as an array of card objects) and `['bettings', 'fightCard', <id>]` (an array of fight-ID strings used by the betting widget). The old loose match `queryKey.includes('fightCard')` inside a plain `for` loop let the bettings query clobber the real data (because it iterates later), and the resulting string array produced 0 fights → `{ skipped: true }` → the parser skipped the event → all fights lingered as stale. Use `queryKey[0] === 'events' && queryKey[1] === 'fightCard'`. Symptom: Oktagon 87 was missing its own title fight (Szabová vs Fernandes) plus 5 others. See `2026-04-11.md`.
- **All 15 data parsers must un-cancel events that reappear on the source site**, not just fights. Each parser has an event-level cancellation-detection block that marks an `UPCOMING` DB event as `CANCELLED` when it's missing from the current scrape (correct). The inverse — what happens when a previously-cancelled event reappears — used to be a no-op: the "existing event" update branch deliberately preserved `eventStatus` ("lifecycle service manages it"), so a row that was flagged `CANCELLED` once (even by an earlier flaky/partial run) stayed `CANCELLED` forever, even though every subsequent scrape re-imported all its fights. Symptom (2026-04-11): live PFL Chicago 2026 missing from the app despite today's successful scraper run — plus Belfast, Sioux Falls, and Brussels all stuck `CANCELLED` while still listed on pflmma.com. Fix applied to all 15 parsers: capture `wasCancelled = event.eventStatus === 'CANCELLED'` pre-update, and when true spread `{ eventStatus: 'UPCOMING', completionMethod: null }` into the update data plus `fight.updateMany` the event's `CANCELLED` fights back to `UPCOMING`. Only fires on reappearance, so never conflicts with the lifecycle service for non-cancelled events. See `2026-04-11.md`.
- **Tapology parser's cancellation sweep is skipped on `COMPLETED` events.** `tapologyLiveParser.ts` gates the "missing-from-page → CANCEL" block on `event.eventStatus !== 'COMPLETED'`. Rationale: once an event is done the card is frozen, and any fight missing from Tapology's current page is almost certainly a UI quirk (pagination, exhibition hidden, stale cache), not a real cancel. Per-fight result writes (winner/method/round/time) still run against `COMPLETED` events — only the sweep is gated. This is what makes `backfillTapologyResults.ts` safe to re-run against finished cards without wrongly CANCEL-ing real fights. Added 2026-04-11; see `2026-04-11.md` Session 2/4.
- **Top Rank / Golden Boy hub discovery uses DOM scoping, not slug filtering.** Top Rank and Golden Boy event URLs on Tapology are named by headliners (`/fightcenter/events/12345-zayas-vs-fortea`) and contain no `top-rank` / `golden-boy` in the slug, so the old `slugFilter: ['top-rank']` excluded every event they promote from `TAPOLOGY_PROMOTION_HUBS` in `runTapologyLiveTracker.ts`. Fixed by switching both entries to empty `slugFilter: []` + `scopeSelector: '#content'`, matching the pattern Gold Star already used. `backfillTapologyResults.ts` carries a duplicated copy of the same map and needs the same fix if either is edited — refactor into a shared module is a tracked low-priority follow-up. See `2026-04-11.md` Session 2/4.
- **Tapology parser can create orphan rows during backfill on COMPLETED events.** When `tapologyLiveParser.ts` fails to match a scraped fight to any DB row (because the DB has a wrong-spelling fighter *and* a pre-existing CANCELLED row for the same pairing that gets skipped by the per-fight block), the parser falls through to the on-the-fly-create path and writes a new `Fight` row with whatever card metadata it has — often producing a row with `cardType: null` and `orderOnCard: max+1`. This is how PFL Africa Pretoria ended up with three rows for Mohamed vs Samuel (two CANCELLED at Main Card #7 with different spellings, one orphan COMPLETED at `cardType=null #12`). **Known bug, not fixed yet.** Next fix: when running against an event that's already `COMPLETED`, unmatched scraped fights should log a warning and skip the create path entirely — creating new rows on finished cards is never correct. See `2026-04-11.md` Session 4.
- **Promotion string naming drift exists across scrapers.** `SELECT promotion, count(*) FROM "Event" GROUP BY promotion` surfaces three casing/spacing variants today, all scraper-created (not admin-created phantoms): `TOP_RANK` (25) vs `Top Rank` (1), `Matchroom Boxing` (14) vs `Matchroom` (2), `Rizin` (9) vs `RIZIN` (2). Different parsers or older scraper versions wrote the variants. **Not yet normalized** — planned follow-up is a one-shot `updateMany` to normalize + find/fix the parser(s) writing the variants so future scrapes stay canonical. See `2026-04-11.md` Session 4.
- **Tapology scrapers built their output `event` object with explicit field enumeration and silently dropped `eventStartTime` on the way to JSON.** 5 of the 7 `scrape*Tapology.js` daily scrapers (Top Rank, Karate Combat, Zuffa Boxing, Dirty Boxing, Golden Boy) extract `eventStartTime` correctly inside `page.evaluate` and assign it to in-memory `data.eventStartTime`. But when the per-event scrape function returned, the per-promotion scraper constructed a NEW `event` object pushed into `allEvents` by **explicitly enumerating** fields (`eventName, eventType, eventUrl, eventSlug, venue, city, state, country, dateText, eventDate, eventImageUrl, status, fights`) — and **omitted `eventStartTime`**. The parser then read `eventData.eventStartTime === undefined` and silently logged "No start time found". Latent since 2026-04-06; every CI run for ~2.5 weeks dropped the field. Gold Star and MVP had `eventStartTime` in their output object, which is why those two were unaffected. Symptom (2026-04-25): 17 upcoming events on the admin panel had `mainStartTime: null`. Fix: add `eventStartTime: eventData.eventStartTime || null` to the output object in all 5 affected scrapers. **General lesson**: when scraper logs say "No start time found" but a local probe extracts the value cleanly from the same URL, the failure is between page-eval return and the JSON file the parser reads, not in page-eval itself. Add `console.log` right after `await page.evaluate(...)` before any field copy/projection to localize the dropoff. **Sub-issue solved at the same time**: page-eval used `document.querySelector('li span.font-bold')` to locate the Date/Time sidebar row; on some renderings the first match was a fight-row label (`[W]`, `[Main Event]`, `[Prelim]`) before reaching the sidebar. Iterate `querySelectorAll('li span.font-bold')` and pick the label matching `/date\s*\/?\s*time/i` — strictly more correct, but not the actual root cause of today's empty column. See `2026-04-25.md`.
- **`backfillStartTimes.ts` is the start-time analog of `backfillTapologyResults.ts`** — for any upcoming event whose three start-time fields (early/prelim/main) are all null, it discovers the Tapology page (URL on the event, then Tapology search, then promotion hub), pulls the `<li>Date/Time:</li>` value, and writes `mainStartTime`. Tapology only exposes a single Date/Time per event, so this script writes mainStartTime only — the lifecycle's `getEventStartTime()` falls through to mainStartTime when prelim/early are null, so this is enough to unblock UPCOMING → LIVE auto-transitions. Currently runs manually (`pnpm tsx src/scripts/backfillStartTimes.ts`) — wiring it to a cron is a tracked followup.
- **CSS `url(...)` extractors must not truncate on literal parens in filenames.** Scrapers that read `background-image` from inline `style` attributes (RAF, PFL) historically used patterns like `/url\("?([^")\s]+)"?\)/` — the `[^)]` character class stops at the FIRST `)`, so a URL ending in `..._banner%20(1).webp` gets captured only up to `...%20(1`, losing the extension and the outer `)`. The truncated URL then 404s during R2 upload, and the parser falls back to the same broken source URL. Symptom (2026-04-11): RAF 08's DB `bannerImage` was `https://cdn.prod.website-files.com/.../69d521cce17d0ba08a8a7365_RAF08%201920x1080fff%20(1` (unclosed paren, no extension), unlike RAF 06/07/09 which live on R2. Root cause: the file was uploaded to Webflow as `RAF08 1920x1080fff (1).webp` (duplicate-download suffix from the uploader's filesystem). Correct pattern: isolate the `background-image` declaration with `split(/;/).find(d => /background-image/i.test(d))`, then slice between `indexOf('url(')` and `lastIndexOf(')')`, and strip optional wrapping quotes. Applied to both `scrapeAllRAFData.js` (Cheerio) and `scrapeAllPFLData.js` (Puppeteer `page.evaluate`, both inline-style and computed-style paths). See `2026-04-11.md`.
