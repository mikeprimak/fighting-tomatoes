### BKFC Blood 4 Blood — wrong start time (11 PM EDT instead of 7 PM EDT)
- **Symptom**: Blood 4 Blood (Daytona) started at 7 PM EDT but the app's lifecycle didn't flip the event to LIVE until 11 PM EDT. App "thought the event hadn't started yet."
- **DB state at time of bug**: `mainStartTime: 2026-05-07T03:00:00Z` (= 11 PM EDT May 6). Actual event start per BKFC.com: 7 PM EDT.
- **Root cause**: The BKFC event page renders `May 6, 2026 7:00 PM` inside `<div data-event-date-est="" class="text-block-8">`. The scraper's Try 1 selector list (`div.text-color-gold`, `p`, `.event-date`, `.header_date`) doesn't match this div. The `text-color-gold` div now holds **only the date string** ("May 6, 2026"), and the time is in the sibling div.
  - With Try 1 missing, the scraper falls through to Try 2 (`[data-countdown-date]`) — and the in-code comment already warned this is the **main card** time (11 PM EDT for typical BKFC PPVs), not the prelims/event start.
  - Net effect: every BKFC event whose page uses the `text-block-8` layout silently gets its main-card countdown time written as `mainStartTime`, ~4 hours late.
- **Fix**: Added `[data-event-date-est]` to the head of Try 1's selector list. The textContent of that element is the canonical "Month DD, YYYY H:MM PM" string the regex already expects. Try 1 now succeeds and Try 2 is no longer reached.
- File: `packages/backend/src/services/scrapeAllBKFCData.js:417-432`
- Note: existing BKFC events with stored bogus times (Hammond, Hollywood, Palm Desert, Nashville etc.) will self-correct on the next BKFC scraper run (overwrite-on-update path is unaffected).
