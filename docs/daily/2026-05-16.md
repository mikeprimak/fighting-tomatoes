## Fan DNA — full architectural plan

The bulk of the session. **Rewarding Users Wave 3 ("Hype DNA") is renamed and rescoped to Fan DNA**: a multi-signal personality engine that surfaces insights about the user as a fan, not just their hype accuracy.

### Architectural consolidations

- **Hype accuracy folds INTO Fan DNA as one trait among many.** The `Hype vs Outcome` profile section that shipped 2026-05-15 gets replaced by a single `Fan DNA` SectionContainer when the new wave ships. Prevents the profile from becoming a wall of single-insight sections.
- **Wave 2 and Wave 3 collapse into one Fan DNA wave** that grows trait-by-trait as data allows. `Hype vs Outcome` is reframed as the first Fan DNA trait, not a standalone deliverable.

### Trait taxonomy

Four families × four tiers by data dependency (= shipping order):
- **A. Affinity** — style, pace, finish, outcome, weight class, card position, org, gender
- **B. Behaviour** — rating bias, cadence/tenure, follow breadth, comment vs lurk, early vs late rater
- **C. Prediction** — hype accuracy (folded in), hot take rate, predicted-winner accuracy, sweet spot
- **D. Identity arc** — tenure, DNA shift over time, discovery diversity, cross-promo openness

- **Tier 1** ships now (existing signals): ~12 traits.
- **Tier 2** unlocks with AI enrichment Phase 1 cron: style affinity, stakes affinity, calibration-by-stakes.
- **Tier 3** needs post-fight scrape: back-and-forth affinity, pace, finish quality, round-arc, surprise factor.
- **Tier 4** needs longitudinal time: DNA shift, year-over-year, Hype Wrapped fuel.

### Tone (Worms-style)

Tonal reference locked: **Worms (the video game)** — dry, absurd, affectionate, theatrical understatement. The narrator finds the user amusing the way a witty butler finds them amusing, not the way a sarcastic friend roasts them.

Tone rules (load-bearing):
- Affectionate, not diminishing. User is delightful; the *data* is the joke.
- Absurd metaphors > insults. *"The bar is somewhere above the moon"* > *"the community thinks you're broken."*
- Specific imagery > judgment. Numbers in setup are the comedic engine.
- **Banned words** (land as verdicts on the user): sicko, broken, cooked, joyless, obsessed, addict, problem, weird, twisted, embarrassing.
- Never punching down at fighters or promotions.
- Two-beat structure: setup (fact) + punchline (wry observation).
- "We" narrator voice has personality.

Soft tone is baseline (~60%), humor is heavy and first-class (~40%). Humor variants iterate fastest, so copy lives as data not code.

### Emotional design principles

Eight principles documented in chat transcript — the *why* behind the tone rules:
1. Be noticed, not surveilled
2. The number IS the joke
3. Surprise without strangeness
4. Validate the weird without flagging it
5. User is the hero of the joke, not the butt
6. Earned wit, not constant performance
7. Negative space is generous (not every action gets a line)
8. Describe behaviour, never assign identity (*"You tend to love wars"* not *"You're a war fan"*)

### Reveal modal integration

The hype/rating reveal modals that shipped 2026-05-15 get a **third beat**: a contextual Fan DNA line tied to the action just taken. Sequence: community chart → comparison → DNA observation → close.

Architectural implication: trait registry must support **two query modes** from day one — `batchCompute` (nightly) and `eventEvaluate({ user, action, fightId, value })` (real-time). Retrofitting event-mode later is painful.

This is the primary user entry point to Fan DNA. Users learn the feature exists by being shown 30+ DNA lines on fight night before they ever tap the profile section.

### Recommendation surfaces inside Fan DNA

- **Fights you might love — UPCOMING is primary view** (rate-it-after closure-loop hook). Past unrated fights = secondary tab. Each rec cites the driving trait: *"Because you tend to love striking matchups."*
- **Fighters you might love** — same pattern; doubles as a follow-growth surface tied to the acquisition narrative.
- **Cross-promotional lanes** are first-class — *"Fighters you'd like from PFL"*, *"Bellator alumni now in ONE"*, etc. Scorer weights *promotion novelty* as a factor. This is load-bearing for the follow-fighter dataset's acquisition value.

### Three-layer freshness strategy

The big design problem: how does the system stay surprising indefinitely without repeating?
- **Layer 1 (~60%)** — canned pool with combinatorial templates, 30-day cooldown per exact line per user.
- **Layer 2 (~30%)** — Claude Haiku 4.5 generates 3 candidates per surfacing, tone-filter picks one, fallback to Layer 1 on quality failure. ~$1/mo at expected volume. Always fresh.
- **Layer 3 (~10%, priority-bumped)** — nightly fun-fact scanner finds surprising patterns (streaks, anniversaries, contradictions, milestones), hand-curated copy per pattern.

Same-trait pacing (corrected from earlier "no same trait twice per session" rule): repeats are fine and realistic (5 KO ratings → multiple finish-affinity mentions is honest). Rules: no back-to-back fires of the same trait, soft cap ~40% per session for any single trait.

### Toggle-storm / harvest contingency

Detection signal — `(userId, fightId, actionType)` tuples, NOT raw action velocity:
- **Toggle storm** (fires meta response): same user × same fight × same action type, 5+ value changes in 10min.
- **Card sweep** (normal DNA fires): different fights, fast pace. Even 12 ratings in 5 min during a live card is engaged-fan behavior.

Response tiers:
1. 5+ toggles → META line from dedicated pool (*"You've changed your mind six times. We're enjoying watching this."*).
2. 10+ toggles → EXIT meta + quiet that (user, fight) for 1 hour. The user gets a wink, not a stutter; the system never visibly runs dry.

Bonus: persistent toggle-prone users earn a real trait (*"You revise your scores more than the average. Considered. We respect it."*). Every pattern, including this one, is its own coherent fandom.

### Resilience architecture (mandatory)

The system must hold up at trait breadth. Design constraints:
1. **Trait registry, file-per-trait, auto-discovery.** Each trait = one folder under `packages/backend/src/services/fanDNA/traits/{trait-id}/` containing `trait.ts` + `copy.json` + `fixtures.test.ts`. Registry auto-discovers at boot.
2. **Copy as data.** Soft + humor variants in `copy.json`, hot-reloadable.
3. **Trait independence.** One trait throwing must not break the screen.
4. **Versioning per trait.** Computed values stored with `{ traitId, version, computedAt, value, confidence }`. Bumping a version triggers backfill.
5. **Backfill cron**, not manual reruns.
6. **Surface independence.** A trait renders the same regardless of surface (profile card, full screen, recommendation reason, share image, weekly email).
7. **Health dashboard** at `/admin/fan-dna` — firings/week, dwell, complaint rate per trait. Dead traits surface themselves.
8. **Sunset path** — `deprecated: true` stops new computation without deleting historical values.
9. **Fun-facts engine** sits alongside trait registry — same pattern, lower bar.
10. **Test harness** — each trait has synthetic-user fixtures.

### Architectural decisions locked

- **Trait registry > hand-wired surfaces.** Surfaces (reveal modal, profile card, future weekly recap) iterate the registry; no per-trait edits across the codebase.

- **Toggle-storm thresholds: 5 META, 10 EXIT, 1-hr quiet.** Quiet is per-(user, fight), not global — other fights still get normal DNA.
- **Engine is silent under cooldown saturation.** When every trait's copy is on cooldown for this user, return null. The system never visibly runs dry.

## Session 4 — First-rater reveal bug fix

Mike reported that rating Chris Avila vs Brandon Jenkins (first user-rating on that fight) showed the "you rated about average" branch in the reveal modal instead of the trailblazer "First to rate this fight" copy. Confirmed working after fix.

### Root cause — prefetch/mutation race

`CompletedFightModal` (and `UpcomingFightModal`) silently prefetches `getFightAggregateStats` on open so the reveal can render instantly via prefetched data + a local user-vote delta. The rate mutation also fires on tap (fire-and-forget). If the prefetch's HTTP call returns AFTER the mutation has committed, the prefetched stats already include the user's just-saved rating — and the delta math in `handleDone` then adds +1 on top:

```
baseTotal = 1 (post-mutation prefetch picked up our rating)
prevRating = null → liveTotal = 1 + 1 = 2
```

`getComparisonText(rating, avg, 2)` falls into the `delta > -1` branch ("about the same as the average fan") instead of the `totalRatings <= 1` trailblazer branch.

DB confirmed: `FightRating` had exactly 1 row for the fight (user `3c4a099c`, rating 6) and `fight.totalRatings = 1`. Backend was honest; the client was double-counting.

### Fix

Both modals now capture the authoritative `aggregateStats` (rating) / `{averageHype, totalHypePredictions, hypeDistribution}` (hype) from the mutation response into a ref in `onSuccess`. In `handleDone` they prefer the committed stats over the racy prefetch+delta path. Falls back to prefetch+delta only if the mutation hasn't returned yet (rare — Done is usually tapped well after the POST resolves). Ref resets on modal open.

Files:
- `packages/mobile/components/CompletedFightModal.tsx` — `committedStatsRef`, `onSuccess` captures `data.data.aggregateStats`, `handleDone` short-circuits when ref is populated.
- `packages/mobile/components/UpcomingFightModal.tsx` — same pattern with `committedHypeStatsRef` against `data.averageHype / totalHypePredictions / hypeDistribution`.

### Verified

Mike tested another first-rated fight after the Expo restart — trailblazer copy now fires correctly. Working tree edits, uncommitted.

### Pattern lesson

The "prefetch on open + fire-and-forget mutation + delta math in handleDone" pattern is racy by design. Any modal using it should prefer server-authoritative post-commit stats from the mutation response when available, and reserve the delta math purely as a fallback for the case where the mutation hasn't resolved by Done-tap time. Saved as memory entry.
