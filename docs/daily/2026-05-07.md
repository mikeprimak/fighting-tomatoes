**What every parser now does:**
1. **Sanity gate** ‚Äî if the upcoming-events scrape returned fewer than `MIN_SCRAPED_EVENTS_FOR_CANCEL` (3), skip ALL cancellation passes. Treat the page as broken.
2. **Two-strike rule** ‚Äî when a fight or event is missing from a sane scrape, increment its `missingScrapeCount`. Only mark CANCELLED on the second consecutive miss. Counter resets to 0 on reappearance.
3. **Existing 75% safety floor** preserved ‚Äî for UPCOMING events, the daily scraper still requires the partial-fight-list scrape to return ‚â•75% of DB's non-cancelled fight count before cancelling individual fights.

**Cleanups along the way (logged in commit body):**
- `goldStarDataParser` had no `eventInProgress` / safety-floor guard at all ‚Äî it cancelled blindly on any missing pair. Now matches the rest of the parsers.
- `topRankDataParser` had a hand-rolled `if (uniqueEvents.size >= 3)` event-level skip. Replaced with the shared `MIN_SCRAPED_EVENTS_FOR_CANCEL` constant for consistency.
- `rizinDataParser` already used `firstName + lastName` (not display `name`) for pair keys with TBA-fighter skips on both branches ‚Äî preserved as-is.

## What this means for next UFC 328-style transient failure

- **Single missing scrape:** counter bumps from 0 ‚Üí 1, logs a strike, event/fight stays UPCOMING. No cascade-cancel. (This is the bug that bit us with UFC 328 ‚Äî now fixed.)
- **Two consecutive missing scrapes (~24h apart):** real cancel.
- **Reappears at any point:** counter resets to 0. Strike forgiven.
- **Whole scrape returns <3 events:** all cancellation logic skipped. Page presumed broken.

## Risk note

The schema change (`missingScrapeCount Int @default(0)` on Event and Fight) is metadata-only ‚Äî never read by app/mobile, never surfaced in API responses. The rollout is reversible by reverting the commits; no data backfill needed.
