## Admin: event-complete no longer flips CANCELLED fights to COMPLETED

### What happened
User set MVPW 02 (live Boxing card on Tapology, event `566bdf8b-e5ed-4e8a-9234-f565e8e8908c`) to COMPLETED in the admin panel, then back to LIVE. During the COMPLETED window the app loaded a bunch of "old cancelled fights" as completed — alternate matchups that had been cancelled across prior re-scrapes of the card (each `orderOnCard` slot has 1 real UPCOMING fight + several historical CANCELLED rows from fighter swaps).

### Root cause
`packages/backend/src/routes/admin.ts` — the bulk fight update on event-complete used `fightStatus: { not: 'COMPLETED' }`, which matches CANCELLED too. So every cancelled alternate on the card was flipped to COMPLETED with `completionMethod='manual'`, `completedAt=now`.

### Fix — commit `8791d98`
```ts
// before
fightStatus: { not: 'COMPLETED' }
// after
fightStatus: { notIn: ['COMPLETED', 'CANCELLED'] }
```
Pushed to `main`; Render auto-deploys the backend.

### Why the DB was already half-healed when I looked
Because MVPW 02 has `scraperType='tapology'`, the Tapology live tracker (VPS, ~30s cadence) re-synced fight statuses from source within a minute of the bad write — restoring real scheduled fights to UPCOMING and pushing the historical alternates back to CANCELLED. The tracker does not clear `completedAt` / `completionMethod` on those transitions, so the *status* was correct but 46 rows carried stale `completedAt='2026-04-17 23:03:57.105'` + `completionMethod='manual'`.

### DB cleanup
Ran on Render directly:
```sql
UPDATE fights
SET "completedAt" = NULL, "completionMethod" = NULL
WHERE "eventId" = '566bdf8b-e5ed-4e8a-9234-f565e8e8908c'
  AND "completedAt" = '2026-04-17 23:03:57.105'
  AND "completionMethod" = 'manual'
  AND "fightStatus" IN ('UPCOMING', 'CANCELLED');
-- UPDATE 46  (10 UPCOMING + 36 CANCELLED)
```
Verified `with_completedAt=0` for this event post-cleanup.

### Lesson
Any `updateMany` that filters fight status with `{ not: 'COMPLETED' }` is a bug magnet — CANCELLED is also "not completed" but must never be auto-completed. When writing bulk fight-status transitions, enumerate the statuses that should transition, not the ones that shouldn't. Worth a grep sweep if similar patterns exist in scrapers or lifecycle code.

### Knock-on behavior worth remembering
When a production-scraper event (UFC / Tapology / OneFC / BKFC / Oktagon / RAF) gets manually muddled via admin, the live tracker will self-correct `fightStatus` on its next run but **won't** reset `completedAt` or `completionMethod`. If the admin endpoint bulk-sets those fields, they'll linger as stale metadata unless explicitly cleared.
