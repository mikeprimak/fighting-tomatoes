# 2026-06-02 — Web app QA sweep (9 fixes)

Batch of web-app-only fixes reported by Mike. All shipped in one commit
(`4253938`), pushed to `main`; Vercel auto-deploys `packages/web` → goodfights.app.
`tsc --noEmit` and `next build` both clean.

## Fixes

1. **SEO — "Duplicate without user-selected canonical" (GSC).** Legacy
   fightingtomatoes.com URLs land on the homepage with junk query strings
   (`?sortby=…`, `?pagenumber=…`, `?u=…`) and Google flagged them as dupes.
   Added a self-referencing canonical (`/`) on the home route so every
   query-param variant consolidates to `https://goodfights.app/`. Required
   splitting `app/page.tsx` into a server wrapper (exports `metadata`) +
   `app/HomeClient.tsx` (the existing client component), since metadata can't be
   exported from a `'use client'` file.

2. **Comment delete + modal close (RateFightModal).**
   - Clearing the comment textarea now deletes the review: when the user had a
     comment and leaves it empty, we call `updateFightUserData(id, {review:null})`
     (rating untouched). Previously empty edits were a no-op.
   - Backdrop now closes only when the mouse *press started* on the backdrop
     (`onMouseDown` records origin; `onClick` checks it). Highlighting text in the
     textarea and releasing outside no longer closes the modal.

3. **Rating not rendering on a live-flagged fight (BKFC).** Root cause: the web
   `LiveFightCard` was showing **hype**, but the **mobile** live card (source of
   truth) shows **ratings** — aggregate rating front-left, user rating on the
   right. Web was out of parity. (The rating itself saved fine — `POST
   /fights/:id/rate` has no status gate.) Fixed: web LiveFightCard now mirrors
   mobile / web CompletedFightCard — aggregate rating (front) over hype (peek),
   user rating star on the right, comment-quote marker. Covers #9 too.

4. **Blog date off-by-one on home "Latest".** `EditorialHero` /
   `EditorialSecondary` formatted dates in the browser's local TZ while the
   article page renders server-side in UTC → browsers behind UTC showed day-1.
   Forced `timeZone: 'UTC'` in all three formatters.

5. **Completed-today events.** Past-mode badge now reads **COMPLETE** instead of
   TODAY (`EventCard`).

6. **Past-events org priority sort.** Within each day, primary orgs
   (UFC, PFL, BKFC, Karate Combat, Dirty Boxing, RAF) float above all other orgs;
   each group stays time-sorted (latest first). Client-side sort in
   `events/past/page.tsx` (`sortPastEvents`). Note: it sorts the loaded flat
   list, so ordering can shift slightly as more infinite-scroll pages load.

7. **Default to Live tab.** Home auto-routes to `/events/live` when any event is
   live, guarded by a `sessionStorage` flag so it fires once per session and
   doesn't bounce the user when they deliberately open Upcoming. Mirrors mobile.

8. **Fighter detail sort placement.** Moved the Date / Highest-Rated sort control
   a line above the RATING column header and pushed it to the far right.

9. **LiveFightCard aggregate square transparent.** Resolved as part of #3: the
   front (rating) square is now opaque grey `#202020` when unrated, so the offset
   hype square behind never bleeds through. Matches web CompletedFightCard.

## Not tested live yet

COMPLETE badge (#5), blog dates (#4). #9 (grey square) should already be correct

# 2026-06-02 (pt 2) — Admin panel: live-default + notification expectation tracker

Backend + web admin panel (`packages/backend/public/admin.html`). Backend
`tsc --project tsconfig.production.json --noEmit`: clean. Not yet exercised live.

## 1. Admin event list defaults to "Live" when something is live
New `bootstrapAdminView()` (runs after login/auth) checks
`GET /admin/events?status=live&limit=1` and sets the event filter dropdown to
**Live** when any event is live — mirrors the mobile app. Stays on "Upcoming"
otherwise.

## 2. Notification expectation tracker ("who's waiting on a ping")
Goal: deliver the "we'll notify you" promise manually until every org has a live
tracker — so the operator needs to see who is expecting a notification and be
alerted if someone starts expecting one during a card they were ignoring.

Source of truth: active `FightNotificationMatch` rows (rule engine: fighter
follows, hype rules, etc.). Each active match on a pending fight = one user waiting.

**Backend**
- `services/notificationExpectations.ts` (new):
  `getActiveEventExpectations()` (per-event distinct counts across LIVE+UPCOMING,
  bounded to pending fights), `getEventFightExpectations(eventId)` (per-fight),
  `checkAndAlertNewExpectations()` (cursor-based via SystemConfig key
  `notif_expectation_alert_cursor`; pushes admin once per new expectation on a
  LIVE or ≤6h-imminent event; first run seeds cursor + skips backlog).
- `routes/admin.ts`: `GET /admin/notification-expectations`,
  `GET /admin/events/:id/notification-expectations`.
- `services/eventLifecycle.ts`: Step 4 calls the alert checker every 5 min
  (best-effort try/catch).

**Admin panel UI**
- "🔔 N waiting · M events" board under the live-count notice: per event waiting
  users, LIVE/SOON badge, `+N new` (last 24h), ⚠️ for no-auto-tracker events;
  click to jump to the event.
- Per-fight 🔔 badge in the fights list (distinct users waiting on that fight).
- Auto-refresh 30s; when a LIVE event's count rises while the panel is open →
  toast + WebAudio beep + (if granted) desktop notification. Bell link in the
  board header toggles desktop alert permission.

## Operational note
Server-side push targets admin accounts from `ADMIN_EMAILS`
(prod = `michaelsprimak@gmail.com`); it only reaches you if that account has the
app installed with notifications on. The in-panel desktop/audible alert covers the
panel-open (incl. background-tab) case.
