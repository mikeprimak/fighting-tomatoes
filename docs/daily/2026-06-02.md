# 2026-06-02 — Web app QA sweep (9 fixes)

Batch of web-app-only fixes reported by Mike, pushed to `main`; Vercel
auto-deploys `packages/web` → goodfights.app. `tsc --noEmit` and `next build`
clean at each step. Commits: `4253938` (initial 9), `f13082b` (revert a wrong
LiveFightCard pass), `0c4411b` (LiveFightCard ratings parity — final #3/#9).

## Fixes

1. **SEO — "Duplicate without user-selected canonical" (GSC).** Legacy
   fightingtomatoes.com URLs land on the homepage with junk query strings
   (`?sortby=…`, `?pagenumber=…`, `?u=…`) and Google flagged them as dupes.
   Added a self-referencing canonical (`/`) on the home route so every
   query-param variant consolidates to `https://goodfights.app/`. Required
   splitting `app/page.tsx` into a server wrapper (exports `metadata`) +
   `app/HomeClient.tsx` (the existing client component), since metadata can't be
   exported from a `'use client'` file.

2. **Comment delete + modal close (RateFightModal).**
   - Clearing the comment textarea now deletes the review: when the user had a
     comment and leaves it empty, we call `updateFightUserData(id, {review:null})`
     (rating untouched). Previously empty edits were a no-op.
   - Backdrop now closes only when the mouse *press started* on the backdrop
     (`onMouseDown` records origin; `onClick` checks it). Highlighting text in the
     textarea and releasing outside no longer closes the modal.

3. **Rating not rendering on a live-flagged fight (BKFC).** Root cause: the web
   `LiveFightCard` was showing **hype**, but the **mobile** live card (source of
   truth) shows **ratings** — aggregate rating front-left, user rating on the
   right. Web was out of parity. (The rating itself saved fine — `POST
   /fights/:id/rate` has no status gate.) Fixed: web LiveFightCard now mirrors
   mobile / web CompletedFightCard — aggregate rating (front) over hype (peek),
   user rating star on the right, comment-quote marker. Covers #9 too.

4. **Blog date off-by-one on home "Latest".** `EditorialHero` /
   `EditorialSecondary` formatted dates in the browser's local TZ while the
   article page renders server-side in UTC → browsers behind UTC showed day-1.
   Forced `timeZone: 'UTC'` in all three formatters.

5. **Completed-today events.** Past-mode badge now reads **COMPLETE** instead of
   TODAY (`EventCard`).

6. **Past-events org priority sort.** Within each day, primary orgs
   (UFC, PFL, BKFC, Karate Combat, Dirty Boxing, RAF) float above all other orgs;
   each group stays time-sorted (latest first). Client-side sort in
   `events/past/page.tsx` (`sortPastEvents`). Note: it sorts the loaded flat
   list, so ordering can shift slightly as more infinite-scroll pages load.

7. **Default to Live tab.** Home auto-routes to `/events/live` when any event is
   live, guarded by a `sessionStorage` flag so it fires once per session and
   doesn't bounce the user when they deliberately open Upcoming. Mirrors mobile.

8. **Fighter detail sort placement.** Moved the Date / Highest-Rated sort control
   a line above the RATING column header and pushed it to the far right.

9. **LiveFightCard aggregate square transparent.** Resolved as part of #3: the
   front (rating) square is now opaque grey `#202020` when unrated, so the offset
   hype square behind never bleeds through. Matches web CompletedFightCard.

## Not tested live yet
Verify on goodfights.app once the deploy lands. Priority checks: rate a fight on a
live event card — aggregate rating (left) + your rating star (right) should show,
front square grey when unrated (#3/#9); clear a comment to delete it (#2); live-tab
auto-route (#7); past-event org sort (#6); COMPLETE badge (#5); blog dates (#4).

# 2026-06-02 (pt 2) — Admin panel: live-default + notification expectation tracker

Backend + web admin panel (`packages/backend/public/admin.html`). Backend
`tsc --project tsconfig.production.json --noEmit`: clean. Not yet exercised live.

## 1. Admin event list defaults to "Live" when something is live
New `bootstrapAdminView()` (runs after login/auth) checks
`GET /admin/events?status=live&limit=1` and sets the event filter dropdown to
**Live** when any event is live — mirrors the mobile app. Stays on "Upcoming"
otherwise.

## 2. Notification expectation tracker ("who's waiting on a ping")
Goal: deliver the "we'll notify you" promise manually until every org has a live
tracker — so the operator needs to see who is expecting a notification and be
alerted if someone starts expecting one during a card they were ignoring.

Source of truth: active `FightNotificationMatch` rows (rule engine: fighter
follows, hype rules, etc.). Each active match on a pending fight = one user waiting.

**Backend**
- `services/notificationExpectations.ts` (new):
  `getActiveEventExpectations()` (per-event distinct counts across LIVE+UPCOMING,
  bounded to pending fights), `getEventFightExpectations(eventId)` (per-fight),
  `checkAndAlertNewExpectations()` (cursor-based via SystemConfig key
  `notif_expectation_alert_cursor`; pushes admin once per new expectation on a
  LIVE or ≤6h-imminent event; first run seeds cursor + skips backlog).
- `routes/admin.ts`: `GET /admin/notification-expectations`,
  `GET /admin/events/:id/notification-expectations`.
- `services/eventLifecycle.ts`: Step 4 calls the alert checker every 5 min
  (best-effort try/catch).

**Admin panel UI**
- "🔔 N waiting · M events" board under the live-count notice: per event waiting
  users, LIVE/SOON badge, `+N new` (last 24h), ⚠️ for no-auto-tracker events;
  click to jump to the event.
- Per-fight 🔔 badge in the fights list (distinct users waiting on that fight).
- Auto-refresh 30s; when a LIVE event's count rises while the panel is open →
  toast + WebAudio beep + (if granted) desktop notification. Bell link in the
  board header toggles desktop alert permission.

## Operational note
Server-side push targets admin accounts from `ADMIN_EMAILS`
(prod = `michaelsprimak@gmail.com`); it only reaches you if that account has the
app installed with notifications on. The in-panel desktop/audible alert covers the
panel-open (incl. background-tab) case.

## 3. Test phone alert button
`POST /admin/notification-expectations/test-alert` pushes the calling admin's own
account on demand. "🔔 Test phone alert" button in the events filter bar; reports
whether a device actually received it (success=0 ⇒ admin account has no push token /
notifications off).

## 4. Manual per-fight notification send (with double-send safety)
For when the automated tracker fails (e.g. last weekend's missed Darren Till ping)
the operator can now send the per-fight push by hand from the admin panel.
- `POST /admin/fights/:id/send-notification` → dispatches via
  `notifyFightStartViaRules`, which only pushes match rows with
  `notificationSent=false` then marks them sent. Repeat calls / double-clicks send
  nothing — that's the safety. Returns `{ dispatched, alreadySent }`.
- `getEventFightExpectations` now returns `{ waiting, pending }` per fight
  (`pending` = distinct unsent recipients). `pending=0` ⇒ already fired.
- Fights list shows a "🔔 Send ping (N)" button when there are pending recipients
  and the fight is UPCOMING/LIVE; confirms before sending; flips to "✓ Ping sent"
  afterward. The 🔔 count badge still shows total waiting users.

## 5. Cancelled fights never notify (bug fix)
Found while reviewing the expectation tracker: a CANCELLED fight kept its active
`FightNotificationMatch` rows, and **neither dispatcher checked fight status** — so
followers of a fighter in a cancelled bout would still get a "starts soon" ping
(e.g. Iwo Baraniewski vs Billy Elekana, cancelled but several alerts set). Guard
added at every dispatch path:
- `notifyFightStartViaRules`: early-return if fight CANCELLED.
- `notifyEventSectionStart`: filter out CANCELLED fight IDs before matching.
- `eventLifecycle`: exclude CANCELLED fights when building section / all-fight IDs.
- admin manual-send endpoint: refuses CANCELLED (`{ cancelled: true }`).

# 2026-06-02 (evening) — News aggregator revived + on Home screen

Mike asked to bring back the old daily news scraper (dormant since ~Oct 2025)
and surface the aggregated articles on the mobile **Home** screen, below the
"Did You Know?" section. The whole feature was still built but disabled — DB
model (`NewsArticle`), scraper (`mmaNewsScraper.ts`), service wrapper
(`newsScraperService.ts`), and route (`GET /api/news`) were all intact; only
the cron and the (now-hidden) News tab had been turned off for Render memory
reasons. We did **not** revive the News tab — news now lives on Home only.

### What shipped
1. **Daily scraper back on a schedule, via GitHub Actions** (not Render — Puppeteer
   blows past Render's 512MB, same reason UFC scraper lives on GHA).
   - New entry point `packages/backend/src/scripts/scrapeNewsToDb.ts` →
     `newsScraperService.scrapeAndSave()` (scrape all sources, upsert by URL).
   - New workflow `.github/workflows/news-scraper.yml`: daily at 13:00 UTC +
     `workflow_dispatch`; needs only the `DATABASE_URL` secret; failure curls the
     admin test-alert (`org=News`). **Note: relies on remote `imageUrl` (OG), not
     `localImagePath`** — GHA's downloaded image files don't persist to Render, and
     the mobile UI already prefers `imageUrl`.
2. **Home screen "Latest News" section** (`packages/mobile/app/(tabs)/home.tsx`):
   horizontal carousel of article cards (image + source + headline) appended after
   "Did You Know?". Taps open the source URL via `Linking`. Reuses the existing
   `apiService.getNews()` client. Section is guarded by `news.length > 0` so it
   stays hidden if the DB is empty.
3. **Scraper robustness fix** (`mmaNewsScraper.ts`): every `await page.close()` was
   unguarded; when `page.close()` threw `ConnectionClosedError` *after* a source had
   already scraped its articles, the throw propagated out of the scraper fn and
   `scrapeAll`'s catch dropped that source entirely. First run lost **Sherdog (10)**
   and **Bleacher Report (4)** this way. Wrapped all 9 `page.close()` calls in
   `try/catch` so a dead connection can't discard scraped articles.

- ⚠️ Sherdog (10), Bleacher Report (4) → scraped fine but lost to the close bug

# 2026-06-02 (remote-control) — Start-time discovery + MVP live-tracker post-mortem

Two scraper issues raised re: **MVP "Han vs. Holm 2" (2026-05-30)**.

## Issue 1 — live tracker didn't fire Holm "up next" (DIAGNOSED, NOT FIXED)

DB-evidenced. Two bugs:
1. **Tracker isn't durable.** The Yahoo boxing tracker is a hand-run CLI loop.
   Automated `yahoo-narrative` actions stop at 01:57 UTC (9:57 PM ET, Robinson
   LIVE); Robinson/Serrano/**Han-Holm** all finished `manual`. The loop died
   ~10 PM ET when the prior session ended → nothing polled Yahoo when Serrano
   ended (~11:22 PM) / Holm walked out (~11:53 PM).
2. **Premature headliner fire.** `follow_notification_events`: Han/Holm WALKOUT
   dispatched at 23:46 UTC = **7:46 PM ET**, ~4h early. `detectAndNotifyLiveFight`
   picks the lowest-orderOnCard fight whose both surnames appear in the 6 newest
   posts; the blog's main-event hype posts make "han"+"holm" match all night.

Status: user redirected to Issue 2; **Issue 1 still open.** Fix = durable wiring
(Event.yahooLiveBlogUrl + lifecycle dispatch + VPS, mirror Sherdog) + exclude the
headliner from early narrative matching.

## Issue 2 — wrong event start time (FIXED, shipped `4fe6774`)

Root cause: **Tapology publishes only the main-card time** (8 PM ET) — verified
the event page + a bout page expose no prelim time. Real card opened ~3:30–5:15 PM
ET, so the event flipped LIVE hours late.

Built **`services/startTimeDiscovery`** (modeled on `broadcastDiscovery`): daily,
per upcoming event with no early-bell time → Brave search → Haiku 4.5 extracts
per-section times (ET-normalized, confidence-gated, grounded) → ET→UTC → write
`prelimStartTime`/`earlyPrelimStartTime` with ordering + provenance guards (never
fabricate, never clobber scraper/admin, floor 0.7). `getStartTime()` already picks
the earliest section, so events now flip LIVE at the real first bell.

Validated against Han vs Holm 2: extracted `prelims 3:30 PM ET` → wrote
`prelimStartTime 2026-05-30T19:30:00Z` @ conf 0.92. Batch dry-run gated out 0.45/0.5
events, applied Zuffa 0.75, didn't clobber BKFC.

Shipped: service `{extract,persist,run,README}` + `scripts/runStartTimeDiscovery.ts`
+ Event provenance columns + migration `20260602000000_add_start_time_discovery_fields`
(applied to prod) + `.github/workflows/start-time-discovery.yml` (daily 13:00 UTC)
+ best-effort tail step on `runAllOrganizationScrapers`.

# 2026-06-02 (evening, pt 2) — News on Home: UI polish, MMA Fighting fix, anti-clump

Follow-on to the news-aggregator revival earlier today. All mobile changes shipped
via OTA to the `production` branch (Android cold-start + TestFlight iOS now; App

### Mobile Home (`packages/mobile/app/(tabs)/home.tsx`)
1. **Latest News → vertical list with lazy loading.** Swapped the horizontal
   carousel for a vertical list of row cards (thumbnail + source + headline),
   backed by `useInfiniteQuery` (8/page) with `getNextPageParam` off
   `pagination.totalPages`.
2. **Fixed lazy-load jank by virtualizing.** First pass appended cards inside the
   page's plain `ScrollView` → every card + full-size remote image stayed mounted
   and each page append re-rendered the whole tree → very jumpy. **Converted Home
   from `ScrollView` to `FlatList`**: all prior sections moved into
   `ListHeaderComponent`, news articles became the virtualized `data` rows,
   lazy-load via `onEndReached` (0.6 threshold), `initialNumToRender/maxToRenderPerBatch=6`,
   `windowSize=9`, `removeClippedSubviews`. Images use `resizeMethod="resize"` to
   downsample large OG/CDN images during decode. Modal moved out as a sibling.
3. **"Did You Know?" rotation**: 10s → **15s**, and the slide swap now **cross-fades**
   (`Animated` opacity out/in, 350ms each) instead of a hard cut.
4. **Renamed section** "Most Hyped" → **"Hyped Upcoming Fights"**.

### MMA Fighting scraper repaired (`services/mmaNewsScraper.ts`)
It had rotted to 0 — the site is now **Next.js with hashed, rotating CSS class
names**, so the old `a._1ngvuhm0` / `div.ls9zuh9` selectors matched nothing.
Rewrote `scrapeMMAfighting()` to key off **stable structure**: the Duet
`.duet--content-cards--content-card` container + article URLs shaped
`/<section>/<id>/<slug>` (regex `^/[a-z0-9-]+/\d+/[a-z0-9-]+`); headline from the
title anchor's text with `img[alt]` fallback; image from `currentSrc`/`srcset`.
Validated live with a throwaway Puppeteer probe (12 articles, real URLs + CDN
images) before editing. GHA run confirmed **10 MMA Fighting articles** in prod —
all **6 sources now operational** (57 total articles across sources).

The 10 MMA Fighting articles all entered in one run (first success), so under the
old `createdAt desc` ordering they shared near-identical timestamps and **clumped**
in the feed. The mixed feed now pulls a recent pool (cap 300), groups by source
preserving recency, and **takes one article per source per round** before
paginating — sources alternate instead of bunching. `?source=` filter keeps
straight recency pagination. (Pushed `4aa6825`; verifying on Render.)
