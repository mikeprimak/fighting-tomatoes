## Shipped (code, pushed to main)

1. **Mobile event detail (`app/event/[id].tsx`) — column headers.** Added
   `ALL / HYPE` (left) + `MY / HYPE` (right) over the Main Card score columns,
   switching to `RATING` once the event is COMPLETED. New `colHeaderRow` styles
   align over the 48px aggregate/user squares of `FightDisplayCard`. (The
   alternate `app/(tabs)/events/[id].tsx` already had these but is essentially
   dead — almost all navigation goes to `/event/:id`.)

3. **Home "Recent Good Fights" — new logic + event-grouped UI.** New backend
   endpoint `GET /community/recent-good-fights`: COMPLETED fights from the last
   14 days with `averageRating > 7` and `totalRatings >= 3`, grouped by event
   (events ranked by their best fight), capped at 3 fights/event and 3 events,
   returned flat (each fight carries `.event`). Home now renders it with the same
   `HypedEventCard`/`HypedFightRow` used by Top Upcoming Fights, generalized with
   a `mode: 'hype' | 'rating'` prop (rating mode shows `averageRating` +
   `totalRatings`, past event date instead of "in N days"). Replaced the old
   `getTopRecentFights` week/month-fallback + flat `CompletedFightCard` list.

4. **Home "Highlighted Fighter" — top fight in border + next fight.** Replicated
   the `winner-predictions` branch treatment onto main: the top-rated fight now
   renders as an inline line *inside* the card border (was a separate
   `CompletedFightCard` outside it). Added a "Next fight" line below it.
   `/community/highlighted-fighter` now also returns `nextFight` (the fighter's
   next UPCOMING/LIVE bout).

5. **RAF 10 fixed.** Root cause: a phantom `Salkazanov vs O'toole` fight, marked
   COMPLETED with RAF 06's winner (`a14062f9`), had bled onto RAF 10
   (`eventId 17c2e271…`, fight `e5cb8c5f…`, 0 ratings/0 reviews). The real RAF 10
   order-8 bout is `O'toole vs Abdurakhmonov`. That one stray COMPLETED fight
   tripped the mobile `eventHasStarted` check → made the headliner (Downey vs
   Braunagel) show "up next" a week early. **Deleted the phantom row.** Also
   **hardened the guard**: `eventHasStarted` now keys off event-live / a LIVE
   fight only, not "any COMPLETED fight exists", so future bleed can't flip an
   upcoming card. Scanned all events: this was the only COMPLETED-on-UPCOMING
   instance.

6. **Web LiveFightCard weight class.** New `packages/web/src/utils/weightClass.ts`
   formats enums to normal case incl. the apostrophe (`WOMENS_STRAWWEIGHT` →
   "Women's Strawweight"); removed the `uppercase` CSS so it shows mixed case.

7. **CompletedFightDetailScreen AI cards.** The Story / The Stakes / The Outcome
   now render as bordered, titled cards matching UpcomingFightDetailScreen
   (were centered borderless italic text). "The Outcome" wraps the existing
   post-fight summary + bonuses + FOTY (still spoiler-reveal gated).

## Data fix (#2, partial — applied to prod DB)

- **UFC 268 order reversed.** `orderOnCard` was fully inverted (`[1]` = an early
  prelim, `[14]` = Usman vs Covington). Reversed to `newOrder = 15 - old`;
  Usman vs Covington now `[1]`. The app assumes `orderOnCard 1 = main event`
  everywhere (`mainCard = orderOnCard <= 6`, next-fight sort), so the inversion
  rendered the card upside-down.
- **Broader scope is large and NOT uniform.** Among 1095 legacy `scraperType=null`
  COMPLETED multi-fight events, a most-rated-fight-position heuristic flags
  ~470 as inverted but ~328 as already-correct (e.g. UFC 199/209/210/241/245 have
  the main event at `[1]`). A blanket flip would break the correct ones. Some
  rows also carry sentinel orders (99/115). **Decision pending** on how to fix
  the other ~470 (heuristic mass-flip vs. authoritative per-event reorder). See
  memory `legacy-event-order-inversion`.
