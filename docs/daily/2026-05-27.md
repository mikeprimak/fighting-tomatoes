## Pride 33 data fix — duplicate event + missing card

A user reported "There is no Alistair Overeem vs Shogun Rua at Pride 33" and
that the app showed **two** Pride 33 events.

### What was wrong

- **Event A** (`4fa7687e-5236-446e-895c-2bea48b11e2e`), dated **Feb 4, 2007** —
  held the real, well-rated **Nick Diaz vs Takanori Gomi** fight (18 ratings,
  avg 9.17) but on the **wrong date** and with no result set.
- **Event B** (`65ca9aed-8968-45ed-8d3a-48c5e4568772`), dated **Feb 24, 2007**
  (the correct date) — but **empty**, 0 fights. A phantom duplicate.

Pride 33 "The Second Coming" actually took place **Feb 24, 2007** at the
Thomas & Mack Center, Las Vegas. So Event A had the right fight + wrong date;
Event B had the right date + nothing on it.

### Fix (one Prisma transaction, against Render external DB)

1. **Deleted** the empty duplicate (Event B) — verified 0 fights / 0
   broadcasts / 0 sent-notifications first. Kept Event A so the existing
   Diaz/Gomi ratings stayed attached.
2. **Re-dated** Event A to 2007-02-24 + set venue `Thomas & Mack Center`,
   location `Las Vegas, Nevada`. (Had to delete B before re-dating A — the
   `@@unique([name, date])` constraint would otherwise collide.)
3. **Fixed Diaz vs Gomi**: `orderOnCard=2`, LIGHTWEIGHT, R2 1:46, result set
   to the official **No Contest (overturned)** (`winner="nc"`, matching an
   existing DB convention). Ratings preserved.
4. **Created** one missing fighter: **Rameau Thierry Sokoudjou**. All other 11
   card fighters already existed and were reused by ID.
5. **Added the remaining 8 bouts** with results (source: Wikipedia Pride 33):
   - #1 Dan Henderson def. Wanderlei Silva — KO (punches) R3 2:08 — MW title
   - #3 **Mauricio "Shogun" Rua def. Alistair Overeem — KO (punches) R1 3:37**
   - #4 Hayato Sakurai def. Mac Danzig — KO (punch) R2 4:01
   - #5 Sokoudjou def. Antonio Rogerio Nogueira — KO (punch) R1 0:23
   - #6 Sergei Kharitonov def. Mike Russow — Submission (armbar) R1 3:46
   - #7 Frank Trigg def. Kazuo Misaki — Decision (unanimous) R3 5:00
   - #8 James Lee def. Travis Wiuff — Submission (guillotine) R1 0:39
   - #9 Joachim Hansen def. Jason Ireland — Submission (armbar) R3 2:33

Final state: **one** Pride 33 event, Feb 24 2007, 9 fights in card order, all
COMPLETED with results. Verified post-run.

### Notes / caveats

- This was a **data fix only** — no code changes, nothing to commit/deploy.
  Temp inspection/rebuild scripts were written under `packages/backend/prisma/`
  and deleted after the run.
- **`James Lee`** (prelim #8) was matched to the existing generic
  `James Lee [MMA]` (`0ecb19c2…`); there are 40+ "Lee" fighters and this one is
  a plausible-but-unverified match for an obscure 2007 prelim fighter. Low risk
  (records aren't tracked on these historic rows — all show 0-0), but flagging.
- This duplicate-with-wrong-date pattern echoes the Matchroom phantom-event and
  Tapology event-dedup issues: a stale/empty event page coexisting with the real
  one. Worth a periodic sweep for `(name, date≠date)` Pride/legacy duplicates.

## Email verification: manual fix + stale-token error page

User `thepizzasport@gmail.com` (Android v2.0.1) reported the verification link
not verifying them. Diagnosed via `docs/playbooks/triage-email-verification.md`:
live token present, unverified, row updated ~16h prior — classic **stale-link**
case (a resend overwrote the token; they were clicking an older email's link).
Manually verified by hitting `GET /api/auth/verify-email` with the live token.
Second logged incident of this exact pattern (first: cray_zegyptian 2026-05-11).

### Root-cause fix: distinguish stale vs expired tokens in the error page

The playbook's "known gap": the error page showed the same generic "Verification
Failed" for stale / expired / consumed tokens, so users couldn't self-resolve.

- **Backend** (`auth.fastify.ts` `/verify-email`): when the live-token lookup
  misses, do a second lookup by token alone. Row still holds it → `TOKEN_EXPIRED`
  (real 24h elapsed). No row holds it → `TOKEN_STALE` (replaced by a newer resend,
  or already verified). Distinct codes + tailored copy. Both branches tested live
  against the Render DB (garbage token → STALE; reversible past-expiry token on
  test@ → EXPIRED, restored after).
- **Landing** (`packages/landing/verify-email.html`): branches on `code` —
  `TOKEN_STALE` shows "📬 Check Your Inbox" and **suppresses the Resend CTA**
  (resending is what created the dead link; steer to the latest email / sign-in).
  `TOKEN_EXPIRED` shows "⏳ Link Expired" + keeps Resend.
- **Web** (`packages/web/src/app/verify-email/page.tsx`): tailored error title to
  match; improved backend `error` text flows through automatically.
