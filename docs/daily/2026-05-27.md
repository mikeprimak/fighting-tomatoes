## Pride 33 data fix — duplicate event + missing card

A user reported "There is no Alistair Overeem vs Shogun Rua at Pride 33" and
that the app showed **two** Pride 33 events.

### What was wrong

- **Event A** (`4fa7687e-5236-446e-895c-2bea48b11e2e`), dated **Feb 4, 2007** —
  held the real, well-rated **Nick Diaz vs Takanori Gomi** fight (18 ratings,
  avg 9.17) but on the **wrong date** and with no result set.
- **Event B** (`65ca9aed-8968-45ed-8d3a-48c5e4568772`), dated **Feb 24, 2007**
  (the correct date) — but **empty**, 0 fights. A phantom duplicate.

Pride 33 "The Second Coming" actually took place **Feb 24, 2007** at the
Thomas & Mack Center, Las Vegas. So Event A had the right fight + wrong date;
Event B had the right date + nothing on it.

### Fix (one Prisma transaction, against Render external DB)

1. **Deleted** the empty duplicate (Event B) — verified 0 fights / 0
   broadcasts / 0 sent-notifications first. Kept Event A so the existing
   Diaz/Gomi ratings stayed attached.
2. **Re-dated** Event A to 2007-02-24 + set venue `Thomas & Mack Center`,
   location `Las Vegas, Nevada`. (Had to delete B before re-dating A — the
   `@@unique([name, date])` constraint would otherwise collide.)
3. **Fixed Diaz vs Gomi**: `orderOnCard=2`, LIGHTWEIGHT, R2 1:46, result set
   to the official **No Contest (overturned)** (`winner="nc"`, matching an
   existing DB convention). Ratings preserved.
4. **Created** one missing fighter: **Rameau Thierry Sokoudjou**. All other 11
   card fighters already existed and were reused by ID.
5. **Added the remaining 8 bouts** with results (source: Wikipedia Pride 33):
   - #1 Dan Henderson def. Wanderlei Silva — KO (punches) R3 2:08 — MW title
   - #3 **Mauricio "Shogun" Rua def. Alistair Overeem — KO (punches) R1 3:37**
   - #4 Hayato Sakurai def. Mac Danzig — KO (punch) R2 4:01
   - #5 Sokoudjou def. Antonio Rogerio Nogueira — KO (punch) R1 0:23
   - #6 Sergei Kharitonov def. Mike Russow — Submission (armbar) R1 3:46
   - #7 Frank Trigg def. Kazuo Misaki — Decision (unanimous) R3 5:00
   - #8 James Lee def. Travis Wiuff — Submission (guillotine) R1 0:39
   - #9 Joachim Hansen def. Jason Ireland — Submission (armbar) R3 2:33

Final state: **one** Pride 33 event, Feb 24 2007, 9 fights in card order, all
COMPLETED with results. Verified post-run.

### Notes / caveats

- This was a **data fix only** — no code changes, nothing to commit/deploy.
  Temp inspection/rebuild scripts were written under `packages/backend/prisma/`
  and deleted after the run.
- **`James Lee`** (prelim #8) was matched to the existing generic
  `James Lee [MMA]` (`0ecb19c2…`); there are 40+ "Lee" fighters and this one is
  a plausible-but-unverified match for an obscure 2007 prelim fighter. Low risk
  (records aren't tracked on these historic rows — all show 0-0), but flagging.
- This duplicate-with-wrong-date pattern echoes the Matchroom phantom-event and
  Tapology event-dedup issues: a stale/empty event page coexisting with the real
  one. Worth a periodic sweep for `(name, date≠date)` Pride/legacy duplicates.

## Email verification: manual fix + stale-token error page

User `thepizzasport@gmail.com` (Android v2.0.1) reported the verification link
not verifying them. Diagnosed via `docs/playbooks/triage-email-verification.md`:
live token present, unverified, row updated ~16h prior — classic **stale-link**
case (a resend overwrote the token; they were clicking an older email's link).
Manually verified by hitting `GET /api/auth/verify-email` with the live token.
Second logged incident of this exact pattern (first: cray_zegyptian 2026-05-11).

### Root-cause fix: distinguish stale vs expired tokens in the error page

The playbook's "known gap": the error page showed the same generic "Verification
Failed" for stale / expired / consumed tokens, so users couldn't self-resolve.

- **Backend** (`auth.fastify.ts` `/verify-email`): when the live-token lookup
  misses, do a second lookup by token alone. Row still holds it → `TOKEN_EXPIRED`
  (real 24h elapsed). No row holds it → `TOKEN_STALE` (replaced by a newer resend,
  or already verified). Distinct codes + tailored copy. Both branches tested live
  against the Render DB (garbage token → STALE; reversible past-expiry token on
  test@ → EXPIRED, restored after).
- **Landing** (`packages/landing/verify-email.html`): branches on `code` —
  `TOKEN_STALE` shows "📬 Check Your Inbox" and **suppresses the Resend CTA**
  (resending is what created the dead link; steer to the latest email / sign-in).
  `TOKEN_EXPIRED` shows "⏳ Link Expired" + keeps Resend.
- **Web** (`packages/web/src/app/verify-email/page.tsx`): tailored error title to
  match; improved backend `error` text flows through automatically.

## Org filter: selected promotions float to front (user request)

User feedback (kevinsartori, Android v2.0.1): wanted to pin/reorder favorite
orgs in the filter top bar so they appear first.

### What shipped
- `store/OrgFilterContext.tsx` — added `orgTouchOrder` (most-recently-toggled
  first), persisted to AsyncStorage key `events_org_touch_order`; updated on
  every org tap in `handleOrgPress`. Exposed via context.
- `components/OrgFilterTabs.tsx` — renders a `orderedOrgs` list computed on
  **screen focus** (`useFocusEffect`), not on tap, so pills don't jump under
  the finger mid-interaction. Order rule: selected orgs first; within each
  (selected / non-selected) group, most-recently-toggled first; untouched orgs
  keep registry order. `selectedOrgs`/`orgTouchOrder` read via refs so the sort
  only fires on focus (and once on registry hydration).
- Mobile-only; no backend/schema. Shipped via EAS update (prod, both runtimes:
  iOS 2.0.1 group `5250ff39`, Android 1.0.0 group `9e1de14b`).

### Git hygiene note

unrelated, complete email-verification feature (TOKEN_STALE/EXPIRED across
`auth.fastify.ts` + landing `verify-email.html` + web `verify-email/page.tsx`

### Follow-up fixes (same evening, live-tested by Mike)
1. **Re-sort fired on tap, not focus.** `availableOrgs` was rebuilt as a new
   array each provider render, so a tap (updates `selectedOrgs`/`orgTouchOrder`
   in the provider) handed `OrgFilterTabs` a fresh array reference →
   `useFocusEffect([availableOrgs])` re-ran the sort immediately. Fixed by
   `useMemo`-ing `availableOrgs` on `registryData` (commit `13f8030`).
2. **Scroll position persisted per screen.** Each screen owns its own
   `OrgFilterTabs` instance and retained its horizontal scroll offset, so
   returning to a screen showed wherever it was last scrolled. Added a
   `ScrollView` ref + `scrollTo({ x: 0 })` in the focus effect so every screen
   load snaps to the left edge (commit `2590432`).

All three commits shipped via EAS update (prod, both runtimes). Verified
working in-app by Mike.

## Legacy-import data integrity: Moreno/Pride 31 mislink + 158 Bellator 1899 dates

User reported the app showed **Alistair Overeem vs Brandon Moreno at Pride 31,
Feb 26 2006** — impossible (Moreno was 12) and worried it signalled a systemic
problem. Investigated; root cause is entirely the **legacy fightingtomatoes
import** (`scraperType=null`, `completionMethod=migration-fix`, created Feb 2026),
NOT the live scraper / Tapology pipeline.

### Scope audit
Per-fighter "gap between earliest and 2nd fight" sweep over all 14,483 fights
surfaced two distinct buckets:
- **Bucket A (the big one):** 158 Bellator events (`Bellator 4`…`276`) all dated
  `1899-11-30T05:00:00Z` — a sentinel/zero date from a date-parse failure in the
  import. 460 fights inherit it. Fighters are correct; only the date is garbage.
- **Bucket B:** 63 fighters with a real-dated but isolated early stray fight. A
  *mix* — most are genuine veteran careers (old UFC `backfill-ufcstats` fights),
  a few are true name-collision mislinks like Moreno. Can't batch-fix; needs
  per-case judgment.

### Fixes (DB-only, Render external)
1. **Moreno/Pride 31:** real opponent was **Sergei Kharitonov** (Overeem won TKO
   knees R1 5:13 — Wikipedia/Tapology/ESPN). The import had the wrong opponent
   name outright (not a collision). Relinked fight `66423874…` fighter2
   Moreno→Kharitonov, set winner=Overeem, method `TKO (knees)`, R1, 5:13. 6 user
   ratings preserved. (Kharitonov not already on the event → no dup.)
2. **158 Bellator dates:** built a number→date map by parsing the raw wikitext of
   *List of Bellator MMA events* (`?action=raw`) deterministically — `[[Bellator
   N]]` → next `{{dts|Y|Month|D}}`. 300 parsed, all 158 DB numbers matched (0
   missing), spot-checked. Updated each event to `<date>T05:00:00Z` (midnight-EST
   convention, matches Pride 31). Verified 0 events remain at ~1899.

### Left for a future pass
Post-fix re-run: **6 fighters** still have a ≥15yr earliest-fight gap — Bucket B
residue. Top mislink suspects: Saeed Hosseini (UFC 13 1997 → PFL 2025),
Adrian Serrano (UFC 26 2000 → 2026). Each needs a "was this fighter active then?"
check before touching.

### Notes
- Data fix only — no code changes, nothing to commit/deploy. Scripts were written
  under the OS temp dir, not the repo.
- Same legacy-import root as today's Pride 33 fix; both are the "periodic sweep
  for legacy date/identity anomalies" that section flagged. The gap-audit script
  is a reusable detector for the next sweep.

## Chael Sonnen missing headshot + web broken-image class fix

User: Chael Sonnen showed a silhouette on mobile but a **broken-image icon** on
web. Two-part task.

### Part 1 — restore the image (data)
`Fighter.profileImage` was `https://fightingtomatoes.com/images/fighters/ChaelSonnen.jpg`
→ **404** (rotted post-2026-04-17 host migration; same class as the
dead-legacy-URL lesson). Pulled the real headshot from the ESPN URL Mike
supplied (`a.espncdn.com/.../2335796.png`), uploaded to R2 via
`uploadFighterImage`, and updated the record to
`https://pub-7bef67fde2894e1eb3061dee36bde224.r2.dev/fighters/chael-sonnen-81e272e2.png`
(verified 200, 281KB). One-off script written under `prisma/` and deleted.

### Part 2 — why web broke instead of falling back (code)
Mobile renders headshots through `getFighterImage` + an `onError` handler that
swaps to the silhouette asset, so a dead URL degrades to the placeholder. Web
rendered a **bare `<img src={profileImage}>` with no `onError`**, so a
non-null-but-dead URL painted the browser's broken-image icon. The `{profileImage
? <img> : <initials>}` guard only covers *null*, never *dead*.

Fix (commit `1aeca0f`, auto-deploys): new shared client component
`packages/web/src/components/FighterAvatar.tsx` that falls back to initials when
the image is missing **or** fails to load. Tracks the *failed URL* (not a bool)
so recycled instances in keyed lists retry on a new `src`. Applied across all 10
headshot surfaces: fighter/fight detail, the 3 fight cards (Upcoming/Live/
Completed), 3 sidebar blocks (Popular/MightLike/FollowedStrip), search, and
followed-fighters. `tsc --noEmit` clean.

Note: `fight-of-the-night` and `weekly-hype` pages use `proxyImageUrl` + a
different render path (image-override editor), left untouched.

### Part 3 — full sweep for the dead-URL class (audit only, no backfill)
Probed all 1,496 non-R2 fighter `profileImage` URLs (skipped the 4,062 R2 ones —
live by construction). Reusable auditor: `prisma/audit-fighter-images.ts`
(`--probe`); dead list persisted to `prisma/dead-fighter-images.json` (both kept
locally, untracked, for a later backfill).

**218 dead:**
- **206 on `fightingtomatoes.com` — every one 404.** Whole legacy host is rotted.
- **12 on `images.tapology.com` — all 403, and still 403 with a browser UA +
  `Referer: tapology.com`** (verified — not a recoverable hotlink block; those
  image IDs are gone/private). The other 1,134 Tapology URLs are alive.

**178 / 218 are on fighters with ≥1 rated fight** — not all obscure. Top by
engagement: Junior Dos Santos (216 ratings), Sergey Spivak (146), Fedor
Emelianenko (65), + Roberto Soldic, Vadim Nemkov, Stamp Fairtex, Tawanchai,
Aung La N Sang. Mostly **Bellator / ONE / PFL / RIZIN** stars plus a regional/
Muay-Thai tail.

**Backfill gap:** the existing `ufc-headshot-backfill.yml` (ufc.com → R2) only
covers **3** of these — its candidate query requires a UFC event. The other
**215 are non-UFC**, so no automated path exists today; they'd need a new source
(Tapology page scrape is the obvious one — 1,134/1,146 Tapology URLs are live).

**Decision (Mike): stop at the audit.** The web fallback already turned all 218
from broken icons into clean initials, so this is quality, not breakage. No
backfill built. Dead list saved for a future pass.
