## Home event band: "No events this weekend" / web empty on Sundays

**Symptom:** Mobile home "This Weekend" section showed "No events this weekend";
web `WeekendEventsSection` rendered nothing at all.

**Root cause:** Both surfaces shared the same window — "today up to the Monday
that starts next week" (rest of the current Mon–Sun calendar week). On a **Sunday**
that collapses to a single day:
`daysUntilNextMonday = (1 - 0 + 7) % 7 = 1` → window = `[Sun, Mon)` = just today.
Cards cluster Fri/Sat, so by Sunday the weekend is over and the next card is
outside the one-day window. Today (Sun Jun 21) had no events; soonest was BKFC
Fri Jun 26. Mobile shows the empty state; web returns `null` when the window is
empty.

**Fix:** Replaced the week-bounded window with a rolling **7-day** window
(`todayKey + 7*DAY_MS`, today + next 6 days) in:
- `packages/mobile/app/(tabs)/home.tsx`
- `packages/web/src/components/home/WeekendEventsSection.tsx`

Also renamed misleading placeholder copy on mobile: "This Weekend" → "This Week",
"No events this weekend" → "No events in the next 7 days". Per-day headings
("Events Today/Tomorrow/<weekday>") were already day-derived and unchanged.

**Verification:** Web `tsc --noEmit` clean. Window now surfaces Jun 26 (BKFC, ONE)
and Jun 27 (UFC Fiziev vs. Torres, ONE, PFL).

### Testability blocker hit + fixed (migration ordering)
Mike tested on the Expo dev build and the Notifications screen showed "Couldn't
load notifications" (Expo surfaced it as an auth/fetch failure). **Root cause:**
`GET /api/notifications` (and the dispatch snooze filter) reference the new
`notificationsSnoozedUntil` column, but the migration hadn't been applied to the
**prod** DB — and the dev backend talks to prod (per the never-use-local-DB
rule). So the list query 500'd. The local Prisma client was also stale (the
earlier `generate` wrote the `.d.ts` but failed the engine-DLL rename under a
Windows file lock held by the running backend).

**Fix:** (1) `prisma migrate deploy` → applied `20260621000000` to prod (safe,
additive, no shadow DB); (2) stopped the 3008 backend (PID held the engine DLL)
→ `prisma generate` clean → restarted `pnpm dev`; (3) verified `GET
/api/notifications` → **200** with a real row + `unreadCount` + `snoozedUntil`.
Lesson saved: [[lesson_dev_backend_prod_db_unmerged_migration]].
