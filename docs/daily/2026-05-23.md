### Vercel auto-deploy flakiness (root-caused mid-session)

Two commits in this session got swallowed by Vercel's auto-deploy webhook
(`b56639f`, `3916d2e`). Initial hypothesis (debounce on close-arriving
commits) was wrong. Root-cause turned out to be two separate things:

1. **Ignored Build Step working as designed.** The web project has
   `git diff HEAD^ HEAD --quiet ./` configured as the Ignored Build Step.
   Run from `packages/web`, it correctly cancels deploys when the commit
   doesn't touch `packages/web`. That's why my empty re-trigger commit
   (`1bd5d45`) and the daily-doc commit (`f9470e7`) appeared as `Canceled`
   in `vercel ls` ‚Äî they touched zero `packages/web` files, so IBS killed
   them. Behavior is correct.

2. **Genuine webhook drop on `b56639f` + `3916d2e`.** Both commits touched
   `packages/web` heavily, yet `vercel ls` showed NO entry for either
   (not Canceled, not Ready ‚Äî just absent). GitHub‚ÜíVercel webhook
   delivery was lost. Not a config issue, just rare flakiness.

Recovery that worked: `b8ef2d6` ‚Äî a real `packages/web` change
(`aria-label="About you"` on the sidebar `<aside>`). Fresh webhook fired,
IBS let it through, build picked up HEAD cumulatively ‚Äî so
`b56639f` + `c798c8b` + `3916d2e` + the recovery commit all went live
together as `web-n80ajek86`.

Updated lesson `lesson_vercel_webhook_skip_close_commits` to reflect the
correct diagnosis. The previous "use empty commit to recover" advice was
wrong ‚Äî empty commits will always be canceled by IBS. Must touch

### The bug: three independent hide-lists, all stripping Matchroom

This is the lesson worth banking. Hiding one promotion required four
coordinated edits because there are three independent client/server
filter layers + a registry flag:

| Layer | File | Mechanism |
|---|---|---|
| Backend API strip | `packages/backend/src/config/hiddenPromotions.ts` | `HIDDEN_PROMOTIONS = ['MATCHROOM']` ‚Üí all `/api/events`, `/api/search`, `/api/community` strip rows whose promotion contains 'MATCHROOM' |
| Backend registry | `packages/backend/src/config/promotionRegistry.ts` | `userVisible: false` ‚Üí `/api/promotions` excludes MATCHROOM from the runtime org list |
| Mobile filter | `packages/mobile/store/OrgFilterContext.tsx` | Local `HIDDEN_ORGS = ['MATCHROOM']` + missing from bundled `ORGANIZATIONS` fallback |
| Web filter | `packages/web/src/lib/orgFilter.tsx` | Same pattern as mobile |
| Mobile screen | `packages/mobile/app/(tabs)/top-fights/index.tsx` | Hard-coded `promotion.includes('MATCHROOM')` filter that pre-dated the OrgFilterContext refactor |

Each layer was independently load-bearing. Unhiding required emptying
HIDDEN_PROMOTIONS, flipping `userVisible: true`, emptying both
HIDDEN_ORGS arrays, adding 'MATCHROOM' to both bundled ORGANIZATIONS
arrays, and removing the dead top-fights screen filter.

### Fix 1 ‚Äî Auth-race on user-data queries (commit `33b2208`)

Closes #10 + #12 + #13.

Root cause for #10 + #12: the search query (`['search', q]`) and event
detail's fights query (`['eventFights', id, isAuthenticated]`) fired on
first paint **before** `AuthProvider.tryRefresh()` finished. The
unauthenticated response from `/api/search` / `/api/fights?eventId=` omits
`userRating` / `userHypePrediction`. That empty result got cached, and
the post-auth invalidation via `queryClient.invalidateQueries(['search'])`
/ `['eventFights']` didn't reliably refetch under React 19 / Next 16
Suspense semantics.

Verified independently with curl: both endpoints return user-scoped fields
when the Authorization header is present (minted a 5min JWT for
avocadomike's user ID and hit `/api/search?q=adesanya` ‚Äî 9 of 20 fights
came back with `userRating`).

Frontend fix:
- **Search page**: `enabled: !authLoading && query.length >= 2` so the
  query waits for the access token to be ready before firing.
- **Event detail**: dropped `isAuthenticated` from the queryKey (was
  causing key-flip to a new query that re-applied SSR `initialFights` as
  fresh data and skipped refetch). Hardcoded `includeUserData: true`,
  added `initialDataUpdatedAt: 0` so SSR paint is instant but treated as
  stale, plus `enabled: !authLoading`.

For #13: `HypeFightModal` + `RateFightModal` were missing `['eventFights']`
from their post-mutation invalidation lists. So submitting a rating on
the event detail screen waited for the 30s `refetchInterval` (or 5min
`staleTime`) to reflect ‚Äî felt like ~60s. Added.
