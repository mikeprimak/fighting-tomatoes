## Changes Made

### Top Rank Scraper: Real Fix for Reappearing "Zayas vs Ennis" (commit f54b12d)

**What happened:** The same bogus "Zayas vs Ennis" event we cleaned up yesterday (2026-04-13) came back this morning. The direct toprank.com scraper ran at its 00:00 UTC cron and recreated the event with `date = 2026-04-14T02:13:33Z` (scraper run time) and 2 historical fights attached.

**Why yesterday's fix didn't work:** Yesterday I added `if (!eventDate) continue;` guards in `topRankDataParser.ts`, but `parseTopRankDate()` never returned a falsy value — both its fallback branches (empty string, unparseable input) returned `new Date()`. The guard was dead code. The broken toprank.com scraper kept feeding in empty `dateText`, the parser silently substituted today, and the event was upserted with today's date.

**Fix in `topRankDataParser.ts`:**
- Return type changed to `Date | null`.
- Empty/whitespace `dateText` → returns `null`.
- Unparseable date (no regex match, `new Date(dateText)` NaN) → returns `null`.
- Dedup loop now skips any event with `parsedDate === null` and logs a warning. The second call site inside `importTopRankEvents` still has its defensive `if (!eventDate) continue;` as a safety net.

**Disabled the daily cron (`.github/workflows/toprank-scraper.yml`):**
- Removed `schedule: cron: "0 0 * * *"`. Kept `workflow_dispatch` for manual runs.
- Rationale: toprank.com's page structure changed and the direct scraper can no longer parse event dates. Every daily run was pure liability. The Tapology Top Rank scraper (`toprank-tapology-scraper.yml`, daily at 15:15) covers Top Rank events correctly and is now the sole automated source.

**Cleanup:** Deleted the resurrected event `46fe298e-31d9-4536-98cd-8ed1e2e6570b` (2 fights removed) directly via Prisma.

**Remaining correct Zayas events in DB:**
- `Zayas vs. Boots` (Top Rank, 2026-06-27) — Tapology-sourced, correct
- `Zayas vs. Ennis` (Matchroom Boxing, 2026-06-27) — existed pre-issue, correct

### Implications going forward

## Evening Session: Google Login Crash + Sentry

### What happened
A user reported the app crashed when tapping "Sign in with Google." No stack trace, device info, or version was available — which exposed the bigger problem: we have no crash reporting at all.

### Mobile: Google login crash investigation
Code audit of `hooks/useGoogleAuth.ts` surfaced plausible culprits:
- `GoogleSignin.configure()` runs at module import time — native-module issues there could crash before React mounts.
- Unconditional `GoogleSignin.signOut()` before every sign-in (line 61) can throw on Android.
- `hasPlayServices({ showPlayServicesUpdateDialog: true })` on devices without Play Services (Huawei, some OEMs) is a classic hard-crash vector.
- Most likely root cause: Android user without current/any Play Services.

No direct fix shipped — instead, installed Sentry so the next occurrence gives us a stack trace.

### Mobile: Added Sentry crash/error reporting
- `npx expo install @sentry/react-native` → pinned at `~7.2.0`.
- `app.json`: plugin `"@sentry/react-native"` auto-added.
- `app/_layout.tsx`: `Sentry.init({ dsn: process.env.EXPO_PUBLIC_SENTRY_DSN, ... })` guarded behind DSN presence so dev without a DSN is a no-op. Wrapped the default export with `Sentry.wrap()` for touch/nav breadcrumbs.
- `store/AuthContext.tsx`: added `tagSentryUser()` helper + `useEffect` on `user?.id` that calls `Sentry.setUser({ id, email, username })` on login, `Sentry.setUser(null)` on logout. Centralized so every auth path (email, Google, Apple, refresh) tags automatically.
- `hooks/useGoogleAuth.ts`: `Sentry.captureException(err, { tags, extra })` in the catch block, skipped for user-cancelled sign-ins.
- `release` tag on init uses `version+buildNumber/versionCode` so crashes group by build.

Type-checked — no new errors (the two `_layout.tsx` TS errors for `gcTime` and `headerBackTitleVisible` are pre-existing).
