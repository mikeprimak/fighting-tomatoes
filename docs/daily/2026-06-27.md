## Fixed: mobile fight-detail showed only 1 comment when 3 existed

Mike reported the Fiziev vs Torres completed-fight detail screen showed only his 1
comment, while web showed all 3.

**Diagnosis — stale React Query cache, not a data/backend bug.** Verified:
- DB + production endpoint `GET /api/fights/65db0c0a-.../reviews` both return all 3
  reviews (all `isHidden:false`, `parentReviewId:null`, distinct users), with top-level
  `userId` + `content`. Web and mobile hit the *same* endpoint; mobile filters out the
  user's own (`review.userId !== user?.id`) and renders it separately from `fight.userReview`.
- Timestamps were the tell: Mike's comment 19:00:09, the other two at 19:25 and 19:37 —
  posted *after* he loaded the screen. The reviews `useInfiniteQuery` had `staleTime:30s`
  but no `refetchOnMount:'always'` (the adjacent stats query has it) and no focus refetch,
  so a still-mounted screen kept the 19:00 snapshot. Web showed 3 because it loaded fresh.

**Fix** (`packages/mobile/components/CompletedFightDetailScreen.tsx`):
- Added `refetchOnMount: 'always'` to the `['fightReviews', fight.id]` infinite query.
- Added a `useFocusEffect(() => refetchReviews())` (imported `useFocusEffect` from
  `expo-router`, already used across ~9 screens) so returning to a back-stacked screen
  pulls comments others posted while it was open/backgrounded.

Mobile JS only — reaches devices via an OTA `eas update`/next build, not yet published
(awaiting Mike's go). Immediate device recovery: reopen the fight screen / relaunch app.

## Fixed: home "Event Last Night" mislabeled a card that ran earlier today

Mike reported the home "Event Last Night" section showed a card that actually ran
*earlier today*. Root cause: the section qualifies any UFC card whose UTC day is
**today or yesterday** (`daysSince >= 0 && daysSince <= 1`, which deliberately catches
the Saturday-night → Sunday-morning case), but the heading was a hardcoded literal
`"Event Last Night"` — so a same-day card (`daysSince === 0`) got the wrong label.

**Fix** — derive the title from the freshest card's `daysSince` (list is most-recent-first):
`daysSince <= 0` → `"Event Earlier Today"`, else `"Event Last Night"`. Qualifying window
unchanged. Shipped in both:
- `packages/web/src/components/home/WeekendEventsSection.tsx`
- `packages/mobile/app/(tabs)/home.tsx`

Commit `fix(home): label same-day UFC card "Earlier Today"...` on `main` (rebased over a
concurrent push, pushed as `85459ca9`). Web auto-deploys to goodfights.app; mobile change
ships with the next OTA/build. Note: this keys off `Event.date`'s UTC calendar day (the
placeholder), consistent with the other home sections — not the real local `mainStartTime`
instant; revisit if far-west-timezone users see a mismatch.

## Fixed: home "This week" band bled a full 7 days ahead (showed "next Friday")

Mike: on Saturday the band was surfacing events a week out (e.g. next Friday). Root cause:
the window was a rolling `today → today + 7 days`, so from Saturday it reached the
following Friday. He wants it scoped to the **upcoming weekend chunk, ending Monday**.

**Fix** — window now runs from today through the **coming Monday** (inclusive). Same edit
in both `WeekendEventsSection.tsx` (web) and `home.tsx` (mobile):

```
const daysUntilMonday = ((1 - now.getDay() + 7) % 7) || 7; // Mon → next Mon (7)
const windowEndKey = todayKey + (daysUntilMonday + 1) * DAY_MS; // through Monday, exclusive bound
```

Every weekday ends the window on Monday (verified): Sat spans Sat/Sun/Mon (excludes next
Fri ✅), Sun→Mon, Fri→Mon, Tue→Mon, etc. **Monday is the one exception** — it reaches the
*next* Monday (8 days) on purpose, so the band doesn't collapse to a single day (the exact
failure the old rolling-7 window was patching; see the prior in-code comment). Monday
inclusive also keeps Sunday-night cards that roll into Monday visible.

Shipped on `main`; web auto-deploys, mobile rides the next OTA/build.
