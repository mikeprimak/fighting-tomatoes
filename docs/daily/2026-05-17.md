## Fan DNA peek endpoint — kill the spinner

Continuation of Rewarding Users + AI Enrichment workstreams. End-of-day-2026-05-16 state had Fan DNA Phase 1 in prod with a visible spinner-then-load before the DNA line appeared in the reveal modal. Mike wanted no pause — the line populated at the same instant the reveal modal opens.

### Architecture

The delay was structural: today's flow opens the reveal modal immediately on Done tap → mutation hits backend → backend computes the DNA line → response comes back → modal swaps spinner for line. Network round-trip exposed by the spinner.

Fix: pre-compute the DNA line for every possible value (1-10) while the user is choosing, cache it client-side, on Done look up `lines[value-1]` and render instantly. The actual mutation still records the impression server-side for the exact line the user saw (no re-evaluation that could randomly diverge).

Mike's exact mental model: 10 evaluator runs (one per value), each producing a possibly-different winning trait/line — confirmed correct since the engine's value passes through trait scoring, so the winning trait can flip between value=2 and value=8 even when the underlying traits don't change.

### What shipped (NOT YET PUSHED)

All four edits are in working tree, untested in dev, uncommitted.

**Backend**
- `services/fanDNA/engine.ts` — `eventEvaluate` accepts `peek?: boolean`. Same flow as before but skips every `recordImpression` write when peek=true. New exported helper `recordCommittedDNALine(args)` records the impression for a pre-peeked line.
- `routes/fanDNA.ts` — new `GET /api/fan-dna/peek?action=X&surface=Y&fightId=Z`. Loops values 1-10 in `Promise.all`, returns `{ lines: Array<DNALine|null> }` of length 10 indexed by `value - 1`.
- `routes/fights.ts` — `UpdateUserDataSchema` + `CreatePredictionSchema` gain optional `dnaCommittedLine`. Rate + hype handlers: if `dnaCommittedLine` present → call `commitPrePeekedDNA` (records impression for the peeked line, echoes line back). If absent → fall through to existing `evaluateFanDNAInline`.

**Mobile**
- `services/api.ts` — new exported type `FanDNACommittedLine`. `updateFightUserData` + `createFightPrediction` accept optional `dnaCommittedLine`. New method `getFanDNAPeek({ action, surface, fightId })`.
- `components/UpcomingFightModal.tsx` — `useQuery` for peek (keyed by `['fanDNAPeek', 'hype', fightId]`, 60s staleTime) fires on modal open. `handleHypeSelection` looks up `peekedDna.lines[level-1]` and `setRevealDnaLine` eagerly. `hypeMutation` args now `{ hypeLevel, dnaCommittedLine? }`. `onSuccess` only overwrites `revealDnaLine` from server if no `dnaCommittedLine` was sent. `dnaLoading` is false when peek loaded.
- `components/CompletedFightModal.tsx` — same pattern keyed for rate.

Type-check on the changed files: zero errors. Pre-existing errors in unrelated files unchanged.

### Behavior matrix

| State | Peek loaded? | dnaCommittedLine sent? | UX |
|---|---|---|---|
| Happy path | ✅ | ✅ | Reveal modal opens with line, no spinner, no delay |
| Peek failed / slow / offline | ❌ | ❌ | Falls through to existing flow (spinner + server eval) |
| EXIT-quiet window | ✅ (returns nulls) | ❌ | Modal shows no DNA line, no spinner (matches engine intent) |
| Tap, change value, tap again | ✅ | ✅ (new) | `setRevealDnaLine` updates on each tap from the cached array |

### Subtle decisions

- **Peek does NOT update toggle-storm counter.** The counter checks `dNALineImpression` rows. Peek doesn't write any → counter only moves on real commits. That's correct; we don't want modal-opening to count as a "rating change."
- **Commit records the exact peeked lineKey.** This is the only way cooldown semantics stay coherent — if peek and commit ran independently, the displayed line could differ from the one the cooldown ledger tracks.
- **`dnaCommittedLine` validation deferred.** Server trusts the client's shape. Worst case a logged-in user fabricates a lineKey and gets a spurious impression row in their own ledger. Low risk, all-self-harm.
- **User-context optimization deferred.** Each peek runs 10 sequential `eventEvaluate` calls in `Promise.all`. Worst-case ~80 small indexed queries. The user's 3-15 second selection window absorbs this — no perf optimization until measurement shows we need it.

## Fan DNA tag-aware personality — `style-clash` (Phase 2 trait #1)

First tag-aware Fan DNA trait. Reads `Fight.aiTags.styleTags` and fires when the AI extractor framed the fight as a stylistic contrast (any `"X vs Y"` tag — "striker vs grappler", "grappler vs striker", etc.). Responds to both rate and hype with separate copy pools per (action × user-value tier).

### What shipped

- `packages/backend/src/services/fanDNA/traits/style-clash/trait.ts` — clash detector (`/\b(?:vs\.?|versus)\b/i`), confidence floor 0.5, score 75.
- `packages/backend/src/services/fanDNA/traits/style-clash/copy.ts` — 6 copy keys × {soft, humor}: `{action}-clash-{high|mid|low}` where high = value≥7, low = value≤3, mid in between.

Registry auto-discovers — no central wiring change. Trait family `affinity` (sits next to org-affinity).

### Scoring placement

Score 75 — beats rating-bias mild (72) but loses to dramatic single-event deltas (rating-bias big 88, trailblazer first-ever 95, promotion-debut 82, hype-accuracy hot-take 95). When a clash fight has neither a hot rating nor a special-moment line, style-clash takes the modal.

### Verified on MVP MMA 1 (eventId `8a9ead55-657b-4175-845b-b63829f46581`)

- 9/49 fights enriched (matches handoff count); 2/9 have a clash pattern.
- **Rousey vs Carano** — `grappler vs striker` (conf 0.85)
- **dos Santos vs Despaigne** — `striker vs grappler` (conf 0.75)
- Simulated all 6 (action, tier) combos against Rousey/Carano — clean interpolation, sentences read naturally without bare-quoting the tag.

Examples (Rousey/Carano):
- hype/9 → *"A 9 on a grappler vs striker. You came for the clash."*
- rate/10 → *"A 10 on a grappler vs striker. The clash delivered."*
- rate/3 → *"A 3 on a grappler vs striker. The clash did not land."*

### Operating principle held

Never bare-quote the tag — every template interpolates it into a worms-tone sentence and pairs it with the user's number. No template starts with `{styleTag}` (tags arrive lowercase and would break sentence flow).

## Fan DNA tag-aware personality — `stakes-aware` (Phase 2 trait #2)

Second tag-aware Fan DNA trait. Reads `Fight.aiTags.stakes`, normalizes the matched phrase to a short label, and fires the same 3-tier × 2-action copy structure as `style-clash`.

### What shipped

- `packages/backend/src/services/fanDNA/traits/stakes-aware/trait.ts` — confidence floor 0.5, score 75, priority-ordered category detection.
- `packages/backend/src/services/fanDNA/traits/stakes-aware/copy.ts` — 6 copy keys × {soft, humor}: `{action}-stakes-{high|mid|low}` where high = value≥7, low = value≤3, mid in between.

Registry auto-discovers. Family `affinity`.

### Category labels (priority order)

| Detector | Label | Example real-data hit |
|---|---|---|
| `\btitle\b(?! conversation)` / `championship` / `belt` | `title fight` | "Vacant WBO super middleweight title" |
| `comeback` / `return(ing|s)` | `comeback` | "Henderson's return to competition after 34-month layoff" |
| `debut` | `debut` | "US debut for Parnasse" |
| `cross.sport` / `novelty` | `cross-sport showcase` | "Cross-sport novelty: kickboxing GOAT vs. boxing undisputed champion" |
| `main event` | `main event` | "Main event spotlight" |
| `ranking` | `ranking shake-up` | "Ranking implications" |

All labels begin with a consonant so `a {stakes}` reads grammatically every time. `title conversation` is explicitly excluded — it appears only in ranking-implication stakes, not actual title fights.

### Scoring placement

Score 75 — same tier as `style-clash`. When both fire on the same event, the engine's tie-break (registry insertion order) picks one; copy quality is flavour-equivalent.

### Verified against Render data (52 enriched fights, 20 with stakes)

Simulated 6 candidate fights × 2 actions × 3 value tiers under peek mode (no impressions written). On fights where `trailblazer` (score 95) didn't dominate, `stakes-aware` won 6/6:

- Mgoyan vs Morales (`Ranking implications`) — rate/9 → *"A 9 on the ranking shake-up. Lights on, score up."*
- Jackson vs Creighton (`Ranking implications`, user already rated) — hype/2 → *"A 2 on the ranking shake-up. Big stage, small interest."*
- All 6 cells (hype/rate × 9/5/2) rendered cleanly; no awkward articles or broken interpolations.

### Decisions made this slice

- **`rematch-fan` deferred.** Audit of all 52 enriched fights returned zero phrases containing rematch/trilogy/unfinished. Building it now ships a silent trait; better to revisit when the enrichment prompt is updated to flag rematches explicitly, or when a recurring-matchup card lands organically.
- **`undefeated` skipped from stakes patterns.** Single-fight occurrence and the label "0-loss record on the line" doesn't slot cleanly into templates. If undefeated phrases proliferate later, add a dedicated label with copy-key suffix.
- **No category-specific copy keys.** Mirrors `style-clash` — one copy key per (action, tier), label interpolated. Avoids combinatorial explosion. The label phrasing carries the category vibe.

## AI enrichment — pipeline rewrite + admin widget

End-of-day pickup from `docs/HANDOFF-ai-enrichment-mvp-2026-05-17.md`. Goal was "hands-off MVP today" — confirm cron is healthy, chase 2 failed-match events, decide observability, audit per-promotion coverage.

### Diagnosis: handoff was wrong about the failure mode

The handoff blamed a surname-pair edge case in `persist.ts` for the BKFC Palm Desert + Jones vs Carranza 2 "extracted but didn't match" cases. Wrong:

- Jones vs Carranza 2 + BKFC 90 didn't even *run* today — they were in the skipped list (out-of-window or recently-enriched).
- The actual 2026-05-17 14:51 UTC run enriched 4 events: Golden Boy (0 extracted), **BKFC Palm Desert** (2 extracted, 0 matched), Gold Star Usyk vs Rico (10/10 ✅), PFL Brussels 2026 (10 matched + 2 ghost).

For BKFC Palm Desert: `event.ufcUrl = https://www.bkfc.com/events/bkfc-palm-desert`. The dispatch in `enrichOneEvent.ts:60` only fetches a structured backbone when the URL matches `ufc.com` or `tapology.com`. Everything else falls through to editorial-only — Brave search returned articles about *other* BKFC events (Till/Chalmers debut announcement at BKFC 90 Birmingham, Alex Terrible's post-fight KO writeup). The LLM then put those fights on Palm Desert's card. Two hallucinations, none matched the actual 13 DB fights.

**Blast radius:** ran a bucket query against 66 UPCOMING events:

| Backbone | Count |
|---|---|
| UFC.com (Puppeteer) | 7 |
| Tapology.com | 9 |
| Has backbone | **16** |
| BKFC.com (no fetcher) | 11 |
| Promotion-specific (RAF, PFL, ONE FC, OKTAGON) | 39 |
| Editorial-only | **50** |

76% of upcoming events had no structured backbone. The hallucination on Palm Desert was the surfaced symptom of that.

### Fix: DB is the authoritative card source

Mike's suggestion: "use my database as the structured card source." Right call — we already have authoritative fight lists for every UPCOMING event. Fetching them externally and surname-matching back was redundant work that *introduced* the failure mode (LLM had room to invent fights).

New flow (commit `2a01f76`):
1. `loadCard(prisma, eventId)` returns `{ fightId, fighter1, fighter2, weightClass, cardSection, orderOnCard, isMainEvent, isTitle }` per UPCOMING fight.
2. `extractFightEnrichment` now takes the card list as input. System prompt says "enrich each fightId" with strict instruction to never invent fights and to OMIT records when editorial doesn't speak to that fight.
3. `persist.ts` writes by fightId — surname matcher deleted entirely. Records with unknown fightIds are dropped as ghosts.
4. UFC.com / Tapology page text still fetched as *supplementary editorial*, not as the card source.

Side benefit found in QA: old pipeline's "10/10 wrote" on PFL Brussels was actually 9 empty padded records + 1 real one (Habirora vs Henderson). New pipeline emits records only when narrative is grounded — honest numbers. Fan DNA consumers unaffected because the padded `styleTags/stakes` were empty anyway.
