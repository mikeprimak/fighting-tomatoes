### Top Rank Scraper: Reappearing "Zayas vs Ennis" Event (commit e53db57)

**Problem:** User reported "Zayas vs Ennis" event kept reappearing after manual deletion — wrong date (Mon Apr 13 instead of Jun 27 2026), contained a historical Zayas vs Fortea fight from 2023.

**Root cause:** The direct toprank.com scraper (`scrapeAllTopRankData.js`) is broken — toprank.com's page structure changed so dates can't be parsed. `dateText` comes back as `""`, and events get created with `new Date()` (scraper run time) as the event date. Location also garbled (`"New York CityComing up next"`). Fights scraped via generic "vs" regex picked up historical fights.

**Compounding:** The two Top Rank scrapers (direct toprank.com at midnight, Tapology at 3:15 PM) were in a daily cancel/un-cancel tug-of-war. Direct scraper found 1 event → cancelled all others. Tapology found 27 events (going back to Aug 2024) → un-cancelled them.

**Fixes in `topRankDataParser.ts` and `scrapeTopRankTapology.js`:**
- Skip events with unparseable dates (prevents broken events from toprank.com)
- 14-day max-age filter on imports (stops re-importing 20+ old events daily)
- Event-level cancellation now requires ≥3 scraped events and respects the date window (prevents tug-of-war)
- Tapology scraper skips old events at scrape time (saves ~20 page loads/run)

**Cleanup:** Deleted broken event via admin API:
`DELETE /api/admin/delete-event/0a85217a-ce44-4860-8177-046466aa1d90` → 2 fights also removed.

**Remaining:** "Zayas vs. Boots" (Jaron "Boots" Ennis, Jun 27 2026) still exists correctly from the Tapology scraper. Has some CANCELLED fights from other orgs leaked in historically, but they're hidden in the app.

### Feedback Reply System (admin panel)

**Problem:** Users submit feedback via the profile screen, admin receives notification emails, but there was no way to respond — no reply channel, no tracking of whether a reply was sent.

**Built:** Admin-panel-based reply flow that sends email from `contact@goodfights.app` and logs the reply to the DB.

**Files changed:**
- `prisma/schema.prisma` — added `replyBody`, `repliedAt`, `repliedBy` to `UserFeedback`
- `prisma/migrations/20260413000000_add_feedback_reply_fields/migration.sql` — new migration
- `src/utils/email.ts` — new `EmailService.sendFeedbackReply()` using existing nodemailer transporter (no Resend — reuses SMTP creds). Explicit `from: "Good Fights Support <contact@goodfights.app>"` + `replyTo: contact@goodfights.app` so user's reply lands in that inbox.
- `src/routes/admin.ts` — new `POST /api/admin/feedback/:id/reply`: validates, sends email, stores reply on the row, marks `isRead=true`
- `public/admin.html` — Reply button on feedback cards, modal with original message + textarea, "✉️ Replied Xm ago" badge, displays sent reply body inline

**Deploy steps:**
1. Run `prisma migrate deploy` against Render external DB (new migration adds 3 nullable columns — safe)
2. Deploy backend to Render
3. SMTP provider must allow sending from `contact@goodfights.app` in addition to `noreply@goodfights.app`. Both are `goodfights.app` — same domain — so existing SPF/DKIM should cover it. If emails bounce, check SMTP allowed-senders list.
