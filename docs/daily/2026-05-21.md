## UFC fighter headshot backfill

### The actual problem (not what I first thought)

Mike noticed Conor McGregor, Glover Teixeira, etc. were missing headshots in
the "Best UFC Fights of All Time" feed. First diagnosis was wrong — McGregor
actually has a working R2-hosted headshot. The real problem turned out to
be a different class of broken images entirely.

**Hundreds of legacy fighters had `profileImage` URLs pointing at
`https://fightingtomatoes.com/images/fighters/{Name}.jpg`** — your old site,
which stopped serving images after the 2026-04-17 GoDaddy migration to
Porkbun (a redirect-only host now). Every one of those URLs 404s. The
existing `backfillUFCHeadshots.ts` script skipped them because it filtered
only on `profileImage IS NULL`, not on broken URLs.

Affected stars (confirmed): Glover Teixeira, Jon Jones, Robbie Lawler, Rory
MacDonald, Dan Henderson, Mauricio Rua, Khabib Nurmagomedov, Daniel Cormier,
Nick Diaz, Anderson Silva, Jose Aldo, Dustin Poirier, plus many others.
Total scope: **1,167 UFC fighters** had `profileImage` either null or
pointing at a dead `fightingtomatoes.com` URL.

A secondary problem: 31 duplicate-name Fighter groups where one row has the
image and the other holds the fight history (e.g. "Doo Ho Choi" vs
"Dooho Choi"). 32 orphan rows total — small scope, deferred to a later fix.

### What we built

1. **Expanded backfill predicate** in `scripts/backfillUFCHeadshots.ts` to
   catch `fightingtomatoes.com/*`, raw `ufc.com/*`, and `fightcrewapp-images.r2.dev/*`
   (the last two healed our own interim mistakes during this session).
2. **Re-ordered by rating impact** — fighters in the most-engaged UFC
   fights surface first, so stars get fixed in the earliest passes.
3. **Switched to new-page-per-fetch** in `services/scrapeUFCAthleteHeadshot.ts`
   to match the proven daily UFC scraper pattern. Reusing a single page
   across many navigations causes Puppeteer's stealth plugin to accumulate
   state and time out on `Page.addScriptToEvaluateOnNewDocument` after
   ~50-80 fighters.
4. **Slug-collision fallback** — when a derived `ufcAthleteSlug` write
   collides with an existing duplicate fighter, retry without the slug
   field. The image is what we actually need.
5. **GH Actions workflow** `.github/workflows/ufc-headshot-backfill.yml`
   — runs from allow-listed GH runner IPs instead of home IP, which
   UFC.com CDN starts 403'ing after sustained scraping. Manual-only
   (workflow_dispatch); not a recurring cron.

### Lessons surfaced

- **Dead legacy URLs hide behind non-null fields.** Any "missing data"
  audit on `profileImage` (or any URL column) needs to test that the URL
  actually serves, not just that it's set. Same risk applies to other
  external URL fields in the schema.
- **Puppeteer + stealth plugin + shared page = guaranteed crash after
  ~80 navigations.** The protocol channel accumulates state; eventually
  `addScriptToEvaluateOnNewDocument` hangs. The proven workaround is the
  pattern the daily UFC scraper already used: fresh page per fetch.
- **UFC.com CDN rate-limits home IPs after sustained scraping.** GH
  Actions IPs are allow-listed. For anything sustained, run there.
- **Estimate per-fighter time honestly.** 750ms rate-limit + page setup +
  R2 round-trip is ~3-4s per fighter, not 1s. 883 fighters takes ~50 min,
  not 15.

## Second pass — slug search fallback + silhouette handling

### What the first run actually did

Workflow #26204697112 completed but reported "failed" status. Looking at
the run summary:

- considered: 860
- uploaded: 236 (21 via existing slug + 262 via derived slug — the count
  is higher than `okFromDerivedSlug` because `[ok-noslug]` slug-collision
  rows are bucketed under `uploaded`)
- skipped (already had image): 47
- **page exists, no og:image: 568** ← the big tail
- errors: 9 (8× HTTP 403 from UFC CDN, 1× R2 fetch fail on a SILHOUETTE)

The "failed" status was cosmetic — `process.exitCode = 1` whenever
`stats.errors > 0`. 851/860 fighters processed cleanly; nine sporadic
errors flipped the workflow status bit.

The 568 no-image tail traces to slug-variance: derived slugs miss when
the canonical UFC.com slug has a suffix (`khalil-rountree` →
`khalil-rountree-jr`), a typo we never knew about (`josh-emmet` →
`josh-emmett`), or because we store a nickname instead of the legal
name (`paulo-borrachinha` → `paulo-costa`).

A secondary bug surfaced: Gable Steveson's `og:image` was UFC's generic
`SILHOUETTE.png` placeholder; the R2 download of that URL failed, the
script fell back to writing the source URL, and Gable's DB row now
serves the placeholder as if it were his headshot.

### What we built (commit `b0b7d19`)

1. **DDG slug search fallback** in `services/scrapeUFCAthleteHeadshot.ts`
   — new `searchUFCAthleteSlugViaDDG(name, handle)` opens
   `html.duckduckgo.com/html/?q=<name>+ufc.com+athlete`, parses the
   first `*.ufc.com/athlete/<slug>` link (unwrapping the DDG `/l/?uddg=`
   redirector), and returns the canonical slug. Tried Mike's manual
   approach first because UFC.com's own `?search=` parameter doesn't
   actually filter — it returns the top-ranked fighters regardless of
   query.
2. **Search wired into the backfill loop** — when existing + derived
   slugs all miss (either 404 or 200-with-no-image), the script calls
   DDG and retries the headshot fetch with the canonical slug. New
   `okFromSearch` stat tracks how many resolve this way.
3. **Silhouette handling** — scraper detects `SILHOUETTE.png` in the
   `og:image` URL and sets `result.isPlaceholder = true`. Backfill
   uploads placeholders through a shared R2 prefix (`fighters/placeholder-…`)
   so all silhouettes dedupe to one R2 object owned by us, not
   hot-linked. The query predicate matches that prefix + raw
   `SILHOUETTE.png` URLs from previous runs, so every future run
   revisits placeholder rows in case UFC has added a real photo since.
4. **Loosened failure threshold** — exit code 1 only when
   `errors > 5% of considered`. Last night's 1% would now succeed.

## Third pass — UFC theme placeholder detection (the real fix)

### What was actually broken

Morning's "429 no-img" tail was *not* a UFC data gap, despite what the
handoff doc claimed. The handoff said UFC.com simply has no photo for
1990s-2000s legends. Half true — there's no real headshot — but the
page renders a *theme asset* placeholder (`themes/custom/ufc/assets/img/
no-profile-image.png`) in the headshot slot, and we were giving up
instead of saving it.

We were already saving the `og:image`-served `SILHOUETTE.png`
placeholder. We just weren't checking for the *other* placeholder
mechanism UFC uses, which kicks in for pages that don't populate
`og:image` at all (most legacy fighters).

### What we built (commit `47b5907`)

In `services/scrapeUFCAthleteHeadshot.ts`: when `og:image` is missing,
also check for `<img src*="no-profile-image">` in the rendered DOM.
If present, return `isPlaceholder: true` with the theme PNG URL. The
existing placeholder upload path (R2 `placeholder-` prefix, predicate
re-pickup) handles the rest unchanged.

Verified locally against Bas Rutten, Pat Miletich, Tank Abbott, Don
Frye (all → placeholder) and Conor McGregor (real photo unchanged)
before commit.

### Lesson

The morning's handoff framed the gap as a UFC limitation, when actually
it was an incomplete read of UFC's HTML. Before declaring a data source
"done," fetch a few residual pages by hand and look at the raw markup —
the "missing" data may just be in a different slot than the one we're
parsing.
