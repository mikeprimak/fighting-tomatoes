## Blog system added to the web app (markdown-file based)

Mike wanted to publish an article previewing the RAF wrestling card
(Khamzat Chimaev vs Dillon Danis). There was no blog infrastructure in
either `packages/web` or `packages/landing`, so we built a minimal
markdown-file blog into the Next.js web app and dropped the first post in
as a draft.

### How it works

Posts are plain `.md` files in `packages/web/src/content/posts/` with
gray-matter frontmatter. No CMS, no DB — files are read at build/request
time on the server.

Frontmatter shape:

```yaml
title, slug, date (YYYY-MM-DD), author, excerpt, tags[], draft (bool)
```

- `src/lib/posts.ts` — reads the posts dir, parses frontmatter
  (`gray-matter`), renders body to HTML (`marked`). **Drafts are included
  in dev but stripped when `NODE_ENV === 'production'`** — so a
  `draft: true` post is previewable locally / on `next dev` but never
  ships to prod (both the listing and a direct `/blog/<slug>` URL 404 in
  prod via `notFound()`).
- `src/app/blog/page.tsx` — index, lists posts newest-first, shows a
  "Draft" badge.
- `src/app/blog/[slug]/page.tsx` — post detail. `generateStaticParams`
  + `generateMetadata`. Shows a "hidden in production" notice on drafts.
  Uses async `params: Promise<{slug}>` per Next 16.2 conventions.
- Rendered markdown styled via a `.blog-content` block in `globals.css`
  (no `@tailwindcss/typography` plugin installed; hand-rolled prose
  styles using the existing theme vars).
- Navbar: added "Blog" link (Newspaper icon), desktop + mobile.
- `sitemap.ts`: added `/blog` + maps published (non-draft) posts.

New deps in `packages/web`: `gray-matter` ^4, `marked` ^18.

## Blog Phase 1: editorial heroes + images + RSS + sitemap fix

Shipped on branch `claude/live-event-icon-indicator-xBqmJ` (NOT main — not
live). Strategic framing from Mike: the blog is a **user-growth engine** —
first-party articles shared to Reddit/Twitter. Web is phase 1; a mobile
**"Home" tab** (editorial + personalized data) is phase 2 (noted, not built).

**Architecture decision — single source of truth, three consumers.** Markdown
files stay the source of truth. They are (a) server-rendered for SEO at
`/blog`, (b) exposed as static JSON at **`/api/editorial`** (`force-static`),
consumed by the web heroes now and the mobile Home tab in phase 2. No DB/CMS,
no content duplication. Phase 2 mobile just fetches `/api/editorial` from the
web-app origin.

Changes (`packages/web`):
- `lib/posts.ts`: added `image` frontmatter field + `DEFAULT_POST_IMAGE`
  (`/good-fights-logo.png` fallback).
- `app/api/editorial/route.ts`: static JSON feed of published posts.
- `components/EditorialHero.tsx`: RT-style latest-post band (client, react-query
  on `/api/editorial`), placed above the data feed on Upcoming (`/`), Live,
  Past, Good Fights. Renders null until loaded (no skeleton jank).
- Blog index: thumbnail per post. Post detail: hero banner image + rendered
  **tags** (were dead metadata) + OG/Twitter `summary_large_image`.
- `app/blog/feed.xml/route.ts`: RSS feed + `<head>` autodiscovery.
- **Sitemap domain fix**: was emitting `https://goodfights.app/...` for every
  URL, but the landing site only rewrites a few static pages — so `/blog` and
  `/events/*` URLs 404'd there. Now uses `lib/site.ts` `SITE_URL`
  (`web-jet-gamma-12.vercel.app`, override via `NEXT_PUBLIC_SITE_URL`).

**Verified**: `pnpm build` exit 0 (`/api/editorial` + `/blog/feed.xml` static,
`/blog/[slug]` SSG); dev on 3100 — editorial API returns the post, `/blog`,
post page, feed.xml, and all four main pages 200.

**Decision flag for Mike**: I defaulted the canonical origin to the Vercel
domain. If the web app gets a custom domain (e.g. app.goodfights.app), set
`NEXT_PUBLIC_SITE_URL` in Vercel.

**Still open**: real images per post (currently logo fallback); Danis post
placeholders + `draft:false`; phase 2 mobile Home tab.

## Aside: live-event app-icon badge question (no code)

Mike asked whether a red dot could show on the home-screen icon while
events are live. Answered: yes, via the app icon badge
(`expo-notifications` `setBadgeCountAsync`), but since the app is closed
it must be driven server-side through the push payload (`badge` field,
ideally a silent content-available push). Reliable on iOS; Android
badges are launcher-dependent and flaky. Existing Expo push pipeline
(`backend/src/services/notificationService.ts`, per-user `User.pushToken`)
makes it feasible. Scoping only — nothing built. Mike dropped it.
