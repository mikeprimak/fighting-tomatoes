# 2026-07-01 — Programmatic SEO step 6: internal linking + best-of-year hubs + /events index

**SHIPPED** (commit `a6b6a87b`). Live verification noted below.

## Why this step (and why linking came first)

The ~3,900 indexable fight pages were pure orphans: every fight card on web was a
`<button>` that opens the rate/hype modal — **no page anywhere linked to
/fights/<slug>**. Sitemap-only discovery gets crawled but ranks poorly. Event
deep pages had the same problem from the other side: the live/upcoming/past tab
pages are client-rendered (SSR HTML = spinner), so nothing crawlable linked to
event pages either.

## What shipped

**1. Fight cards are real links now** — `FightCardLink` (new shared shell):
`<a href="/fights/<slug||id>">`; plain click still `preventDefault()`s into the
modal (UX unchanged), modifier/middle click and crawlers get the deep page. All
three cards (Completed/Upcoming/Live) converted, which fixes every consumer at
once: event pages, fighter pages, home, /fights/top, search, activity.

**2. Fighter pages SSR their fight history** — the history was a client-only
query (spinner in SSR HTML), so step 1 alone wouldn't have put fighter→fight
links in the HTML. Fighter page now server-fetches fights and passes
`initialData` (stamped stale via `initialDataUpdatedAt: 0`; client refetches
with `includeUserData` — same pattern as the event page). Verified: 15 fight
links in Gaethje's SSR HTML.

**3. `/fights/best/[year]` hubs** — the proprietary-data showcase ("best fights
of 2025" queries; nobody else has fan ratings):
- Backend `/api/fights/best?year=` + `/api/fights/best-years`: calendar-year
  buckets (permanent URLs — deliberately NOT the rolling windows of
  `/community/top-recent-fights`), `totalRatings >= 10` floor, hidden-promo +
  `isVisible` scoped. Yields 25 years of data; 18 clear the page floor
  (2025: 307 qualifying fights).
- Web: SSR ranked list (CompletedFightCard, rank + event shown), year-chip nav
  cross-linking all qualifying years, ItemList JSON-LD, `noindex,follow` under
  10 fights, 404 on malformed years, `/fights/best` → temporary redirect to the
  latest qualifying year.

**4. `/events` SSR index** — upcoming (30) + recent results (30) as crawlable
rows, chips to the client tab views. Required adding `slug` to the events LIST
endpoint (select + response schema — the detail endpoint got it in step 1, the
list never did).

**5. Discovery wiring** — root sitemap: `/events` + 18 year-hub URLs; footer
(server component → SSR HTML on every page): Fighters / Events / Best Fights.

## Verified locally (dev, against local backend)

- Event page: 13 `/fights/<slug>` links in SSR HTML; fighter page: 15.
- `/fights/best/2025`: title/h1, 50 ranked cards, 18 year chips, ItemList
  JSON-LD; thin year (2003) emits `noindex, follow`; `/fights/best/abcd` 404s.
- `/fights/best` 307s to `/fights/best/2026`.
- `/events`: rows with slug hrefs; sitemap.xml carries 18 year URLs.

## Deploy race note

Web build fetches `best-years` at build time; Vercel builds faster than Render
deploys. If the build baked an empty sitemap/year list, ISR (1h revalidate on
those fetches) self-heals, or force a web rebuild with a web-touching commit —
check prod sitemap.xml for `fights/best/` URLs after both deploys.
