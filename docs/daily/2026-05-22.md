# 2026-05-22

## Glory in Giza phantom-fights incident → parser hardening across all 19 scrapers

Started as a marketing session — Mike wanted to ship the weekend's Twitter Hype
Index post. The first DB pull surfaced two events for the same real-world card
("Glory in Giza" under Gold Star + under Matchroom Boxing) with the Gold Star
copy showing phantom fights that aren't actually happening (Tyson Fury vs
Arslanbek Makhmudov, Conor Benn vs Prograis, Frazer Clarke vs Justis Huni, etc.).
Mike pivoted: fix the data first, marketing can wait. Session followed.

### What was wrong

Two layered bugs on the Gold Star `Usyk vs. Rico: Glory in Giza` event row:

1. **Mass-import of phantom fights on 2026-04-11.** Before the May 3 ufcUrl-only
   lookup fix (`lesson_tapology_event_lookup_ufcurl_only`), the Gold Star
   parser's event lookup had an OR-name fallback that merged fights from a
   sibling Tapology event into ours. 10 phantom fights got attached this way.
2. **Chicken-and-egg cancellation loop preventing auto-cleanup.** The per-event
   cancellation safety floor required the scrape to return ≥75% of the DB's
   non-cancelled fight count. The phantom fights inflated the DB count to 21;
   75% = 16; the real card is 10 fights; 10 < 16 → cancellation pass skipped
   forever. Phantoms permanent.

### The cleanup

`fix(data): migrate Usyk hypes from Gold Star to Matchroom; delete Gold Star Glory in Giza`

Transaction-scoped script at `/tmp/migrate-and-delete-goldstar.ts`:
- 4 user predictions on Gold Star's Usyk vs Verhoeven row → re-pointed to
  Matchroom's canonical Usyk vs Verhoeven row (no user conflicts)
- 2 predictions on phantom fights (Fury, Benn) → cascade-deleted (Mike opted
  not to message the users; predictions on fights that won't happen are dead)
- 5 stale `FightNotificationMatch` rows pointing at Gold Star fights → cleaned up
- 21 Gold Star fights deleted (cascades remaining predictions/ratings/etc.)
- Gold Star Glory in Giza event deleted

Post-state: only one Glory in Giza visible (Matchroom, 12 fights, matches the
DAZN-published card). 4 hypes on Usyk preserved.

### Broader audit (read-only)

Ran an orphan-fight audit across all 19 scraper-managed events. 102 stale fights
surfaced across 13 events (MVP Han vs Holm 2: 35; Top Rank Mason vs Cordina: 21
including the same phantom Conor Benn vs Prograis row that haunted Gold Star;
Top Rank Zayas vs Boots: 21; etc.). **Filtering to UPCOMING/LIVE only: 0
user-visible orphans.** Every one of the 102 stale rows was already CANCELLED.
The chicken-and-egg loop only manifests at >25% orphan ratio AND scrape size
consistently below floor; Gold Star Glory in Giza was the unique case where
both held. Other events had their phantoms cancelled early before accumulation
locked the loop in.

### Parser fix (architectural hardening)

`fix(scrapers): add absolute scrape-size floor to cancellation gate`

Added `MIN_HEALTHY_SCRAPE_FIGHTS = 5` and two helpers to `cancellationGuards.ts`:
- `computeCancellationSafetyFloor(dbNonCancelledCount)` — extracts the 75%
  floor calc
- `isScrapeHealthyForCancellation(scrapeSize, dbNonCancelledCount)` — returns
  true if scrape ≥75% floor OR ≥MIN_HEALTHY_SCRAPE_FIGHTS absolute

The absolute floor breaks the chicken-and-egg loop: any scrape returning ≥5
fights is treated as healthy regardless of DB inflation. Two-strike rule in
`decideStrike` still mitigates single-render misses past this floor.

Applied to all 19 scraper-managed parsers (17 data + ufcLiveParser +
oneFCLiveParser + pflLiveParser). Each parser still defines `cancellationSafetyFloor`
for use in log messages; the OR-logic change is purely in the
`scrapeLooksComplete` computation. Log messages updated to show both floors when
firing. Production-tsconfig + strict-tsconfig type-checks clean.

### Targeted blacklist (immediate-resurrection prevention)

`fix(scraper): blacklist Tapology event 140464 in Gold Star scraper`

Tapology lists this Matchroom-promoted card on Gold Star's promotion page
because Gold Star is a co-promoter. Without intervention, tomorrow's Gold Star
scraper would re-create the duplicate Event row (clean this time — no phantoms,
since the May 3 fix is in place — but still a duplicate of the Matchroom row).

Added `SKIP_TAPOLOGY_EVENT_IDS` set + `isBlacklistedEventUrl()` helper at the
top of `scrapeGoldStarTapology.js`. The scraper's main loop checks each
discovered event URL and skips matches. Entry includes a comment explaining
when to add more IDs.

This is a targeted bandaid, not a proper fix. The real solution is
**cross-scraper event deduplication** — when one scraper imports a card that
another scraper has already imported, skip or link. That's a separate workstream
(see follow-up below).

### Follow-ups

1. **Cross-scraper event dedup** (new project). When the same real-world card
   has two visible Event rows from different scrapers (Matchroom + Gold Star,
   ESPN + Tapology, etc.), the user sees confusing duplicates. Glory in Giza
   was today's example; will recur on any boxing card where Matchroom is the
   primary promoter and another org co-promotes. Two design options: (a) at
   import time, check for date+headliner match against existing events from
   other scrapers, skip if found; (b) post-hoc dedup pass that runs nightly.
   Probably (a) — earlier intervention, less corrective data work.
2. **Hygiene cleanup of 102 CANCELLED phantom rows.** Low priority, invisible
   to users, but worth a one-shot delete.
