## INCIDENT RESOLVED ÔÇö root cause was connection-pool amplification, NOT overload

Follow-up investigation (Mike reopened it the next morning; doubted the overload
hypothesis ÔÇö correctly). **The preliminary "aggregate load overwhelmed the starter DB"
diagnosis above was wrong.** Hard findings from inspecting the live DB:

- DB was healthy when checked (15 conns, `max_connections=103`); planner stats fine
  (`pg_class.reltuples` accurate; the `n_live_tup=0` was just a reset activity counter, a
  red herring). So it was **episodic**, not a permanent state.
- **Real root cause:** the backend calls `new PrismaClient()` in ~50
  route/service/parser modules, all loaded into the one long-running server process. Each
  opens its OWN connection pool (Prisma default `cpus*2+1` per client, miscounted high in a
  container), none with `connection_limit`. On a fight night, lifecycle + scrapers + live
  trackers + auth/search + the **newly-live web Home** (`/api/events?includeFights`, which
  `useAnyLiveEvent` polled every 60s **per tab**) opened connections across all pools at
  once Ôćĺ marched toward 103 Ôćĺ small instance OOM Ôćĺ `"database system is starting up"`
  recovery Ôćĺ crash loop. "Never seen before" because the web Home (commit `a7a5edee`, on
  main) is a brand-new heavy/repeating caller, on top of today's concurrent dev sessions.
- **All four symptoms are one cause** (DB refusing connections): slow `/api/events`,
  blog order ignored (`editorial.ts` silently falls back to date-order when the
  `systemConfig` read fails), intermittent admin login.

**Fix shipped to `main` (deploys backend + web):**
- `packages/backend/src/lib/prisma.ts` ÔÇö process-wide PrismaClient singleton with a
  bounded `connection_limit` (default 10, env `PRISMA_CONNECTION_LIMIT`) appended to
  `DATABASE_URL`. All 48 runtime modules now import it instead of `new PrismaClient()`.
- Web `useAnyLiveEvent` drops `includeFights` (liveness needs only event-level fields) ÔÇö
  kills a heavy fights+predictions aggregation that ran every 60s per tab.
- Guard added to CLAUDE.md + memory `prisma-single-client-rule` so it isn't re-introduced.
- Pushing to main also restarts the backend, clearing the leaked pools.

**Still open (secondary, not the crash cause):** `/api/events?includeFights` is itself slow
(~7s) ÔÇö it loads full fight cards + all hype predictions just to render summary cards.
Consider a main-event-only payload + short server-side cache (events change slowly). The
repo `packages/backend/render.yaml` is **stale** (says `plan: free`, user
`fightcrewapp_user`); live DB user is `fightcrewappdb_05gc_user` ÔÇö don't trust it for plan
info. If crash-looping recurs, also consider a Render connection pooler (pgbouncer) and/or a
plan bump for memory headroom.

## INCIDENT ÔÇö TRUE ROOT CAUSE (supersedes the two analyses above)

The DB kept crash-looping **even after** the connection-pool fix deployed, and **with zero
traffic and zero live trackers running** ÔÇö which ruled out load entirely. Render DB logs +
`pg_database` gave the real answer:

- Production had **9 orphaned `prisma_migrate_shadow_db_*` databases** (OIDs 27490Ôćĺ124424 =
  9 separate runs). Shadow DBs are created ONLY by `prisma migrate dev` / `db push` /
  `migrate diff` ÔÇö each is `CREATE DATABASE` + a full replay of every migration.
- Because `DATABASE_URL` always points at **prod** (CLAUDE.md "never use local DB"), running
  `prisma migrate dev` (the `db:migrate` script) while developing today's `aiEventSummary`
  migration ran it **against the 256 MB production instance**, repeatedly. Each run spiked
  memory Ôćĺ **OOM Ôćĺ Postgres crash** Ôćĺ shadow DB orphaned Ôćĺ retry Ôćĺ crash loop.
- This is why it was "new in the last 12h," "load-independent," and "fine for months" ÔÇö the
  shadow DBs never existed before today.

**The connection-pool amplification (the earlier analysis) was a real, secondary contributor
but NOT the crasher.** The crasher was `migrate dev` on prod.

**Fix applied:**
1. Dropped all 9 orphaned shadow DBs via psql (DB stable since). Instance now: `fightcrewapp`
   (76 MB, real), `fightcrewapp-db` (46 MB, stray/unused ÔÇö left in place, investigate later),
   system DBs.
2. `.claude/settings.local.json`: removed `Bash(npx prisma migrate dev:*)` + broad
   `npx prisma migrate:*` allows; added a `deny` list (migrate dev/reset/diff, db push,
   `pnpm db:migrate`/`db:reset`).
3. CLAUDE.md hard rule + memory `prisma-never-migrate-dev-on-prod`.

**Standing rule:** apply migrations with `prisma migrate deploy` ONLY; author new migrations
against a throwaway LOCAL Postgres, never `migrate dev` against the Render DB.

**Open:** investigate the stray `fightcrewapp-db` database (46 MB, app doesn't use it);
consider a plan bump to 1 GB for headroom; the connection-pool fix + home slim-down still
stand as worthwhile load reductions.

## INCIDENT REOPENED + TRUE ROOT CAUSE FOUND ÔÇö slow `/api/events` query (afternoon)

The DB crash-looped AGAIN after the shadow-DB drop + connection-singleton deploy ÔÇö with **no
user load** (users don't arrive until evening), just operator testing + the automatic every-5-min
`eventLifecycle` job. So neither prior fix was the persistent cause.

**Smoking gun (Render DB logs):** repeated `duration: ~6080 ms` on the events-list query ÔÇö
`SELECT events.id, date, promotion FROM events WHERE isVisible AND (id IN (SELECT eventId FROM
fights ...) OR date in range) AND eventStatus IN (UPCOMING,LIVE)` = `routes/index.ts` `getEvents`
(`OR: [{ fights: { some: {} } }, { date }]`). Fires on every events page load + the 5-min
lifecycle. `EXPLAIN (ANALYZE)` on a **calm** DB = **6.1s idle**, so a real query problem, not a
load symptom:
- **`work_mem` was 64 kB** (Render base config, not our code) Ôćĺ the 14,610-row `fights` subquery
  couldn't be hashed Ôćĺ `Materialize` re-scanned 74├Ś (loops=74).
- **JIT on** + inflated subplan cost (~1.3M) Ôćĺ **3.5s** of JIT compile overhead per run.

Repeated 6s CPU/-memory-heavy queries on the 256 MB instance Ôćĺ OOM kill Ôćĺ "not properly shut
down; automatic recovery" Ôćĺ repeat. "New in 12h / fine for months" = the slow query is old; it
only became fatal when fired repeatedly (multi-window dev + fight-day automation), not from a
code change. Confirmed via git: `main` had no new commits 06-04 23:15 Ôćĺ the 09:00 connection fix,
and the 06-04 feature code ran 34h fine ÔÇö so NOT the messy-branch merges and NOT this morning's
live-tracker/enrichment work.

**Fix (DB-level, no deploy, reversible, MEASURED 6130 ms Ôćĺ 6.3 ms):**
```sql
ALTER DATABASE fightcrewapp SET work_mem = '4MB';   -- was 64kB; safe because connection_limit=10 caps concurrency
ALTER DATABASE fightcrewapp SET jit = off;
ANALYZE events; ANALYZE fights;
```
Verified: ~25+ min stable (no restart), 0 slow queries across two lifecycle cycles, operator
confirmed the upcoming-events screen is near-instant again. App picks up the settings as its
pooled connections recycle ÔÇö a backend restart applies them instantly.

**ÔÜá´ŞĆ Durability:** these settings live in the DB catalog (survive restarts, NOT a full Render
reprovision). If crashes return after a reprovision, re-apply. Memory:
`render-db-workmem-jit-events-query`.

**Follow-ups (not urgent):** (1) code rewrite of the `fights: { some: {} }` OR filter in
getEvents to drop the whole-table subquery (needs deploy); (2) consider a Render plan bump above
256 MB for headroom.
