## What shipped (code)

**Backend**
- `src/lib/seoIndex.ts` — single source of truth for the SEO index gate:
  `fighterIndexWhere()` / `eventIndexWhere()` / `fightIndexWhere()` (Prisma
  `where` builders) + `isIndexable(prisma, entity, id)` (a `count` scoped to the
  id against the same `where`, so the sitemap whitelist and the per-page robots
  tag can never drift).
- `src/routes/sitemap.ts` — `GET /api/sitemap/:type` (fighters|events|fights):
  returns `{ entries: [{slug, lastModified}], total, page, limit, totalPages }`,
  filtered by the gate. `lastModified` = best enrichment timestamp
  (`aiProfileEnrichedAt` / `aiEventEnrichedAt` / `aiPostFightEnrichedAt`) falling
  back to `updatedAt`. Paginated at 50k/page with a warn log if a type ever
  crosses Google's 50k/file ceiling. Registered in `routes/index.ts`.
- `shouldIndex: boolean` added to the three detail endpoints (fighter + event in
  `routes/index.ts`, fight in `routes/fights.ts`) via `isIndexable`.

**Web (`packages/web`)**
- `app/robots.ts` — now enumerates all four sitemaps (root + fighters + events +
  fights) + `host`. (Multiple `Sitemap:` lines = standard discovery; no
  sitemap-index file needed.)
- `app/sitemap.ts` — trimmed to static + hub + blog only; retired the old capped
  `events?limit=50` fetch (events/sitemap.ts covers every indexable event now).
- `app/{fighters,events,fights}/sitemap.ts` — one child sitemap per type, each
  fetching `/api/sitemap/:type` via `src/lib/sitemapData.ts`.
- Detail pages (`fighters|events|fights/[id]/page.tsx`) emit
  `robots: { index: false, follow: true }` in `generateMetadata` when
  `shouldIndex === false` (renders for users, kept out of the index + sitemap;
  `follow` so link equity flows to indexable pages).

## Gate change from the plan spec

`Event.totalRatings` is a **dead field** (always 0 —
`lesson_dataset_aggregates_dishonest`), so the event gate can't use it. Event
fan-engagement lives on the *fights*. New event gate: indexable iff it has ≥1
indexable fight (Prisma-expressible via `fights: { some: fightIndexCore } }`,
self-consistent with the fight gate) OR `aiEventSummary` OR active broadcasts OR
upcoming/live. This lifted indexable events from **28** (strict summary-only) to
**~640** without indexing thin/junk legacy cards. Also: Prisma `where` can't
express a relation-count threshold ("≥3 fights"), and "has an indexable fight" is
a stronger quality signal than raw card size anyway.
