### DBX duplication + stuck status (data repair, prod via Render)

QA surfaced a fight in My Activity rendering as an upcoming card that the user
hadn't (knowingly) rated. Investigation found two systemic DBX problems:

1. **Duplication** — the DBX 5 scrape (2026-02-14) over-pulled DBX 3's entire
   card: 13 duplicate bouts on "DBX 5: Clark vs. Bittencourt" that belong to
   "DBX 3: Rozenstruik vs. Perez". Canonical = the DBX 3 copy. Idempotent
   migration (dry-run first): for each dup, ratings/predictions/reviews/comments
   migrated to canonical (per-user conflicts resolved by keeping canonical's),
   aggregates recomputed, then the DBX 5 dup deleted. **13 dups merged, 0 user
   ratings lost.** DBX 5 went 29 → 16 fights (its real card).
2. **Stuck status** — nearly every fight on the 7 COMPLETED DBX events was still
   `UPCOMING` (the tapology path never transitions DBX fights). **83 fights
   flipped UPCOMING → COMPLETED.**

Verified after: 0 remaining cross-event dup pairs, 0 stuck UPCOMING/LIVE on
completed DBX events. Throwaway inspect/fix scripts deleted (no code committed).

**Why this survived the big 2026-05-23 phantom cleanup:** that sweep's Pass 2
only deleted pairs on **5+ Tapology events** (rematch-safety threshold). DBX
dupes were on only 2 events → below threshold → survived. Today's repair fills
that gap with the same principles.

`snapshots/137070-zuffa-boxing-2026-01-24/*.html` (real Tapology event HTML) to

— a Stimulus target (stable, unlike Tailwind classes), exactly ONE per page,

- **Layer 1 (the fix):** all 8 `scrape*Tapology.js` (DirtyBoxing, Gamebred,
  GoldStar, GoldenBoy, KarateCombat, MVP, TopRank, ZuffaBoxing) now scope

  regression test `services/tapologyFightExtraction.test.ts` (committed copy of

## Fighter-profile provenance — protect Opus bios from the Haiku cron

Surfaced while finishing the top-367 hand-author backfill (block F): the daily
Haiku cron (`.github/workflows/fighter-profile-enrichment.yml`, 18:00 UTC) and
the hand-author passes both crawl the **same engagement-ranked list** and write
the **same `aiProfile*` columns with no provenance marker**. Two problems:
(1) the cron re-enriches on record-change / >180d-stale, so once a hand-authored
fighter fights again, **Haiku would overwrite the Opus bio**; (2) on block F the
cron got there *first* and left Haiku bios (em dashes, violating house style)
that I overwrote with Opus.

**Shipped:**
- Migration `20260527000000_add_ai_profile_source` — new `Fighter.aiProfileSource`
  (`'handauthored' | 'cron-haiku'`). One-time backfill stamped all 369
  then-profiled rows `'handauthored'` (verified: 369 handauthored, 0 NULL among
  profiled).
- `persistFighterProfile()` now takes a required `source`; cron passes
  `'cron-haiku'`, `fighter-profile-write.ts` passes `'handauthored'`.
- Cron candidate filter gained `AND aiProfileSource IS DISTINCT FROM 'handauthored'`.
  **Verified against Render: 0 handauthored rows can enter the candidate set**
  (738 candidates, all unprofiled tail). Haiku can no longer overwrite an Opus
  bio; hand-author writes stay unconditional so Opus always wins a conflict.
- Production `tsc` clean on touched files. Migration applied to Render via
  `migrate deploy` (not `migrate dev` — non-superuser lesson).
