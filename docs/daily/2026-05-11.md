# 2026-05-11

## Support — Email verification failure (cray_zegyptian@hotmail.com)

User reported on Android v2.0.1 that every click of the email verification link took them to an error page.

**Root cause:** Multiple verification emails had been issued (`updatedAt` on the user row showed a fresh token issued hours before the complaint). The resend endpoint overwrites `emailVerificationToken` unconditionally, so every prior email's link returns `TOKEN_INVALID`. User was clicking an older link.

**Fix for the user:** Hit `GET /api/auth/verify-email?token=<live-token-from-DB>` directly. Endpoint returned 200, `isEmailVerified` flipped to true, token cleared.

**Followups:**

- `packages/backend/src/routes/auth.fastify.ts` — added a `request.log.warn` on the `TOKEN_INVALID` branch logging token prefix + IP + UA. Previously this branch was silent in Render logs, so we had no way to verify a user's reported click. Next time the same complaint comes in we can grep Render logs and correlate the click timestamp.
- `docs/playbooks/triage-email-verification.md` — new playbook with the diagnosis flow, DB-lookup snippet, decision table, manual fix curl, and the canned reply for the user.
- Confirmed backend Sentry is still TODO (item #1 in `docs/areas/support-tooling.md`). When it lands, this kind of triage will mostly be unnecessary.
