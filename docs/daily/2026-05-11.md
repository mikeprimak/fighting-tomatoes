## Support — Email verification failure (cray_zegyptian@hotmail.com)

User reported on Android v2.0.1 that every click of the email verification link took them to an error page.

**Root cause:** Multiple verification emails had been issued (`updatedAt` on the user row showed a fresh token issued hours before the complaint). The resend endpoint overwrites `emailVerificationToken` unconditionally, so every prior email's link returns `TOKEN_INVALID`. User was clicking an older link.

**Fix for the user:** Hit `GET /api/auth/verify-email?token=<live-token-from-DB>` directly. Endpoint returned 200, `isEmailVerified` flipped to true, token cleared.

**Followups:**

- `packages/backend/src/routes/auth.fastify.ts` — added a `request.log.warn` on the `TOKEN_INVALID` branch logging token prefix + IP + UA. Previously this branch was silent in Render logs, so we had no way to verify a user's reported click. Next time the same complaint comes in we can grep Render logs and correlate the click timestamp.
- `docs/playbooks/triage-email-verification.md` — new playbook with the diagnosis flow, DB-lookup snippet, decision table, manual fix curl, and the canned reply for the user.
- Confirmed backend Sentry is still TODO (item #1 in `docs/areas/support-tooling.md`). When it lands, this kind of triage will mostly be unnecessary.

## Bugfix — Back button from "fight is next" notification

**Report:** Tapping a fight-start / walkout notification opens the fight detail screen. First tap of the back button does nothing; second tap eventually navigates. Desired: first back tap lands on Live Events.

**Root cause (two bugs compounding):**

1. `NotificationHandler.tsx` handles the cold-start tap via `getLastNotificationResponseAsync()` AND registers `addNotificationResponseListener()`. On warm-from-background and sometimes on cold-start, the same tap is delivered to both paths, so `router.push('/fight/${id}')` runs twice. First back press pops the duplicate (visually identical screen — looks like nothing happened); second back actually navigates.
2. Even with single push, fight detail is stacked above the `(tabs)` navigator. Back goes to whatever tab was last active — not necessarily Live Events.

**Fix (`packages/mobile/components/NotificationHandler.tsx`):**

- Consolidated cold-start + listener into a single `handleResponse(response, isInitial)` function.
- Deduplicate via `Set<string>` keyed by `response.notification.request.identifier`.
- For any `data.fightId` notification: `router.navigate('/(tabs)/live-events')` then `router.push('/fight/${fightId}')`. The navigate switches the active tab so back lands on Live Events.

**Shipped:** commit `2b9ca25`, EAS update on production branch (iOS update `019e191f-e7b3-7294-9e04-a881559d0a02`, Android `019e191f-e7b3-716b-a601-6d7483af1be7`). JS-only; users get it after two app restarts.

## Broadcast Discovery — diagnosed + hardened + auto-apply shipped

User asked "is the weekly discovery job deployed?" Investigation showed it IS deployed — cron has been firing Mondays 09:00 UTC since 2026-05-03. But every run since the first one had silently produced zero findings, because:

1. `extract.ts` swallowed every Anthropic error (auth, billing, network) as a warning and returned `[]`. Exhausted credits looked identical to "no broadcasters found."
2. The May 11 cron drained 50+ Brave queries and 400'd on every Anthropic call with "credit balance too low" — and the workflow still exited 0.

### Fixes shipped this session

- **Anthropic credits topped up** (user action). Re-triggered the May 11 cron via `workflow_dispatch`: 5 minutes, 140 Brave queries, 70 Anthropic calls, **48 new findings inserted**, 43 existing defaults re-confirmed, 0 errors.
- **Fail-loud on auth errors** — `extract.ts` now distinguishes fatal (401/403/credit-balance/invalid-key) from transient errors. Fatals throw; transients still fall through as warnings. Future credit drains will exit non-zero and the workflow's failure step will hit `/admin/test-alert`. Commit `c97db24`.
- **Auto-apply for high-confidence findings** — new `scripts/auto-apply-discoveries.ts`. Applies PENDING discoveries with confidence ≥ 0.90 AND resolved `channelSlug` AND tier set AND changeType IN (NEW, CHANGED). Additive upsert only — never deactivates sibling defaults the way the admin Apply endpoint does. Wired into `broadcast-discovery.yml` as a post-discovery step so the Monday cron self-cleans. Commit `61819e3`.
