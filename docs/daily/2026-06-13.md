## Tapology Cloudflare → Scrapfly (the real fix)

Continuation of the 2026-06-11/12 investigation. Root cause confirmed earlier:
Tapology escalated Cloudflare from JA3/TLS fingerprinting to a JS challenge that
escalates to an **interactive Turnstile widget on datacenter IPs** (GitHub
Actions runners + the Hetzner VPS). No headless browser solves Turnstile, so
every Tapology daily scrape / live track / backfill broke at once. The
real-Chrome+stealth path (`tapologyBrowser.js`) only clears it from a
**residential** IP — fixes local dev, not CI/VPS.

Decision (Mike): go with **Scrapfly** Web Unlocker, **test for one month**.

## Scrapfly request budget — Tapology live polling slowed 30s → 150s

Follow-up to the Scrapfly fix above. Scrapfly free tier = 1000 req/mo, and
**each live poll is one request**. `scraperService.ts` ran a single global
`SCRAPE_INTERVAL_MS = 30s` for *every* tracker, so Tapology burned ~120 req/hr
— one long card (~6–8 hr LIVE window) could eat half the monthly cap. Earlier
belief that this was already bumped to 150s was wrong; it had never shipped.

**Shipped to `main` (`f5406eb4`, perf(scrapers))** — worktree off `origin/main`,
rebased onto `b884a824` (notify-flags commit) so it FF'd cleanly; pivot branch
untouched. Split the cadence per scraper type via `scrapeIntervalFor()`:
- `tapology` → `TAPOLOGY_SCRAPE_INTERVAL_MS = 150s` (the only Scrapfly-metered path)
- own-site scrapers (ufc/bkfc/pfl/onefc/oktagon/sherdog) stay at 30s — they fetch
  their own sites, cost no Scrapfly requests, and want real-time cadence.

5× fewer Scrapfly requests on Tapology; the added ≤150s lag is invisible since
Tapology isn't a real-time source (results lag, backfill cron exists). Only A of
the two optimizations was done — boxing-org backfill-only (C) is moot (orgs
shelved). Per-type type-safety verified by inspection (worktree had no
node_modules; change is a trivial `(string): number` helper).

**⚠️ Needs VPS deploy to take effect:** `ssh` + `bash
/opt/scraper-service/packages/backend/vps-update.sh` (git pull → build →
restart). Until then the VPS runs the old 30s cadence.

## Adaptive downshift for stalled Tapology trackers (B′)

**Shipped to `main` (`e41219ed`, perf(scrapers))** — worktree off `origin/main`,
FF-clean, pivot untouched. `tsc --project tsconfig.production.json` = 0 errors
(verified via a junction to the main checkout's node_modules — worktrees have
none).

Mike's correction: Tapology is *largely reliable* and follows most events
closely; only a rare few stall. So the downshift is a pure safety net for the
rare dud, tuned to never touch a healthy card:
- Each Tapology scrape that yields `fightsUpdated > 0` (a real status/result
  transition, `changed===true` in `tapologyLiveParser`) bumps `tracker.lastChangeAt`.
- `nextTapologyDelayMs()`: if the card has been flat (no change) for
  **`TAPOLOGY_FLATLINE_MS = 45min`**, poll every **`TAPOLOGY_SLOW_INTERVAL_MS = 15min`**;
  otherwise the full 150s. The first change resets `lastChangeAt` → snaps back
  to 150s automatically.
- Mechanism: Tapology trackers now use a **self-rescheduling `setTimeout`** loop
  (so the delay can vary per tick); own-site scrapers keep the fixed 30s
  `setInterval`. Tracker is registered in `activeTrackers` *before* the first
  scrape so an auto-complete mid-scrape isn't undone by a late insert; the loop
  bails if `!activeTrackers.has(eventId)` after a scrape. `stopTracker`'s
  `clearInterval` clears the `setTimeout` handle fine (Node interchangeable).
- 45min is conservative — a normal card never goes that long without a fight
  moving, so it stays at full speed all night. Worst-case fully-stuck card:
  ~42 requests vs the old ~960.

**⚠️ Same VPS deploy covers both A and B′:** `ssh` + `bash
/opt/scraper-service/packages/backend/vps-update.sh`. Both are on `main` now
(`f5406eb4` then `e41219ed`); one pull/build/restart picks up both.
