## Changes Made

### Backend: Gamebred parser — drop name OR-fallback in event lookup
- **Bug**: The May 1 Gamebred Bareknuckle MMA event in the app showed Anthony Smith vs Chase Sherman, who actually fought on the April 10 Gamebred event. Several other fights similarly misplaced.
- **Root cause**: Tapology hadn't yet appended a "vs. X" headliner suffix to either upcoming event, so events 140031 (Apr 10) and 140032 (May 1) both rendered with the title "Gamebred Bareknuckle MMA". The parser's lookup used `OR: [{ufcUrl}, {promotion + name}]` — when the second event was scraped, the name branch matched the first event's row and merged its fights into one card. ufcUrl-by-itself would have routed each event to its own row.
- **Fix**: `gamebredDataParser.ts` now looks up events by `ufcUrl` only. The Tapology event ID is the unique stable identifier; the name fallback was redundant and caused the merge.
- File: `packages/backend/src/services/gamebredDataParser.ts:183-190`

### Backend: Data repair script + run
- New: `packages/backend/src/scripts/fixGamebredEventCollision.ts`
- Re-fetches the Tapology Gamebred hub + each event page (cheerio, no puppeteer), ensures one Event row exists per ufcUrl, then re-points fight.eventId for any fight currently sitting on the wrong row. User ratings stay attached to the fight rows so they survive the move.
- DRY_RUN=1 supported. Idempotent — re-running with no remaining mismatches is a no-op.
- **Run against prod**: created the missing Apr 10 (140031) row, moved 10 fights from the merged May 1 row to it (incl. Smith vs Sherman, Cooper III vs Eliziario, Pena vs Alexandre, Petrov vs Sosa, Polanco vs Schnurpel, Rosa vs Sojo, Santos vs Inocente, Santos vs Quinones, Soriano vs Ramos, Sowma vs Kovalenko). 117 fights already on the correct row. 2 ambiguous-pair skips (Cade Howell vs Christopher Wingate appears on both Gamebred Boxing 3 and 4 — they fought twice).

The bug only triggers when two upcoming events on the same hub share an exact title — uncommon for promos that always include a "vs." suffix. ## Backend: RAF results-backfill wrapper (Phase 2 wrapper-set complete)

Resumed the **retroactive results system** (paused 2026-04-28). RAF was the open per-org wrapper. Plumbed `BackfillOptions` into `rafLiveParser.ts`, added `backfillRAFResults.ts`, wired the orchestrator. RAF now rides the same daily 10:00 UTC cron as UFC/BKFC/ONE FC/Oktagon/Matchroom/PFL.

### Why RAF, not Tapology

Initial assumption was that RAF should ride the Tapology backfill pipeline. Code search proved the opposite — RAF has its own fully-built native scraper stack against `realamericanfreestyle.com`:

- `services/scrapeAllRAFData.js` (daily)

User confirmed the RAF native pipeline stays — daily, live, AND retro all use realamericanfreestyle.com. Tapology is not involved in the RAF flow at all.

### What landed

- `packages/backend/src/services/rafLiveParser.ts` — `parseRAFLiveData` now takes a 3rd optional `BackfillOptions` arg. Five flags plumbed (mirrors BKFC):
  - `nullOnlyResults` — RAF parser already gates winner/method writes on `!dbFight.winner` / `!dbFight.method`, so this is a no-op for safety/consistency
  - `skipStaleLiveReset` — gates the "lifecycle-premature COMPLETED → UPCOMING" downgrade
  - `skipNotifications` — gates the `notifyNextFight()` call on COMPLETED transitions
  - `skipCancellationCheck` — gates the entire CANCELLED↔UPCOMING reconciliation pass
  - `completionMethodOverride` — when set, stamps `Fight.completionMethod` + `Fight.completedAt` on COMPLETED transitions (audit trail)
  - Header log line now reads `[RAF BACKFILL]` vs `[RAF PARSER]` based on whether any flag is set
- `packages/backend/src/services/backfillRAFResults.ts` — new wrapper. Clone of `backfillBKFCResults.ts` shape (child-process JS scraper → JSON output → parser). Path differences: `scrapeRAFLiveEvent.js`, `live-event-data/raf/`, `raf-live-` filename prefix. Stamps `completionMethod = 'backfill-raf'`.
- `packages/backend/src/scripts/backfillResults.ts` — added `case 'raf':` in the dispatcher, import, and updated the header coverage comment.

### Note on RAF auto-publish behavior

RAF is not in `DEFAULT_PRODUCTION_SCRAPERS` (`['ufc', 'oktagon', 'bkfc', 'onefc', 'pfl']`), so `buildTrackerUpdateData` writes only to shadow fields (`trackerWinner`, `trackerMethod`, etc.) — same as the live tracker. The orchestrator's candidate query filters on the published `winner` field, which means RAF events with shadow-only data will keep showing up as candidates each day until an admin publishes. That's idempotent (each rerun produces the same shadow values) but a touch wasteful. If RAF publish becomes a recurring annoyance, the right fix is either to add RAF to `DEFAULT_PRODUCTION_SCRAPERS` or to have the backfill query also exclude rows where the shadow already matches.

## Backend: admin backfill health widget (Phase 2 complete)

Final Phase 2 item. Sentinel for "is the daily backfill silently broken / do we have an org without a wrapper". Commit `c142df9`.

### What landed

- **`GET /admin/health/backfill`** in `packages/backend/src/routes/admin.ts` — admin-only. Queries events where `eventStatus='COMPLETED'`, `date < now - 48h`, and at least one fight with `winner=NULL`, then aggregates by `scraperType`. Returns rows with `missingWinners`, `events`, `oldestEventDate`/`oldestEventName`, and a `covered` boolean (whether the org has an entry in the orchestrator's switch). Top-level `uncoveredOrgs` array is the explicit sentinel — any scraperType with stale events but no wrapper.
- **Widget on the Operations tab** in `packages/backend/public/admin.html` — between Production Live Trackers and OKTAGON manual controls. Renders three states:
  - **Empty (good)**: green "All caught up" panel
  - **Has uncovered orgs**: orange banner "⚠ Org without wrapper: <list>" above the table
  - **Has stale events**: table with columns `Scraper / Missing / Events / Oldest stale event`, rows sorted by missingWinners desc, each scraperType tagged `covered` (green) or `uncovered` (orange)
- Loaded into `loadOperationsData()`'s Promise.all alongside the other Operations sections; `Refresh` button on the section header reruns the fetch.
