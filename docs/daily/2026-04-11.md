### Backend: ONE FC fighter names â€” prefer JSON-LD performer over URL slug (Task 2)

- **Symptom**: ONE FC athlete URL slugs frequently truncate or garble real display names. `/athletes/rittidet/` is single-word â†’ parser stored `{firstName: '', lastName: 'Rittidet'}` when the real name is Rittidet Lukjaoporongtom. `/athletes/nuapet-tded99/` â†’ "Nuapet Tded99" when it should be Nuapet Torfunfarm. At least 17 ONE FC fighters in the DB had `firstName = ''` from single-word slugs.
- **Root cause**: both the daily scraper (`scrapeAllOneFCData.js`) and the live scraper (`oneFCLiveScraper.ts`) derived names from `/athletes/<slug>/` URL kebab-case parsing. The slug is whatever ONE FC decided when creating the athlete page and often lacks the full surname / camp suffix.
- **Fix**: use schema.org JSON-LD `performer` array as the primary name source. Each event page has a `<script type="application/ld+json">` with an Event entity whose `performer` array alternates `PerformingGroup` entries (the matchup name, e.g. "Kompet vs. Attachai") with `Person` entries for each fighter (the full name, e.g. "Kompet Sitsarawatsuer"). Walk the array, match matchup DOM elements to JSON-LD entries by normalized versus-text key, extract `fighterAFullName` / `fighterBFullName`, pass them through to the parser.
- **Parser update** (`oneFCDataParser.ts:parseOneFCFighterName`): accepts an optional `fullName` parameter; preference order is now JSON-LD fullName â†’ URL slug â†’ display name. Nickname extraction handles both ASCII quotes and curly quotes (`"`, `"`, `'`, `'`), including the rare pre-name form `"Petnueng" Isaac Mohammed` â†’ `{firstName: 'Isaac', lastName: 'Mohammed', nickname: 'Petnueng'}`.
- **Live parser update** (`oneFCLiveParser.ts:findFightByFighters`): was matching DB fights by `lastName` only on each side. Changed to match via token intersection (firstName âˆª lastName) so the live tracker keeps matching legacy single-word DB rows during the transition. Once a fighter gets renamed from `{'', 'Rittidet'}` to `{'Rittidet', 'Lukjaoporongtom'}`, the scraper emits `lastName: 'Lukjaoporongtom'` which would otherwise not match the old row. Token-based matching bridges both shapes.
- **Verified in isolation** against the captured `one150.html`: 14/14 matchups resolved correctly to their JSON-LD full names, including Nuapet Torfunfarm, Rittidet Lukjaoporongtom, Kompet Sitsarawatsuer, and the nickname-bearing fighters (Regian "The Immortal" Eersel, Rasul "Sagittarius" Rahimov, "Petnueng" Isaac Mohammed).

### Backend: ONE FC name backfill script (`scripts/fix-onefc-broken-names.js`)

Written to repair existing broken rows after the scraper fix.

- **How it works**: scrapes every ONE FC event in the DB (`scraperType = 'onefc'`, `ufcUrl IS NOT NULL`), builds a global `slug â†’ fullName` map from each event's JSON-LD paired with matchup face anchor hrefs, then for every ONE FC fighter row derives a "reverse slug" from their stored name (`dbNameToSlug(firstName, lastName)`) and looks it up. If the JSON-LD full name differs from the stored name, it proposes a rename (or merge if a correctly-named row already exists).
- **Safety guards** (learned the hard way via dry-run iteration):
  1. **First-word match + word-count â‰¥** filter. Rejects proposals like `Vero Nika â†’ Vero` (JSON-LD would shorten a real name) and `Banluerit â†’ Banluelok` (first-word mismatch = slug collision with an unrelated fighter). Without this filter 11 risky renames would have landed.
  2. **Scope guard**: skip any fighter whose fight history contains non-ONE promotions. The backfill MUST only touch fighters that are pure ONE FC â€” otherwise a rename or merge would quietly affect how the fighter appears in their UFC/Bellator/etc. history. 30 cross-promotion rows excluded.
  3. **Dry-run by default**, `--apply` opt-in.
- **Applied to prod** (Render DB): **10 renames, 5 merges, 0 failures**. Fixes include the originally-flagged Nuapet Tded99 â†’ Nuapet Torfunfarm, plus single-word â†’ full-name merges for Rodtang Jitmuangnon, Takeru Segawa, Kompet Sitsarawatsuer, Petkhaokradong Lukjaomaesaithong, Kongklai Sor Sommai, Tonglampoon FA Group, and renames for Tubtimthong IngfahhotelUbon, Yodphupa Petkiatpet, Kompet, Ratchasiesan Laochokcharoen (with nickname "Left Meteorite"), plus several Muay Thai camp-rename cases (Petnamkhong, Denkriangkrai, Isannuea, Sulaiman, Khunsuk, Lamsing) where the fighter's first name stayed but the camp/gym suffix on the last-name field changed.
- **Remaining leakage**: 10 ONE-FC-only fighters still have `firstName = ''` (Dedduanglek, Hyu, Jaosuayai, Misaki, Phetjeeja, Pompet, Ranma, Ratchasiesan, Shoma, Taku). Their ONE FC appearances weren't in any of the 10 events that had scrape-able JSON-LD data â€” either their events pre-date the scrape window or they haven't appeared in a JSON-LD-populated event yet. Accepted as MVP leakage; will self-heal whenever they appear on an event the daily scraper processes.

### Backend: audit other live scrapers for the ONE FC per-face sticker bug (Task 3)

**Diagnostic query** across all promotions to find suspicious A-side winner skew:

```sql
SELECT e.promotion,
       COUNT(*) FILTER (WHERE f.winner = f."fighter1Id") AS a_wins,
       COUNT(*) FILTER (WHERE f.winner = f."fighter2Id") AS b_wins,
       COUNT(*) AS total
FROM "fights" f
JOIN "events" e ON e.id = f."eventId"
WHERE f.winner IS NOT NULL
GROUP BY e.promotion;
```

Results showed suspicious skews:
- **Karate Combat**: 10/0 A-wins (100%)
- **MVP**: 9/1 A-wins (90%)
- **RIZIN**: 4/0 A-wins (100%, tiny sample)
- **PFL / RAF / Oktagon / ONE**: 60â€“63% A-side (mild skew)
- **UFC**: 57.5% (normal)

**Verified Tapology tracker is not buggy**. Fetched the live Tapology HTML for Karate Combat 59 and MVP Dubois vs Harper, ran the scraper's parent-chain walk against each, and cross-referenced with the DB.
- Tapology reorders displayed fighters so the winner appears on the LEFT of the row after the fight is decided. Winner row gets `from-[#d1f7d2]` (green) gradient, loser row gets `from-[#ffecec]` (pink). These classes live on the fighter's own cell, not a shared ancestor.
- The scraper emits the winner by NAME (from the green-cell fighter), and `tapologyLiveParser.getWinnerFighterId` matches that name to whichever DB fighter has it. So the winner ID resolves correctly regardless of whether the match is A-side or B-side in the DB's fighter1/fighter2 ordering.
- Confirmation on MVP Dubois vs Harper: all 10 fights resolved correctly. One of them (Watson vs Makinen â†’ Makinen) resolved as a B-side win because the daily scraper recorded Watson as fighter1 but Makinen won. If the bug existed, 10/10 would be A â€” but it's 9/1, because the daily scraper HAPPENED to record 9 of 10 fights in an order that matches Tapology's post-fight reorder. The 100% / 90% A-side skew is **sampling noise**, not a bug.
- There's a weak secondary check (`$corner.find('.bg-green-500').length > 0`) that returns true for both parents in some DOM layouts, but it's redundant because the primary `from-[#d1f7d2]` class check is exclusive to the winner cell. Not worth rewriting for MVP.

## Decisions

- **Task 1 (historical ONE FC winner backfill) descoped for MVP.** The pre-fix ONE FC live tracker silently defaulted every winner to `fighter1` due to the per-face sticker bug, so roughly half of historical ONE FC winners in the DB are wrong. Fixing them requires re-scraping every past event and re-parsing with the new sticker logic. User decision: defer until post-MVP; the scraper fix prevents future regressions and that's enough for now. Handoff doc updated to mark Task 1 as `~~descoped~~`.
- **Full `pnpm build` fix (handoff Task 5) deferred.** Discovered that the backend's `build` script in `package.json` ends with `|| true`, which swallows all TypeScript errors and exits 0. So Render has been deploying successfully this whole time despite the broken type-check â€” the errors are type-check errors, not runtime errors. The `tsconfig.tracker.json` workaround and the `|| true` shortcut are both buckets under the same leak, but the leak isn't actively hurting anything. Documented as post-MVP tech debt. Recommended minimal follow-up: remove `|| true` AFTER Task 4 (generalize the tracker tsconfig) lands, so type errors become loud again.
- **Backfill scope guard locked to ONE-FC-only fighters.** Any fighter row with fights outside ONE FC is skipped entirely. Prevents quietly changing how a fighter appears in UFC/Bellator/etc. events. 30 rows excluded on top of the 11 filtered by the first-word / word-count safety rule.
