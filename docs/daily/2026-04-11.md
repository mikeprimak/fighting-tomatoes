### Backend: ONE FC fighter names — prefer JSON-LD performer over URL slug (Task 2)

- **Symptom**: ONE FC athlete URL slugs frequently truncate or garble real display names. `/athletes/rittidet/` is single-word → parser stored `{firstName: '', lastName: 'Rittidet'}` when the real name is Rittidet Lukjaoporongtom. `/athletes/nuapet-tded99/` → "Nuapet Tded99" when it should be Nuapet Torfunfarm. At least 17 ONE FC fighters in the DB had `firstName = ''` from single-word slugs.
- **Root cause**: both the daily scraper (`scrapeAllOneFCData.js`) and the live scraper (`oneFCLiveScraper.ts`) derived names from `/athletes/<slug>/` URL kebab-case parsing. The slug is whatever ONE FC decided when creating the athlete page and often lacks the full surname / camp suffix.
- **Fix**: use schema.org JSON-LD `performer` array as the primary name source. Each event page has a `<script type="application/ld+json">` with an Event entity whose `performer` array alternates `PerformingGroup` entries (the matchup name, e.g. "Kompet vs. Attachai") with `Person` entries for each fighter (the full name, e.g. "Kompet Sitsarawatsuer"). Walk the array, match matchup DOM elements to JSON-LD entries by normalized versus-text key, extract `fighterAFullName` / `fighterBFullName`, pass them through to the parser.
- **Parser update** (`oneFCDataParser.ts:parseOneFCFighterName`): accepts an optional `fullName` parameter; preference order is now JSON-LD fullName → URL slug → display name. Nickname extraction handles both ASCII quotes and curly quotes (`"`, `"`, `'`, `'`), including the rare pre-name form `"Petnueng" Isaac Mohammed` → `{firstName: 'Isaac', lastName: 'Mohammed', nickname: 'Petnueng'}`.
- **Live parser update** (`oneFCLiveParser.ts:findFightByFighters`): was matching DB fights by `lastName` only on each side. Changed to match via token intersection (firstName ∪ lastName) so the live tracker keeps matching legacy single-word DB rows during the transition. Once a fighter gets renamed from `{'', 'Rittidet'}` to `{'Rittidet', 'Lukjaoporongtom'}`, the scraper emits `lastName: 'Lukjaoporongtom'` which would otherwise not match the old row. Token-based matching bridges both shapes.
- **Verified in isolation** against the captured `one150.html`: 14/14 matchups resolved correctly to their JSON-LD full names, including Nuapet Torfunfarm, Rittidet Lukjaoporongtom, Kompet Sitsarawatsuer, and the nickname-bearing fighters (Regian "The Immortal" Eersel, Rasul "Sagittarius" Rahimov, "Petnueng" Isaac Mohammed).

### Backend: ONE FC name backfill script (`scripts/fix-onefc-broken-names.js`)

Written to repair existing broken rows after the scraper fix.

- **How it works**: scrapes every ONE FC event in the DB (`scraperType = 'onefc'`, `ufcUrl IS NOT NULL`), builds a global `slug → fullName` map from each event's JSON-LD paired with matchup face anchor hrefs, then for every ONE FC fighter row derives a "reverse slug" from their stored name (`dbNameToSlug(firstName, lastName)`) and looks it up. If the JSON-LD full name differs from the stored name, it proposes a rename (or merge if a correctly-named row already exists).
- **Safety guards** (learned the hard way via dry-run iteration):
  1. **First-word match + word-count ≥** filter. Rejects proposals like `Vero Nika → Vero` (JSON-LD would shorten a real name) and `Banluerit → Banluelok` (first-word mismatch = slug collision with an unrelated fighter). Without this filter 11 risky renames would have landed.
  2. **Scope guard**: skip any fighter whose fight history contains non-ONE promotions. The backfill MUST only touch fighters that are pure ONE FC — otherwise a rename or merge would quietly affect how the fighter appears in their UFC/Bellator/etc. history. 30 cross-promotion rows excluded.
  3. **Dry-run by default**, `--apply` opt-in.
- **Applied to prod** (Render DB): **10 renames, 5 merges, 0 failures**. Fixes include the originally-flagged Nuapet Tded99 → Nuapet Torfunfarm, plus single-word → full-name merges for Rodtang Jitmuangnon, Takeru Segawa, Kompet Sitsarawatsuer, Petkhaokradong Lukjaomaesaithong, Kongklai Sor Sommai, Tonglampoon FA Group, and renames for Tubtimthong IngfahhotelUbon, Yodphupa Petkiatpet, Kompet, Ratchasiesan Laochokcharoen (with nickname "Left Meteorite"), plus several Muay Thai camp-rename cases (Petnamkhong, Denkriangkrai, Isannuea, Sulaiman, Khunsuk, Lamsing) where the fighter's first name stayed but the camp/gym suffix on the last-name field changed.
- **Remaining leakage**: 10 ONE-FC-only fighters still have `firstName = ''` (Dedduanglek, Hyu, Jaosuayai, Misaki, Phetjeeja, Pompet, Ranma, Ratchasiesan, Shoma, Taku). Their ONE FC appearances weren't in any of the 10 events that had scrape-able JSON-LD data — either their events pre-date the scrape window or they haven't appeared in a JSON-LD-populated event yet. Accepted as MVP leakage; will self-heal whenever they appear on an event the daily scraper processes.

### Backend: audit other live scrapers for the ONE FC per-face sticker bug (Task 3)

**Diagnostic query** across all promotions to find suspicious A-side winner skew:

```sql
SELECT e.promotion,
       COUNT(*) FILTER (WHERE f.winner = f."fighter1Id") AS a_wins,
       COUNT(*) FILTER (WHERE f.winner = f."fighter2Id") AS b_wins,
       COUNT(*) AS total
FROM "fights" f
JOIN "events" e ON e.id = f."eventId"
WHERE f.winner IS NOT NULL
GROUP BY e.promotion;
```

Results showed suspicious skews:
- **Karate Combat**: 10/0 A-wins (100%)
- **MVP**: 9/1 A-wins (90%)
- **RIZIN**: 4/0 A-wins (100%, tiny sample)
- **PFL / RAF / Oktagon / ONE**: 60–63% A-side (mild skew)
- **UFC**: 57.5% (normal)

**Verified Tapology tracker is not buggy**. Fetched the live Tapology HTML for Karate Combat 59 and MVP Dubois vs Harper, ran the scraper's parent-chain walk against each, and cross-referenced with the DB.
- Tapology reorders displayed fighters so the winner appears on the LEFT of the row after the fight is decided. Winner row gets `from-[#d1f7d2]` (green) gradient, loser row gets `from-[#ffecec]` (pink). These classes live on the fighter's own cell, not a shared ancestor.
- The scraper emits the winner by NAME (from the green-cell fighter), and `tapologyLiveParser.getWinnerFighterId` matches that name to whichever DB fighter has it. So the winner ID resolves correctly regardless of whether the match is A-side or B-side in the DB's fighter1/fighter2 ordering.
- Confirmation on MVP Dubois vs Harper: all 10 fights resolved correctly. One of them (Watson vs Makinen → Makinen) resolved as a B-side win because the daily scraper recorded Watson as fighter1 but Makinen won. If the bug existed, 10/10 would be A — but it's 9/1, because the daily scraper HAPPENED to record 9 of 10 fights in an order that matches Tapology's post-fight reorder. The 100% / 90% A-side skew is **sampling noise**, not a bug.
- There's a weak secondary check (`$corner.find('.bg-green-500').length > 0`) that returns true for both parents in some DOM layouts, but it's redundant because the primary `from-[#d1f7d2]` class check is exclusive to the winner cell. Not worth rewriting for MVP.

### Root cause
Two bugs in `scrapeTopRankTapology.js`, both present in the other six `*Tapology.js` scrapers too:

Tapology event pages include a sidebar listing other upcoming/live events with their posters, and Oktagon 86 was LIVE on 2026-04-11, so its sidebar poster won the selector on every event page scraped today. The wrong bytes were then uploaded to R2 under each event's correct slug.

## Late-night session: backend TypeScript build cleanup (Task 5 from the prior list)

Followed `docs/typescript-build-cleanup.md`'s order of attack. **81 errors → 0**, full `pnpm build` now works end-to-end, `|| true` cover-up removed, `tsconfig.tracker.json` workaround deleted.

### Pile B — missing import (1 error)

- `packages/backend/src/services/newsScraperService.ts` imported `prisma` from `../utils/prisma`, which doesn't exist. Every other service file in this repo just inlines `const prisma = new PrismaClient()` — matched that pattern instead of inventing a shared module.

### Pile C — DOM types in oneFCLiveScraper (26 errors)

- Added `"DOM"` to `lib` in `packages/backend/tsconfig.json`. The scraper's `page.evaluate()` callbacks reference `document`, `HTMLAnchorElement`, `HTMLImageElement`, etc. — those types live in `lib.dom.d.ts` and were disabled in the base config.
- Trade-off per the brief: every backend file now has DOM globals available, so technically someone could accidentally use `document` in a Node-only file. Accepted — the ergonomic cost is low and Puppeteer callbacks are the only legit use site.
- This was the recommended fix (Approach 1) because it also lets the ONE FC live tracker build under the main `pnpm build`, collapsing the `tsconfig.tracker.json` workaround.

### Pile A1 — `description` not in `FastifySchema` (19 errors)

- **Not a root-module augmentation.** `FastifySchema` is declared in `fastify/types/schema`, not the root `fastify` module — augmenting `declare module 'fastify'` silently no-ops.
- Fix: added a second `declare module 'fastify/types/schema'` block in `packages/backend/src/types/fastify.d.ts` that re-adds `description?`, `summary?`, `tags?`. One file, zero changes to route files. Cleaner than deleting 19 fields across 4 routes, and preserves the descriptions for when `@fastify/swagger` is eventually added.

### Pile A2 — `any is not assignable to never` on log.error (35 errors)

- **The brief misdiagnosed this.** It called it a Fastify `reply.send()` generic-inference regression. It's not — every erroring site is `request.log.error(msg, error)`, not `reply.send`.
- **Root cause**: Pino 9's `LogFn` type uses template-literal `ParseLogFnArgs<TMsg>`. When a message string contains no `%s` / `%d` placeholders, the rest args tuple evaluates to `[]`, so the rest parameter type becomes `never`. Passing a second arg then fails as "any not assignable to never".
- **Fix**: swapped argument order from `log.error(msg, err)` to `log.error(err, msg)`. This is also the *correct* idiomatic Pino usage — the error now serializes via `pino-std-serializers` as a structured `err` field instead of being dropped at runtime. **Double win**: type-safe AND the error logs actually contain errors now.
- Applied via sed to all 35 sites in `auth.fastify.ts` (18), `index.ts` (13), `news.ts` (4). Uses regex `request\.log\.error\('([^']*)', ([A-Za-z_][A-Za-z0-9_]*)\);` so it handles both `error` and `verifyError` identifiers.
- **Out of scope but worth noting**: `crews.ts` (15 calls) and `upload.ts` (2 calls) have the same broken pattern, but they're in `tsconfig.production.json`'s `exclude` list so they aren't type-checked. Same fix applies, same runtime win for error logs — worth cleaning up next time someone's in there.

### Build script cleanup (the main goal)

- `packages/backend/package.json:8` — removed `|| true` and `--noEmitOnError false`. New script: `prisma generate && tsc --project tsconfig.production.json && { cp src/services/*.js dist/services/ 2>/dev/null || true; }`.
- **Important**: the `|| true` is now inside a brace group `{ ...; }` so it only rescues the `cp` step (Windows/bash glob-empty compat), not the entire chain. Under the old form `tsc && cp || true` the `|| true` actually applied to the whole `&&` chain due to shell left-associativity — meaning a failing tsc would silently become exit 0. That's the exact bug we were fixing. Verified the brace-group scoping with `bash -c 'false && true && { false || true; }; echo $?'` → exit 1.
- Snapshot verification: `npx tsc --noEmit -p tsconfig.production.json` → 0 errors, `pnpm build` → exit 0, `dist/` fully populated including `oneFCLiveScraper.js`, `newsScraperService.js`, and all 21 route files.

Root cause: `incremental: true` in `tsconfig.json` + a stale `tsconfig.production.tsbuildinfo` from a prior broken build — tsc thought nothing had changed and skipped emit entirely.
