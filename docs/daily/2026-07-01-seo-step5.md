# 2026-07-01 — Programmatic SEO step 5: event lifecycle template (preview→results, one URL)

**SHIPPED + verified live same session** (commit `0c23c6c1`, one push — backend on
Render + web on Vercel). Unlike steps 3/4, no ordered two-push dance was needed:
the web changes degrade gracefully when the backend fields are absent (blocks just
don't render), and event pages have no build-time prerender to bake stale.

## Why this step

The event page is the URL that serves **two search-intent waves on one URL**
(plan's evergreen strategy): "how to watch X" before the card, "X results" after.
Before this, the page had one static title (the bare event name), no JSON-LD, and
no lifecycle-aware body content — the results wave especially hit a page whose
HTML never said "results" anywhere.

## What shipped

**Backend** (`routes/index.ts`)
- `/api/events/:id` select + response schema grew `aiEventSummary` +
  `aiEventConfidence` (fast-json-stringify strips unlisted props, so schema too).

**Web — `events/[id]/page.tsx`**
- **Lifecycle metadata swap.** UPCOMING/LIVE → `"<name> — Fight Card & How to
  Watch"`; COMPLETED → `"<name> Results & Fan Ratings"` with the derived card
  rating in the description ("fans rated this card 7.0/10 across 164 ratings").
- **SportsEvent JSON-LD, server-rendered**: organizer (promotion), location
  (venue + address), startDate (mainStartTime ?? date), confidence-gated
  aiEventSummary as description, `EventScheduled`/`EventCancelled` status,
  per-fight `subEvent`s (competitors + fight-page URL + per-fight
  `AggregateRating`), and a **card-level `AggregateRating` for completed events**
  — the rich-snippet play on "X results" queries.
- **Card rating is always DERIVED from fights** (weighted by rating count).
  `Event.averageRating`/`totalRatings` are dead fields
  (`lesson_dataset_aggregates_dishonest`) — never read the event row's aggregates.

**Web — `EventDetailClient.tsx`** (all SSRs via the client's initial render)
- Preview state: card-wide AI "why care" paragraph, gated at
  `aiEventConfidence >= 0.5` (same gate as the mobile home screen).
- Results state: fan-verdict line ("Fans rated this card **7.0/10** across 164
  ratings — best fight: Fiziev vs Torres (8.9/10)") — rating-only, reveals no
  outcomes, so it needs no spoiler gate (per-fight rating squares were already
  un-gated).
- Results state: **spoiler-safe collapsed `<details>` "Full results (spoilers)"**
  — one plain-text line per finished fight ("Rafael Fiziev def. Manuel Torres —
  KO/TKO R2"), main event first. This is the key spoiler/SEO compromise: web
  spoiler-free defaults **ON** at SSR (winners/methods are masked in the fight
  cards Google sees), so the collapsed list is how real results text gets into
  indexable HTML without flashing spoilers at browsing users.
