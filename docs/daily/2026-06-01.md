## Involuntary-logout root cause fixed (refresh-token rotation)

Mike was still getting logged out overnight / on morning app-open despite TWO
prior fixes (`49c22ac`, `a657d10`) — both of which only handled deploy-time 5xx.
Logouts happened with **no deploys**, so the redeploy theory was wrong.

**Root cause (server-side, `routes/auth.fastify.ts` `/refresh`):**
1. The endpoint **rotated** the refresh token on every call — minted a new one,
   shrank the old token's expiry to a **60s grace**. If the refresh *response*
   was lost (flaky morning connection) or the app was **suspended** mid-refresh,
   the client never stored the new token and the old token's 60s grace lapsed →
   next refresh 401s → hard logout. 60s was never long enough to survive a
   backgrounded app.
2. The `catch` returned **401 for ANY error** — including transient DB /
   cold-start failures — which the client treats as a genuine logout, defeating
   its careful transient-vs-invalid handling.

Both client refresh paths (`services/api.ts` AND `store/AuthContext.tsx` —
there are two copies) already handle transient correctly; the bug was the server
turning recoverable conditions into hard 4xx.

**Fix (`a63c355`, backend → auto-deploys on Render, helps the CURRENT app):**
- **Stop rotating** the refresh token. Mint a fresh access token, keep the same
  refresh token, slide its 90-day expiry forward. Kills the entire rotation-race
  class — a lost response or suspended app can't strand the client now.
- Only return **401 for genuine JWT failures** (JsonWebTokenError/TokenExpired/
  NotBefore); transient/DB errors now return **503** so both client paths keep
  the session and retry.
- Tradeoff: no rotation-based reuse detection. Acceptable here; reliable sessions
  >> marginal hardening. Existing users keep their tokens (no forced re-login).

Biometric quick-unlock (below) remains the *mitigation*; this is the actual fix.

## Biometric quick-unlock (Face ID / Touch ID / fingerprint sign-in)

**Why:** A user complained the app kept logging him out (involuntary logouts from
backend resets during heavy dev work — refresh tokens were getting invalidated
server-side). He asked for biometric login so getting back in is a snap. The
logout root cause isn't fully fixed, so this is the mitigation: make re-login
effortless.

**Key design decision — re-login, NOT refresh-token reuse.** The obvious approach
(gate the stored refresh token behind a biometric scan) fails in exactly this
scenario: the refresh token is the thing the server rejected. So instead we
capture the user's email/password at login time, store them in the OS keychain
(encrypted at rest via `expo-secure-store`), and gate a *full re-login* behind a
Face ID / Touch ID / fingerprint scan. A full re-auth is the only thing
guaranteed to work after server-side token invalidation.

### What shipped
- **`packages/mobile/utils/biometricAuth.ts`** (new) — availability check
  (`hasHardware && isEnrolled`), human label resolver (Face ID / Touch ID /
  Fingerprint), `promptBiometric()` (passcode fallback left on), and
  store/get/clear credential helpers keyed in SecureStore + an `biometricEnabled`
  flag in AsyncStorage.
- **`store/AuthContext.tsx`** — new `biometricAvailable` / `biometricEnabled`
  state (probed on mount), and `loginWithBiometric()`, `enableBiometricLogin()`,
  `disableBiometricLogin()`. `login()` refreshes stored creds when biometric is
  already on (covers password changes). `loginWithBiometric()` **self-heals**:
  if the stored password is genuinely rejected it clears creds + disables;
  network/server errors leave the setup intact.
- **`app/(auth)/login.tsx`** — "Unlock with {label}" button (shown only when set
  up on this device), auto-prompts once on mount, and offers to enable biometrics
  via an Alert right after a successful manual login (respects a "Not now"
  dismissal flag).
- **`app/advanced-settings.tsx`** — Security section with a toggle. Enabling from
  here needs the password (we don't retain it post-login), so it opens a confirm
  modal. Turning off / deleting the account clears stored creds.
- **`app.json`** — added the `expo-local-authentication` config plugin with the
  iOS `faceIDPermission` (NSFaceIDUsageDescription) string.
- Installed `expo-local-authentication@~17.0.8` (Expo SDK 54).

### Not done / next

## "Profile redesign not showing in prod" — Android OTA cold-start lag (NOT a dev build)

Mike reported the redesigned profile screen still wasn't showing on his Android
after many relaunches. **First theory (dev build on his phone) was WRONG — he had
the real prod build.** Corrected root cause: **Android only swaps in an OTA on a
true cold start (process death); with `LAUNCH_WAIT_MS=0` the swap is deferred to
the launch after download, and Android keeps the process warm across swipe-close,
so on an active phone the OTA lags a long time.** His old install was vc37 (redesign
not baked in, depending on the OTA). Reinstalling pulled **vc38** (redesign baked
in) and showed it instantly — the "instant on reinstall" tell proves the embedded
bundle had it, i.e. **vc38 is now live on Play.**

**Takeaway: the store build is the reliable delivery vehicle; the OTA is a slow
best-effort bonus on Android.** To get a change to all users, ship + fully roll out
the store build (Play 100% / App Store phased 100%) rather than relying on the OTA.

Verified the publish itself was fine by querying the update server *as the device*:
```
curl -i https://u.expo.dev/4a64b9f8-325e-4869-ab78-9e0674d18b32 \
  -H "expo-platform: android" -H "expo-runtime-version: 1.0.0" \
  -H "expo-channel-name: production" -H "expo-protocol-version: 1" \
  -H "accept: multipart/mixed"
```
→ HTTP 200 + the redesign bundle (group `ddfdbcba…`, branch `production`, rt
`1.0.0`). So any real Android binary on `production`/rt `1.0.0` gets it on a cold
start. `preview` channel is empty; `development` builds report versionName `1.0.0`
(vs `2.1.0` for the store build) — fastest tell.

**Confirmed live store state (via Apple/Play public lookup):**
- **iOS App Store version is now `2.1.1`** (released 2026-06-01 ~23:31 UTC) — Apple
  approved + it released. Redesign + biometric baked in. (Supersedes the "Waiting
  for Review" note above.)
- **Android Play version is `2.1.0`** (vc37 live; vc38 still in review). vc37 gets
  the redesign via the production OTA; vc38 will bake it in.

**Ensuring all users get it:** rely on the **store build**, not the OTA. iOS
converges as users auto-update to 2.1.1 (baked in); Android converges as users
auto-update to vc38 (baked in). **One action: confirm Play Production rollout for
vc38 is at 100%** (and App Store phased release at 100% if chosen). The OTA will
opportunistically catch vc37 stragglers on a cold start but is too slow/unreliable

Settings (pure JS) so "which build is this?" is diagnosable in seconds.
