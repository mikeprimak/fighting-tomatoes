# 2026-07-01 — Programmatic SEO workstream launched; steps 1 & 2 shipped live

## Context

Session started as a strategy conversation ("how do I make income at low usership?")
and converged on **programmatic SEO as the primary growth workstream** — the one
compounding, coder-friendly lever that also feeds income (web ad/affiliate) and the
acquisition narrative (web traffic as a separately-sellable asset). New source of
truth: `docs/plans/programmatic-seo-2026-07-01.md`; memory
`project_programmatic_seo_workstream`.

Honest framing captured: at ~200 active users direct app monetization is DOA
(ads/subs/affiliate all need traffic we lack — cf. `project_affiliate_too_small_revisit`).
Web traffic is the near-term income surface; SEO is how it grows. Fastest channel
overall would be influencer/community, but that's deferred by choice (user will do
direct influencer relationships "someday"; shareable card + Reddit already in motion).

## Shipped (live on prod, verified)

### Step 1 — slugs + 308 redirects + auto-slug middleware
- Nullable `@unique slug` on Fighter/Event/Fight (migration
  `20260701000000_add_seo_slugs`, additive; **deployed to prod via `migrate deploy`**).
- `src/lib/slug.ts` — deterministic ASCII slugifier + collision helper.
- `scripts/backfillSlugs.ts` — dry-run-default, idempotent, batched. Backfilled
  **all** prod rows: 9941 fighters, 1572 events, 14842 fights → **0 null**. (First
  fights run got cut off by a `timeout`; re-ran to finish — script is idempotent.)
  Real collisions were tiny: ~18 dup-name events, ~38 dup-fighter name variants
  (both known/deferred data issues), and fight `-2` slugs are almost all real rematches.
- `src/lib/slugHooks.ts` — a single Prisma `$use` middleware auto-fills `slug` on
  create/upsert for the three models. Chosen over per-call-site wiring (~15+ scattered
  scraper parsers + `fightUpsert`/`fighterMatcher` + live trackers) and over `$extends`
  (keeps the exported `PrismaClient` type unchanged). P2002-on-slug is retried; any
  other P2002 rethrown (unchanged behavior). Verified E2E against prod (create + upsert
  create-branch, fight slug resolves fighter names) then cleaned up test rows.
- Backend `/api/fighters/:id`, `/api/events/:id`, `/fights/:id` resolve **id-or-slug**
  and return `slug` (added to selects **and** response schemas — fast-json-stringify
  strips unlisted props). Fighter follower subqueries + fight notification-engine call
  now use the resolved real id, not the URL param.
- Web fighter/event/fight pages 308-redirect UUID→slug (`permanentRedirect`), emit
  `canonical`, and pass the **real UUID** to their clients so all client data calls
  (rate/follow/aggregate-stats) are unchanged. Event page fetches fights by the
  resolved UUID (the `?eventId=` filter only matches UUIDs).

### Step 2 — fight page `<h1>` + SportsEvent/AggregateRating JSON-LD
- `packages/web/src/app/fights/[id]/page.tsx` now server-renders a single semantic
  `<h1>` (matchup) + `SportsEvent` JSON-LD with `AggregateRating` from our community
  rating (emitted only when `totalRatings>0` — never fabricated), `competitor` Person
  entries, and `superEvent` (parent event). Event is a rich-snippet-eligible type, so
  this is how our unique fan-rating data can surface as stars in search.
- The recap / story / stakes already SSR via the client component's initial render
  (spoiler-free + auth default off server-side). The client's rating *widget* is
  `stats`-gated (client-only) so the numeric rating isn't in SSR HTML — JSON-LD carries
  it for Google.
