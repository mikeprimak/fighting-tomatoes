### 1. Click tracking (owned data)

- Migration `20260626000000_add_broadcast_clicks` ‚Äî **hand-authored SQL, applied to prod via `migrate deploy`** (status was clean, only this migration pending). Table is live on Render.

### 2. Tracked redirect endpoint
- **`GET /api/r/b?c=<channelId>&e=&r=&s=&t=&p=&b=`** (`routes/broadcasts.ts`). Resolves the real target server-side (eventDeepLink ‚Üí affiliateUrl ‚Üí homepageUrl), logs a `BroadcastClick`, then 302s out. **No open-redirect** ‚Äî destination is never carried in the URL, always re-resolved from `channelId`, so updating an affiliate URL takes effect immediately for all clients. Logging failures never block the redirect.
- `broadcasts.ts` now rewrites every `deepLink` in the API response into a tracked `/api/r/b` URL (via `buildTrackedLink`). **Web + mobile needed zero render changes** ‚Äî they already open `entry.deepLink`.

### 3. Platform tagging
- `x-client-platform: web` / `mobile` header added to both api clients' central request fns (web `lib/api.ts`, mobile `services/api.ts`); added to CORS `allowedHeaders`; read in the broadcasts route and baked into the tracked link as `p`. Lets us segment click revenue web vs mobile.

### 4. Admin panel (web-based, `public/admin.html`, Broadcasts tab)
- **Affiliate URL** field added to the channel editor (the lever). Channel list shows a üí∞ "Affiliate: Yes/No" column + "X of Y channels monetized" summary.
- New **"üí∞ How-to-Watch Click Revenue"** panel: total / monetized / homepage-lost clicks, by region, by platform, per-channel click table with an "Add link" shortcut for un-monetized channels. Backed by new `GET /admin/broadcast-clicks/stats?days=`.
- **Always-available reference card** at the top of the Broadcasts tab (collapsible `<details>`): how the monetization works, the lever, "do this in order" signup steps, programs-by-region table, the reality-check caveat, compliance note. ### üî¥ INCIDENT ‚Äî web prod down: "Failed to load events" on every screen (caught + fixed same session)
**Symptom:** goodfights.app rendered header/sidebar but no fight data anywhere; "Failed to load events." Mobile was fine.

**Root cause:** the `x-client-platform` header added this session made the **web** api client send a custom header on every request ‚Üí browser does a **CORS preflight** ‚Üí the live CORS config rejected the unlisted header ‚Üí browser blocked *every* API call. (Mobile uses native fetch, no preflight, so it was unaffected ‚Äî a useful tell.)

**The trap:** the backend has **TWO CORS configs**, and I edited the wrong one.
- `src/middleware/cors.ts` (express `cors` package) ‚Äî wired into `src/app.ts`. **`app.ts` is NOT the running entrypoint.** This file is effectively dead. My session edit added `x-client-platform` here (in commit `776adb97`) ‚Äî did nothing for prod.
- **`src/server.ts` ‚Üí `@fastify/cors` (lines 104‚Äì127) is the LIVE config.** Its `allowedHeaders` was `['Content-Type','Authorization']` and never got the new header.
- **Diagnosis signal:** the live preflight returned `access-control-allow-headers: Content-Type, Authorization` ‚Äî matching *neither* cors.ts file (no `X-Requested-With`, no `PATCH`). That mismatch is what revealed a third, separate config was actually in effect.

**Fix (commit `dd961864`):** added `x-client-platform` to `server.ts:125` `allowedHeaders`. Render redeploy ~210s. Verified live: preflight now returns `...Authorization, x-client-platform`; a real `GET /api/events?upcoming=true` with the web origin + header returns `200` + 20 events + correct ACAO. **No web redeploy needed, no data affected.**

**Lesson:** before adding any request header to an api client, confirm WHICH CORS config is actually live (`server.ts` @fastify/cors here, NOT middleware/cors.ts). Captured as memory `lesson_backend_two_cors_configs_server_ts_live`.

## Live-event bug (remote control) ‚Äî ONE FC duplicate-fight on `onefc` daily scrape

**Report (Mike, during live event):** "ONE Friday Fights 160 & The Inner Circle 20" ‚Äî in real life the card was on the final fight (Numsurin Chor Ketwina vs Songchainoi Kiatsongrit), but the app showed **Hern Looksuan vs Okuwaki as up next**.

**Investigation (prod DB):** The event had **duplicate Fight rows** under different fighter-name forms:
- `#9 COMPLETED` *Hern **NF** Looksuan vs Ryuya Okuwaki* (KO, tracker-advanced 13:50)
- `#9 UPCOMING` *Hern Looksuan vs Ryuya Okuwaki* (phantom, **created 14:31 mid-event**, 0 ratings/predictions, never tracked)
- Same pattern at `#5` (Panpadej **NF** Looksuan vs Panpadej Looksuan) ‚Äî already auto-CANCELLED.

The orphan `#9 UPCOMING` row was what surfaced as "up next." The real main event resolved on its own during triage (Numsurin completed UD 16:27, event ‚Üí COMPLETED).

**Immediate fix:** cancelled the orphan `#9` row (`959227b6-‚Ä¶`, set `fightStatus=CANCELLED`, `completionMethod=manual`) ‚Äî matches the existing cancelled-dup pattern on this same card; 0 user data. Card now has 0 stray UPCOMING rows.

**Root cause:** ONE FC reports the same Thai fighter under drifting name forms ‚Äî JSON-LD gives canonical `Hern Looksuan`, the display name embeds a camp token `Hern NF Looksuan`. Each distinct firstName/lastName ‚Üí a **separate Fighter row** ‚Üí the order-sensitive Fight unique key `(eventId, fighter1Id, fighter2Id)` never matches ‚Üí the **daily scrape silently creates a duplicate bout** the live tracker had already advanced. The **live tracker was already token-aware** (`oneFCLiveParser.findFightByFighters` dedups by first/last token overlap); the **daily scraper's `upsertFightSwapAware` was not** ‚Äî that asymmetry was the gap. (`upsertFighterWithFuzzyMatch` in `fighterMatcher.ts` is **dead code** ‚Äî references the dropped `fighterAlias` table ‚Äî so it was not an option.)

**Durable fix (uncommitted working tree):**
- `src/utils/fightUpsert.ts` ‚Äî new opt-in `{ sameEventNameDedup: true }` guard on `upsertFightSwapAware`. On the CREATE branch (i.e. exact-id lookup missed), it scans the same event for a fight whose two fighters' name tokens overlap the scraped pair (either orientation, prefers non-CANCELLED) and **updates that fight instead of creating a duplicate**. Fighter ids are deliberately NOT re-pointed (the existing fight may carry live-tracker state); the freshly-created canonical Fighter row is left fight-less and harmless. Mirrors the live tracker's matcher so the two paths can't diverge. Added as a 2nd overload returning `Fight` (non-null) ‚Äî backward compatible; Tapology `{crossEventDedup:true}` callers and all no-guard callers unaffected.
- `src/services/oneFCDataParser.ts` ‚Äî daily fight upsert now passes `{ sameEventNameDedup: true }`.

**Verification:** Replayed the heuristic over the real 18-fight card: **2 true-dup matches (Hern, Panpadej), 0 false positives** ‚Äî safe despite many shared gym tokens (Saenchai/PK/Looksuan), because tokens are whole first/last name strings and both fighters must pair. Backend `tsc --project tsconfig.production.json --noEmit` ‚Üí **0 errors**.

**Shipped:** commit `425dc320`, pushed to `main`. **The ONE FC daily scrape runs on GitHub Actions** (`onefc-scraper.yml`, cron `0 */6 * * *`) which does `actions/checkout` + `pnpm build` each run ‚Äî so the push to `main` IS the deploy; the next scheduled run (‚â§6h) uses the new dedup guard. No Render redeploy and no VPS deploy needed (the backend's `dailyAllScrapers.ts` crons are not wired into a running server, and the VPS live trackers were already token-aware).
