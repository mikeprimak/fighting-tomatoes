### Top Rank "Zayas vs Ennis" — Third Recurrence

**What happened:** The bogus event came back again this morning, created at 2026-04-15T02:10:50Z with run-time date and 2 fights. Same shape as the 04-13 and 04-14 recurrences.

**Root cause this time:** Yesterday's fix `f54b12d` (parser null-return + cron disable) was committed locally at 01:47 EDT on 04-14 but not actually **pushed to GitHub** until ~22:42 EDT on 04-14 (bundled with the Sentry commit `d40c60c`). The scheduled run at 02:09 UTC on 04-15 fired on `head_sha=8dc3abd` — the commit *before* the fix. So the cron was still active on GitHub's side, and the parser still had the `return new Date()` fallback. Empty `dateText` → event stamped with run time. Exactly the same bug we thought we killed, just running from a stale commit.

**Cleanup:** Deleted event `b6b9623a-f8cb-4d43-bf08-f29c0158c0ff` (2 fights removed) via Prisma.

**No code changes needed** — `f54b12d` is now actually on `origin/main`, the cron is gone from the workflow on main, and the parser returns null for unparseable dates. Next scheduled slot (00:00 UTC 04-16) will not fire.

**Lesson:** Local `git log` showing a commit on `main` does not mean it's pushed. `git status` would have shown "ahead of origin/main by 1 commit." For fixes that depend on GitHub-hosted CI (scheduled workflows, Actions-driven scrapers), always verify `git push` landed before declaring the fix live.

## Guest UX: Hype/Rate Modals

### Problem
Guests (users who tapped "Continue as Guest" on login) could open the hype and rating modals but:
1. Tapping a flame (hype) or star (rating) did nothing — silent no-op, no feedback.
2. The "See Comments >" link was gated behind `isAuthenticated &&`, so guests couldn't navigate from the modal to the detail screen to read discussion.

### Fix
The `VerificationContext.requireVerification(actionDescription)` helper already existed and already handles the guest case — it pops `VerificationRequiredModal` with a "Sign In / Create Account" prompt. It was wired throughout the detail screens but not the two quick modals.

**`components/UpcomingFightModal.tsx`:**
- Imported `useVerification`.
- `handleHypeSelection` now calls `requireVerification('hype this fight')` before proceeding — guests get the login prompt on flame tap.
- `handleNotifyPress` similarly calls `requireVerification('get notified about this fight')`.
- Pulled the "See Comments >" link out of the `isAuthenticated &&` block; only the comment `TextInput` stays gated. Removed `isAuthenticated` from the `preFightComments` query's `enabled` so the count loads for guests too.

**`components/CompletedFightModal.tsx`:**
- Same pattern: `useVerification` imported, `handleRatingSelection` calls `requireVerification('rate this fight')`.
- "See Comments >" link moved outside `isAuthenticated &&`; review input stays gated. `fightDetailData` query no longer requires auth to run.

### Behavior now
- Guest taps flame/star → `VerificationRequiredModal` (isGuest branch) opens with Sign In / Create Account buttons.
- Guest sees "See Comments >" on both modals and can navigate to `UpcomingFightDetailScreen` / `CompletedFightDetailScreen` to read discussion. (Detail screens already use `requireVerification` for post/upvote/follow actions.)
- Logged-in, verified users: no change.

## Legacy Rating Repair — 170 ratings across 15 users

### Problem
User `notmotcha@gmail.com` (user reported it as `not-motcha@gmail.com` — hyphen was wrong) reported that his recent GoodFights ratings were missing. Investigation found his legacy account had 198 ratings but only 167 in GF — 31 missing. His complaint that they were "from the last month or so" was inaccurate; his most recent legacy *and* GF rating is both UFC 325 (2026-01-31). The missing ratings were spread across 2005–2025, not concentrated in a single window.

### Root cause
The **original pre-Feb-2026 legacy migration** left behind gaps — individual legacy fights that either (a) failed fuzzy name matching, or (b) matched to a GF fight UUID that was later deleted when its event was re-scraped. The April 13 incremental migration was scoped to events since Feb 1, so it didn't revisit older data.

Also confirmed: `legacy-data/fight-mapping.json` is partly stale — for the sampled rows I checked (UFC 323–325), the GF UUIDs in the mapping file no longer exist in the DB. Not actively harmful (the April 13 migration re-matches by name+date) but worth knowing — the mapping should not be trusted as a source of truth.

### Fix
New script `packages/backend/scripts/legacy-migration/mysql-export/repair-missing-ratings.js`:
- Scans every legacy user's ratings / tags / reviews in MySQL (not date-limited).
- Matches legacy fights to GF fights by **sorted fighter lastnames + event date ±1 day** against the *live* GF DB (ignores the stale mapping file).
- INSERT-only — `skipDuplicates: true` on every `createMany`; nothing is updated or deleted.
- Won't create new users (only repairs users who already exist in GF).
- Recalculates `averageRating`/`totalRatings` only for the fights actually touched.
- Dry-run by default; `--execute` to write. `--user <email>` for single-user repair. `--only-ratings` to skip tags/reviews.

### Results
Ran `--execute` against production Render DB. Deltas matched dry-run exactly:

| | before | after | delta |
|---|---|---|---|
| `FightRating` | 67,413 | 67,583 | **+170** |
| `FightTag` | 2,255 | 2,297 | **+42** |
| `FightReview` | 797 | 798 | **+1** |
| notmotcha ratings | 167 | 199 | **+32** |

386 legacy ratings were safely skipped because the underlying fight doesn't exist in current GF at all (long-tail events never scraped, or fighters that don't exist as rows).

Top users repaired: charlie.wilsea (36), notmotcha (32), arballoanhel (15), brandca126 (15), cry.hagen (12), muradianherman (9), fightrate / rick_hockey8 (7 each).

### Lesson / when to re-run
If another user complains about missing ratings, run `node repair-missing-ratings.js --user <their-email>` to dry-run, then `--execute` if it shows gaps. The script is idempotent and safe to re-run any time.
