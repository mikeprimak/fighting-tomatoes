# 2026-06-29

## Shareable hype/rate card (BACKLOG §7, "Shareable hype/rate card") — Phase 1 SHIPPED to prod

Converted the post-hype and post-rate **reveal modals** into a branded, shareable
card. Virality + install driver: a user hypes/rates a fight → sees a card they can
share → the share carries a `goodfights.app/fights/<id>` link that pulls recipients
in to rate the same fight.

### What shipped (commit `503dd7bf`, merged to main, OTA'd to production)
- **`components/ShareableFightCard.tsx`** (new) — branded card, **always dark-themed**
  (hardcoded palette, not device theme, so every shared image looks identical).
  - `GOOD FIGHTS` wordmark rendered as **text, not an image** → stays crisp when the
    view is later rasterized to a PNG.
  - Fighter photos (`getFighterImage` + default-alpha fallback) + names, event name +
    date (`formatEventName`/`formatDate`), big hype/rating value with heatmap color
    (`getHypeHeatmapColor`), `goodfights.app` CTA.
  - `forwardRef` exposes the card View — **already wired for the future PNG capture**.
  - Takes a **minimal `ShareCardFight` type** (not `FightData`) so it stays decoupled
    from both `FightData` and the parents' loose local `Fight` interfaces.
- **`HypeRevealModal` / `RatingRevealModal`** — rebuilt around the card. **Dropped the
  comparison line + Fan DNA "cute phrase"** (per Mike). Kept the distribution chart as
  in-app context (not part of the shared image). Added a primary **Share** button +
  secondary Close. Removed the now-unused `averageHype`/`dnaLine`/`dnaLoading` props.
- **`utils/shareFightCard.ts`** (new) — `shareFightLink()` uses the **built-in RN
  `Share`** (no native deps → OTA-able): punchy message + the fight deep link. On iOS
  the URL is passed as a distinct field (unfurls via the future web OG image); on
  Android it's embedded in the message text.
- `UpcomingFightModal` / `CompletedFightModal` — pass `fight` into the reveal modals.

### Process notes / lessons
- Started in a **git worktree** (`worktree-feature+shareable-hype-card`) for isolation
  from a concurrent data/correctness window. The worktree had **no `node_modules`**
  (fresh from origin), which made Expo Go "spin forever" — that was the worktree, not a
  real Expo-Go incompatibility. **Expo Go on SDK 54 DOES run this app** (all the native
  modules are `expo-*`, which Expo Go bundles); `expo-dev-client` is only for native
  builds. Run `expo start` from the **normal checkout**, not a deps-less worktree.
- Mike abandoned the worktree mid-task → merged the feature commit to main, pushed,
  OTA'd to prod. He tests on prod (only the production store app is installed, so an OTA
  reaches ALL users — no per-device targeting without a separate dev/preview build).
