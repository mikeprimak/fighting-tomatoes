### Mobile: FollowFighterButton race fix
- **Bug**: After tapping "+" to follow, a second tap to unfollow sometimes did nothing.

## Tapology live tracker: bogus round/time on decision results

- **Bug**: Inoue vs. Nakatani showed "Inoue won by UD in Round 55"; multiple other Tapology-tracked events had the same shape (round = a fighter's weight in kg/lbs, time = scheduled total like `36:00`).
- **Root cause** in `tapologyLiveScraper.ts`:
  - `parseRound` regex `/(?:round|r)\s*(\d+)/i` had no word boundary on the `r` alternative, so it matched the trailing `r` in "Champion" + whitespace + "55 kg" → 55.
  - `parseTime` regex `/(\d{1,2}:\d{2})/` matched `36:00` from `"12 Rounds, 36:00 Total"` (the scheduled distance, not the end time).
  - Both regexes ran against the entire `<li>` text, which includes weight-class blurbs, ages, and the scheduled-distance line — not just the result row.
- **Fix**:
  - Scope round/time extraction to the parent of the method `<span>` (the result row), not the whole `<li>`.
  - Use a positional pattern `([0-5]:[0-5]\d)\s*Round\s+(\d+)` to pull the leading end-time + ended-in-round pair from rows like `"KO/TKO 1:47 Round 7/12, 19:47 Total"`. Avoids grabbing the trailing total elapsed.
  - For decisions/draws/NC, leave round/time undefined — the fight went the distance, there is no end-round/time.
  - Tightened `parseRound` to require literal word `round` with `\b`; tightened `parseTime` to `[0-5]:[0-5]\d` so it can't match `36:00`/`24:00`-style totals.
  - Parser now explicitly clears `round`/`time` when the scraped method is a decision-class result, so re-running corrects existing rows instead of silently leaving stale data.
- **Files**:
  - `packages/backend/src/services/tapologyLiveScraper.ts`
  - `packages/backend/src/services/tapologyLiveParser.ts`
- **Cleanup**: One-time `updateMany` cleared `round`/`time` on 33 decision-class fights across 6 events (Inoue vs. Nakatani, Whittaker vs. Suarez, Baumgardner vs. Shin, Fury vs. Makhmudov, DBX 6, Dubois vs. Harper).
- **Affects all Tapology-tracked promotions** (Top Rank, Golden Boy, Gold Star, Matchroom, MVP, RIZIN, PFL, BKFC, Zuffa Boxing, Karate Combat, Dirty Boxing) — same scraper module.

## Section-start notifications for non-tracker events

- **Problem**: bell on a fight only worked for events with a live tracker (UFC/BKFC/PFL/RIZIN/Zuffa/KC/DBX). For non-tracker orgs (Matchroom, Oktagon, ONE FC, boxing co-promotions, etc.), `notificationsAllowed` was gated on `hasLiveTracking`, so users couldn't even set a bell. Ships the "non-tracked-org card-start fallback" lane from `project_notification_system_v2.md`.
- **Design**: deliver one push per (user, event-section) ~15 min before each section's start time, using `event.{earlyPrelimStartTime, prelimStartTime, mainStartTime}` as the trigger. Fights bucket by `cardType` (early prelim / prelim / main, unknowns→main). Falls back to a single `event.date`-based fire when no section start times are set. Lead time matches existing `notifyMinutesBefore: 15`.
- **Backend**:
  - `services/notificationService.ts` — new `notifyEventSectionStart(eventId, sectionFightIds, eventName, sectionLabel)`. Queries `fightNotificationMatch` rows where `fightId IN sectionFightIds AND isActive AND !notificationSent`, groups by user, sends one push per user, marks all their rows in the section as sent. Title: `"${event} ${section} starts soon!"`. Body: N=1 → `"Smith vs Jones is up soon."`, N≥2 → `"X fighters you follow are fighting soon."`. Re-fires for the next section because the next section's match rows are still unsent.
  - `services/eventLifecycle.ts` — new "Step 1.7" between dispatch (1.5) and fight-completion (2). Iterates `UPCOMING|LIVE` events with `!isProductionScraper(scraperType)`, builds section list from non-null section start times + cardType buckets, fires the notifier 15 min before each section. `notificationSent` prevents re-fires across the 5-min cadence.
  - `routes/index.ts` + `routes/fights.ts` — dropped `hasLiveTracking &&` from the `notificationsAllowed` expression in 4 sites. Admin allowlist (`SystemConfig` key `notify_promotions`) is now the only gate. To enable for a non-tracker promotion, add it to that list via the admin panel.
- **Mobile**:
  - `components/UpcomingFightModal.tsx` — when bell is shown and `fight.event.hasLiveTracking === false`, renders an italicized caption under the bell row: `~15 min before [section]. (No live tracker for this org.)` Section computed from `fight.cardType`. Sets honest expectations before the user taps so they don't expect walkout-precise timing.
- **Edge cases handled**:
  - For events with both prelim and main start times: Step 1 flips event to LIVE at the earliest section, Step 2 immediately marks ALL upcoming fights COMPLETED on non-tracker events. Step 1.7 doesn't filter by fight status — it queries match rows directly — so the main-card section still fires its notification 15 min before mainStartTime even though the fight rows are technically COMPLETED in our DB by then. Semantically OK: "fighting soon" refers to the real-world broadcast, not our internal status.
  - Fights with null/missing `cardType` (common for orgs whose scrapers don't populate it) get bundled into the main-card bucket.
- **Files**:
  - `packages/backend/src/services/notificationService.ts`
  - `packages/backend/src/services/eventLifecycle.ts`
  - `packages/backend/src/routes/index.ts`, `packages/backend/src/routes/fights.ts`
  - `packages/mobile/components/UpcomingFightModal.tsx`
- **Open**:
  - No double-fire suppression with the morning-digest lane (lane 1) yet — digest doesn't exist yet. Add when it lands.
  - Admin still has to add non-tracker promotions to `notify_promotions` for the bell to appear. No auto-enrollment.

## Followed Fighters: stale `localUnfollows` on re-follow from carousel

- **Bug reported by user**: tap + in carousel, tap toggle off in My list (works), tap + on the same fighter again in carousel — fighter appears in My list with toggle stuck OFF, also stays in carousel for a beat, and after refetch the toggle is still OFF.
- **Root cause**: `localUnfollows` (the per-screen Set of "just toggled off" IDs) is only cleared on `useFocusEffect`. When the user re-followed via the carousel, `handleCarouselFollow` set the optimistic `justFollowed` entry but never removed the stale ID from `localUnfollows`. The render path then read `isFollowing = !localUnfollows.has(id)` → `false` (toggle OFF) and `followedIdSet` filtered the fighter out (so the carousel didn't hide them). The toggle-on path in `handleToggleFollow` already cleared `localUnfollows[id]`; the carousel path was the missing mirror.
- **Fix**: `handleCarouselFollow` now drops the fighter's ID from `localUnfollows` before seeding the optimistic entry and firing the mutation.

## Notes
- Classic React Query optimistic-update anti-pattern: prop-derived state synced via `useEffect` will get clobbered by any parent refetch that lands while the user's intent is in flight. The intent-ref pattern (only sync once the prop catches up) is the durable fix when you can't move fully to React Query's built-in optimistic updates via cache mutations.
- The two button instances on each modal (one per fighter) and the one on `followed-fighters.tsx` and `CompletedFightModal` all benefit from the same fix since they share the component.
- Followed Fighters local-state symmetry rule: any handler that initiates a follow must clear that fighter's ID from `localUnfollows`; any handler that initiates an unfollow must drop their entry from `justFollowed`. Both paths today do both — keep that invariant on future edits.

## PFL scraper: detect card sections + correct "Early Card" mapping

- **Bug found by user during PFL Sioux Falls (live tonight)**: admin panel showed "Early Prelims: 7:00, Main Card: 10:00", but the upcoming-events screen in mobile collapsed all 12 fights under a single "Main Card 10pm" section. Two issues stacked:

  - Walk `.card-date` headers and `[id^="fightCardWrapper"]` nodes in document order to build a `wrapperId → cardType` map. Search the entire document (not just `#fight_card_component`) because the headers sometimes sit as siblings above the AJAX-loaded fight card.
  - For each fighter link, record its enclosing wrapper. When pairing fighters into fights, resolve cardType from either fighter's wrapper.
  - Map "Early Card" → cardType `Prelims` (and `prelimStartTime`); reserve "Early Prelims" for the rare three-tier case.
  - Prefer the `.card-date` "Main Card" time over the site-wide `DateTime.fromISO` countdown (which is keyed to the next featured event, not necessarily the current one).
- **Parser** (`pflDataParser.ts`): `prelimStartTime` and `earlyPrelimStartTime` now pass through `null` (was `|| undefined`) so a re-scrape clears stale values from previously-buggy parses. `mainStartTime` left as `|| undefined` since `parsePFLEventStartTime` returns null on transient AJAX hiccups and we don't want to wipe a known-good main start time.
