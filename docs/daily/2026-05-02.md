## Changes Made

### Mobile: FollowFighterButton race fix
- **Bug**: After tapping "+" to follow, a second tap to unfollow sometimes did nothing.
- **Root cause**: `useEffect(() => setOptimistic(isFollowing), [isFollowing, fighterId])` re-synced local state from the prop on every parent re-render. After `onSuccess` invalidates `['fight']`/`['fights']`, the parent refetch could land mid-tap and overwrite the user's optimistic flip back to "following". Also `disabled={mutation.isPending}` was swallowing rapid taps inside the small in-flight window.
- **Fix**: Track user intent in an `intentRef`. The prop-sync effect now only adopts the prop when it agrees with the latest intent (or no intent is pending). Removed `disabled`. Bumped `hitSlop` from 8 → 14.
- File: `packages/mobile/components/FollowFighterButton.tsx`
- Commit: `3763cbd`
- Shipped: EAS Update to `production` (Android runtime 1.0.0, iOS runtime 2.0.1). Two app restarts to apply.

## Co-Promo Event Deduplication

- **Bug**: User reported two events for one card — `Golden Boy: Benavidez vs. Zurdo` and `Benavidez vs. Zurdo` (Top Rank). Boxing co-promotions appear on both promoters' Tapology pages, both scrapers create their own Event row, and the parsers' dedup keys (`ufcUrl` OR `name + date`) miss because URLs differ and each scraper prefixes the name with its own promo.
- **Fix**: Added a post-scrape merge pass that finds Events with matching main-event fighter pairs (`orderOnCard = 1`, ignoring CANCELLED fights) within ±36h on different promotions, picks a canonical (more fights → more ratings → earlier `createdAt` wins), re-points all child rows (ratings/reviews/predictions/comments/tags/follow-matches/crew data/recommendations/activities) to canonical fights with conflict-resolution favoring canonical, recomputes aggregate stats, and deletes the duplicate event. Runs in a single Prisma transaction per pair.
- **Files**:
  - `packages/backend/src/services/mergeDuplicatePromoEvents.ts` (core logic)
  - `packages/backend/scripts/merge-duplicate-promo-events.ts` (CLI with `--dry-run`)
  - `.github/workflows/dedupe-events.yml` (daily 21:00 UTC, after Matchroom at 20:00)
- **Decisions**:
  - Drops the duplicate's promotion entirely instead of combining strings — combined promo strings (e.g. `"Top Rank / Golden Boy"`) would silently break promotion filters and notification rules expecting exact matches.
  - Strict main-event matcher: requires unambiguous main events (single fighter pair at lowest non-CANCELLED `orderOnCard`) on both sides. Rejects events with ambiguous mains rather than guessing — the user's parsers' self-healing already CANCELS stale fights, so the live roster is the right thing to compare against.
- **Applied**: 2 pairs merged on prod
  - `Zayas vs. Ennis` (Matchroom) → merged into `Zayas vs. Boots` (Top Rank); 4 fights collapsed
  - `Golden Boy: Benavidez vs. Zurdo` → merged into `Benavidez vs. Zurdo` (Top Rank); 1 fight collapsed
  - 0 ratings affected (events were upcoming).

## Tapology live tracker: bogus round/time on decision results

- **Bug**: Inoue vs. Nakatani showed "Inoue won by UD in Round 55"; multiple other Tapology-tracked events had the same shape (round = a fighter's weight in kg/lbs, time = scheduled total like `36:00`).
- **Root cause** in `tapologyLiveScraper.ts`:
  - `parseRound` regex `/(?:round|r)\s*(\d+)/i` had no word boundary on the `r` alternative, so it matched the trailing `r` in "Champion" + whitespace + "55 kg" → 55.
  - `parseTime` regex `/(\d{1,2}:\d{2})/` matched `36:00` from `"12 Rounds, 36:00 Total"` (the scheduled distance, not the end time).
  - Both regexes ran against the entire `<li>` text, which includes weight-class blurbs, ages, and the scheduled-distance line — not just the result row.
- **Fix**:
  - Scope round/time extraction to the parent of the method `<span>` (the result row), not the whole `<li>`.
  - Use a positional pattern `([0-5]:[0-5]\d)\s*Round\s+(\d+)` to pull the leading end-time + ended-in-round pair from rows like `"KO/TKO 1:47 Round 7/12, 19:47 Total"`. Avoids grabbing the trailing total elapsed.
  - For decisions/draws/NC, leave round/time undefined — the fight went the distance, there is no end-round/time.
  - Tightened `parseRound` to require literal word `round` with `\b`; tightened `parseTime` to `[0-5]:[0-5]\d` so it can't match `36:00`/`24:00`-style totals.
  - Parser now explicitly clears `round`/`time` when the scraped method is a decision-class result, so re-running corrects existing rows instead of silently leaving stale data.
- **Files**:
  - `packages/backend/src/services/tapologyLiveScraper.ts`
  - `packages/backend/src/services/tapologyLiveParser.ts`
- **Cleanup**: One-time `updateMany` cleared `round`/`time` on 33 decision-class fights across 6 events (Inoue vs. Nakatani, Whittaker vs. Suarez, Baumgardner vs. Shin, Fury vs. Makhmudov, DBX 6, Dubois vs. Harper).
- **Affects all Tapology-tracked promotions** (Top Rank, Golden Boy, Gold Star, Matchroom, MVP, RIZIN, PFL, BKFC, Zuffa Boxing, Karate Combat, Dirty Boxing) — same scraper module.

## Notes
- Classic React Query optimistic-update anti-pattern: prop-derived state synced via `useEffect` will get clobbered by any parent refetch that lands while the user's intent is in flight. The intent-ref pattern (only sync once the prop catches up) is the durable fix when you can't move fully to React Query's built-in optimistic updates via cache mutations.
- The two button instances on each modal (one per fighter) and the one on `followed-fighters.tsx` and `CompletedFightModal` all benefit from the same fix since they share the component.
