### Backend: Tapology live tracker wrongly auto-completed an upcoming event
- **Symptom**: DBX 6: Garcia vs. Perez (scheduled for tonight, Apr 10) showed as COMPLETED in the mobile app with zero fights visible. The event hadn't even started.
- **Investigation**:
  - Event row: `eventStatus: COMPLETED`, `completionMethod: tapology-tracker-auto`, `updatedAt: 2026-04-10T18:03:32Z` — marked complete at 2:03 PM EDT, hours before the 10 PM EDT main card.
  - All 12 fights: `fightStatus: CANCELLED`, updated in a burst between `18:03:31` and `18:03:32`, none had a `winner`, none had `trackerWinner`, all had `cardType: null`. The mobile app was hiding them because they were CANCELLED.
  - Event had `scraperType: tapology`, `ufcUrl: https://www.tapology.com/fightcenter/events/140498-dbx-6-garcia-vs-perez` (correct URL).
- **Root cause**: Two cascading bugs in the Tapology live tracker:
  1. `tapologyLiveParser.ts:316-318` set `hasStarted = true` if the scraped page reported `status === 'live' | 'complete'` OR any scraped fight was marked complete. When `hasStarted` was true, the "missing from scraped data" sweep at line 334-338 would CANCEL every DB fight that wasn't in the scraped list. With a broken/empty scrape (0 fights returned, or 0 matches because of name mismatches), **every single fight on the card got cancelled in one pass**. The exact reason the tapology scrape came back useless for DBX 6 isn't fully known — either Tapology's page was flagged 'complete' prematurely, or the fighter-name matcher failed for all 12 fights — but either way the parser had no safety net.
  2. `runTapologyLiveTracker.ts:252-278`'s `autoCompleteTapologyEvent` considered the event complete if **every** fight was in a terminal state (COMPLETED **or** CANCELLED). So once the parser nuked all fights to CANCELLED, the runner immediately marked the event COMPLETED with `completionMethod: tapology-tracker-auto`.
- **Data fix** (manual Prisma updates):
  - Event: `eventStatus` → UPCOMING, `completionMethod` → null
  - All 12 fights: `fightStatus: CANCELLED` → UPCOMING (via `updateMany`)
  - Corrected start times (the DBX scraper had populated wrong times — 4 PM EDT instead of 10 PM): `date` → `2026-04-10T04:00:00Z` (matches DBX 5 midnight-ET convention), `prelimStartTime` → `2026-04-11T00:00:00Z` (8 PM EDT), `mainStartTime` → `2026-04-11T02:00:00Z` (10 PM EDT).
- **Code fix — two guards, belt-and-suspenders**:
  1. `tapologyLiveParser.ts`: compute `scrapeLooksValid = scrapedData.fights.length > 0 && result.fightsMatched > 0`. The missing-from-page cancellation sweep AND the UPCOMING→LIVE flip now both require `scrapeLooksValid`. An empty or unmatched scrape logs a skip message and does nothing to the DB.
  2. `runTapologyLiveTracker.ts`: `autoCompleteTapologyEvent` now additionally requires at least one fight to actually be `COMPLETED` (not just CANCELLED). `allTerminal && !hasAnyCompleted` is treated as a broken-scrape symptom and logs a skip instead of marking the event complete.
- Files: `packages/backend/src/services/tapologyLiveParser.ts` (scrapeLooksValid guard around cancellation sweep and LIVE flip), `packages/backend/src/scripts/runTapologyLiveTracker.ts` (hasAnyCompleted guard in autoCompleteTapologyEvent)

### Backend: Tapology tracker now handles No Contest and can add fights missed by the daily scraper
- **Symptom** (PFL Africa 1: Pretoria, live during this session):
  1. Sankara vs Cruz ended in a No Contest on Tapology (unintentional eye poke, R2 0:18). The fight never surfaced the NC in-app.
  2. Abdelrahman Mohamed vs David Samuel was on Tapology's card (position 5) and was missing entirely from the DB — the daily PFL scraper had skipped it. The live tracker saw it, logged "not found in DB", and moved on.
- **Root cause**:
  1. `tapologyLiveParser.ts` only wrote results when `dbFight.fightStatus !== 'COMPLETED'`. If the lifecycle job had already marked Sankara vs Cruz COMPLETED (with no method), the tracker never backfilled the NC result. It also had no NC-aware winner handling — `result.winner` was undefined for NC fights, so `updateData.winner` was never set, leaving the fight with no clear "no contest" state.
  2. `tapologyLiveParser.ts` logged missing fights to `fightsNotFound` and continued — there was no path to create fighters/fights on the fly.
  3. `normalizeMethod` in `tapologyLiveScraper.ts` checked KO/TKO branches before `"no contest"` and `"draw"`, which could misclassify text like "No Contest, Accidental Knockdown" if the phrasing changed. (In this case the existing order worked, but the ordering was fragile.)
- **Scraper changes** (`tapologyLiveScraper.ts`):
  - `normalizeMethod`: check `"no contest"` and `"draw"` before the KO/TKO branch so neither can be misclassified.
  - In the `<li>` loop, extract Tapology's card position from `#boutCompactNumber{boutId}` (opener = 1, main event = N) and expose it as `boutOrder` on each scraped fight. Used by the parser to assign `orderOnCard` for newly-created fights.
  - Broaden the result-keyword matcher in the uppercase-span scan to catch `No Contest`, standalone `NC`, `Disqualification`, `RTD`, and `Retirement`.
  - Added `boutOrder?: number` to the `TapologyFight` type.
- **Parser changes** (`tapologyLiveParser.ts`):
  - Added `parseFighterName` + `findOrCreateFighter` helpers (same upsert pattern as `ufcLiveParser.ts`'s live-creation path).
  - When a scraped fight has no DB match, the parser now creates both fighters and the fight row in place, using `boutOrder` for `orderOnCard` (falling back to `maxOrder + 1` if that slot is taken). The new fight is pushed into the in-memory `event.fights` array and immediately marked as "matched" so the cancellation-detection sweep won't cancel it the same run.
  - Completion handling is now **idempotent**: the block no longer gates on `dbFight.fightStatus !== 'COMPLETED'`. Instead it compares each result field (winner/method/round/time) against the DB value and writes any difference. This lets us backfill NC results onto a fight that the lifecycle job prematurely marked complete, and lets us land the result on a newly-created fight in the same pass.
  - No Contest and Draw results now set `winner` to the sentinel strings `'nc'` / `'draw'` (per the `Fight.winner` schema comment), instead of leaving `winner` null.
  - `ParseResult` gained a `fightsCreated` counter and the final log line reports it.
- **Mobile changes**:
  - `CompletedFightCard.tsx`: new centered blue NC badge (color `#3B82F6`, absolutely positioned at `bottom: -14` inside `fighterNamesContainer`, same vertical level as the green winner method text) when `fight.winner === 'nc'`. Hidden in spoiler-free mode via `!hideSpoilers`. Neither fighter gets the green winner border because all per-fighter checks compare against fighter IDs and `'nc'` never matches. If the tracker captured `round`, it appends ` R2` etc.
  - `CompletedFightDetailScreen.tsx`: replaced/augmented the "Outcome unavailable" branch so `winner === 'nc' && isOutcomeRevealed` renders `"No Contest · R2 0:18"` in blue matching the card badge. The existing "Outcome unavailable" grey text still renders when `winner` is truly null. Spoiler-free "Rate to reveal" flow still works because `'nc'` is truthy.

### Backend: Three daily Tapology scrapers were silently failing in CI (Golden Boy, Top Rank, Dirty Boxing)
- **Symptom**: `goldenboy-tapology-scraper.yml`, `toprank-tapology-scraper.yml`, and `dirty-boxing-tapology-scraper.yml` workflow runs all exited 1 every day. The scrape step reported success ("found 8 events / 155 fights") but the import step immediately crashed.
- **Golden Boy root cause**: directory name mismatch. `scrapeGoldenBoyTapology.js:257` wrote output to `scraped-data/golden-boy/` (hyphenated) but `goldenBoyDataParser.ts:687` read from `scraped-data/goldenboy/` (no hyphen). The import crashed with ENOENT on `latest-events.json`. Fix: changed scraper output dir to `goldenboy` to match the parser and the rest of the codebase (admin panel key, image dirs, etc. all use `goldenboy`).
- **Top Rank root cause**: identical bug. `scrapeTopRankTapology.js:260` wrote to `scraped-data/top-rank/` but `topRankDataParser.ts:785` read from `scraped-data/toprank/`. Fix: same rename.
- **Dirty Boxing root cause**: not a directory mismatch — both scraper and parser agree on `scraped-data/dirty-boxing/`. Actual bug was in `dirtyBoxingDataParser.ts:264`'s findFirst OR clause:
  ```
  OR: [
    { ufcUrl: eventData.eventUrl },
    { name: eventData.eventName },
    { name: { contains: 'Dirty Boxing' } }   // ← too greedy
  ]
  ```
  Prisma `findFirst` doesn't order results. When importing "DBX 5: Clark vs. Bittencourt", the third clause legitimately matched the unrelated "Dirty Boxing: Premier Event" row. The parser then tried to `update` that row and set its `ufcUrl` to DBX 5's Tapology URL — which the *actual* DBX 5 row already legitimately owned → `P2002 Unique constraint failed on (ufcUrl)`. The crash happened on the first event every run.
- **DB state**: I queried the Event table expecting corrupted rows from past failed runs, but the DB was **completely clean** — exactly 7 Dirty Boxing rows, one per scraped event, each with the correct unique `ufcUrl`. No data fix needed. The bug was purely in the parser's lookup logic, not in persisted state.
- **Dirty Boxing fix** (`dirtyBoxingDataParser.ts:263-310`):
  1. Removed the `name contains 'Dirty Boxing'` clause entirely.
  2. Scoped the name-match clause to the same promotion: `{ promotion: 'Dirty Boxing', name: eventData.eventName }`.
  3. Added a `P2002` fallback on the `update` call: if the unique `ufcUrl` conflicts (legacy duplicate row owns it), retry the update with `ufcUrl` omitted so the rest of the fields still sync. Same pattern as `ufcDataParser.ts:485-517`.
- **Files**:
  - `packages/backend/src/services/scrapeGoldenBoyTapology.js` (output dir rename)
  - `packages/backend/src/services/scrapeTopRankTapology.js` (output dir rename)
  - `packages/backend/src/services/dirtyBoxingDataParser.ts` (findFirst scope + P2002 fallback)

### Mobile: Verified comment-passthrough from fight-card modals to detail screens
- **Question**: When a user types a comment in a fight-card modal and taps "See Comments" (which saves the comment and navigates to the detail screen), does the newly-entered comment show up on first paint, or does the user have to nav away and back? Checked for all three card flows.
- **Flow architectures traced**:
  - **Upcoming**: `UpcomingFightCard` → `UpcomingFightModal` → `POST /fights/:id/pre-fight-comment` → `onSuccess` invalidates `['preFightComments', fightId]` → `router.push('/fight/:id')`. `UpcomingFightDetailScreen` renders the list straight from `useQuery(['preFightComments', fight.id])`, so the invalidation + refetch is what makes it work.
  - **Completed**: `CompletedFightCard` → `CompletedFightModal` → `PUT /fights/:id/user-data` (saves as a **review**, not a pre-fight comment) → `onSuccess` invalidates `['fight', fightId]` → `router.push('/fight/:id')`. The route `app/fight/[id].tsx` fetches via `useQuery(['fight', id, isAuthenticated], { refetchOnMount: 'always' })`, so every mount triggers a fresh fetch regardless of cache state. `CompletedFightDetailScreen` renders the user's own review via `<CommentCard comment={{ content: fight.userReview.content, ... }}>` — reading the **prop** directly, not a local useState — so when the refetch lands the display re-renders with the new content.
  - **Live**: `LiveFightCard` opens the same `CompletedFightModal`, so same save path and same detail screen. Architecturally identical to Completed.
- **Initial suspicion was wrong**: I first guessed `CompletedFightModal`'s `onSuccess` was missing a `['preFightComments', fightId]` invalidation and briefly applied a one-line "fix". Reverted it after reading `CompletedFightDetailScreen.tsx:1896-1921` — the user's own comment is rendered from `fight.userReview.content` (prop), NOT from the `preFightComments` query. That query is a separate "pre-fight hype" comments tab, unrelated to post-fight reviews. Adding a `preFightComments` invalidation on review save would have been noise. No code change shipped.
- **Manual test plan executed** (all three flows, unique markers `U-001`, `C1-002`, `C2-003`, `L-004`):
  - **U (Upcoming)**: Typed hype comment, tapped See Comments → appeared on first paint of `UpcomingFightDetailScreen`. ✅
  - **C1 (Completed, cold — first-ever review on this fight)**: Selected rating + typed comment, tapped See Comments → appeared on first paint of `CompletedFightDetailScreen` as the top "My Review" card. ✅
  - **C2 (Completed, warm — editing an existing review)**: Modified existing comment, tapped See Comments → updated content visible on first paint. ✅
  - **L (Live)**: Same path as Completed (via shared modal), tested on a LIVE event fight card → appeared on first paint. ✅
- **Why it works without any invalidation of local state**: The critical detail is that `CompletedFightDetailScreen` reads the user's own review content from the `fight` PROP (via `<CommentCard content={fight.userReview.content}>`) rather than from a local `useState` that would otherwise shadow prop updates. There IS a local `comment` state at `CompletedFightDetailScreen.tsx:344` (`useState(() => fight.userReview?.content || '')`), but it's only used as the edit buffer for the TextInput when `isEditingComment === true`. On first paint after navigation the edit form is hidden, so the stale useState initializer doesn't matter. Combined with `refetchOnMount: 'always'` on the route's fight query, the prop reliably contains fresh data by the time the user sees the detail screen.
- **Latent issue noted but not fixed**: If a user taps the **Edit** button on their own review immediately after landing from the modal, the TextInput opens prefilled with whatever `fight.userReview?.content` was at the FIRST render of the detail screen — not the latest value. If there was any race where the cached fight prop hadn't updated yet on first render, the edit buffer could be stale. Not observed in today's testing (the `refetchOnMount: 'always'` + awaited mutation in the modal covered it), but structurally the `comment` useState has no sync effect analogous to the rating-sync at `CompletedFightDetailScreen.tsx:391-428`. Leave as-is until it actually reproduces.
- **Result**: All three flows tested and working. Marking as verified. No code changes shipped from this investigation.
- **Files touched**: none (one speculative edit to `CompletedFightModal.tsx` was made and then reverted — file is clean).

## Notes
- **Pattern**: "all fights CANCELLED" should never be treated as a legitimate event-completion signal. Real events always have at least one fight that reaches COMPLETED. The `allTerminal && hasAnyCompleted` check is the correct invariant for auto-completion.
- **Pattern**: any "detect X by absence from scrape" logic needs a sanity check on the scrape itself first. The DBX 6 incident is structurally identical to yesterday's RAF scraper bug (detecting event completion by absence of a ticket button) — absence-based signals are brittle and need a validity guard before being trusted.
- **Scraper self-check to consider for all live trackers**: before any destructive DB update, require the current scrape to contain at least a threshold number of matching fights vs. the DB. If the scrape can't see what's already in the DB, don't trust it to tell us what changed.
