### Backend: Tapology live tracker wrongly auto-completed an upcoming event
- **Symptom**: DBX 6: Garcia vs. Perez (scheduled for tonight, Apr 10) showed as COMPLETED in the mobile app with zero fights visible. The event hadn't even started.
- **Investigation**:
  - Event row: `eventStatus: COMPLETED`, `completionMethod: tapology-tracker-auto`, `updatedAt: 2026-04-10T18:03:32Z` — marked complete at 2:03 PM EDT, hours before the 10 PM EDT main card.
  - All 12 fights: `fightStatus: CANCELLED`, updated in a burst between `18:03:31` and `18:03:32`, none had a `winner`, none had `trackerWinner`, all had `cardType: null`. The mobile app was hiding them because they were CANCELLED.
  - Event had `scraperType: tapology`, `ufcUrl: https://www.tapology.com/fightcenter/events/140498-dbx-6-garcia-vs-perez` (correct URL).
- **Root cause**: Two cascading bugs in the Tapology live tracker:
  1. `tapologyLiveParser.ts:316-318` set `hasStarted = true` if the scraped page reported `status === 'live' | 'complete'` OR any scraped fight was marked complete. When `hasStarted` was true, the "missing from scraped data" sweep at line 334-338 would CANCEL every DB fight that wasn't in the scraped list. With a broken/empty scrape (0 fights returned, or 0 matches because of name mismatches), **every single fight on the card got cancelled in one pass**. The exact reason the tapology scrape came back useless for DBX 6 isn't fully known — either Tapology's page was flagged 'complete' prematurely, or the fighter-name matcher failed for all 12 fights — but either way the parser had no safety net.
  2. `runTapologyLiveTracker.ts:252-278`'s `autoCompleteTapologyEvent` considered the event complete if **every** fight was in a terminal state (COMPLETED **or** CANCELLED). So once the parser nuked all fights to CANCELLED, the runner immediately marked the event COMPLETED with `completionMethod: tapology-tracker-auto`.
- **Data fix** (manual Prisma updates):
  - Event: `eventStatus` → UPCOMING, `completionMethod` → null
  - All 12 fights: `fightStatus: CANCELLED` → UPCOMING (via `updateMany`)
  - Corrected start times (the DBX scraper had populated wrong times — 4 PM EDT instead of 10 PM): `date` → `2026-04-10T04:00:00Z` (matches DBX 5 midnight-ET convention), `prelimStartTime` → `2026-04-11T00:00:00Z` (8 PM EDT), `mainStartTime` → `2026-04-11T02:00:00Z` (10 PM EDT).
- **Code fix — two guards, belt-and-suspenders**:
  1. `tapologyLiveParser.ts`: compute `scrapeLooksValid = scrapedData.fights.length > 0 && result.fightsMatched > 0`. The missing-from-page cancellation sweep AND the UPCOMING→LIVE flip now both require `scrapeLooksValid`. An empty or unmatched scrape logs a skip message and does nothing to the DB.
  2. `runTapologyLiveTracker.ts`: `autoCompleteTapologyEvent` now additionally requires at least one fight to actually be `COMPLETED` (not just CANCELLED). `allTerminal && !hasAnyCompleted` is treated as a broken-scrape symptom and logs a skip instead of marking the event complete.
- Files: `packages/backend/src/services/tapologyLiveParser.ts` (scrapeLooksValid guard around cancellation sweep and LIVE flip), `packages/backend/src/scripts/runTapologyLiveTracker.ts` (hasAnyCompleted guard in autoCompleteTapologyEvent)

### Backend: Tapology tracker now handles No Contest and can add fights missed by the daily scraper
- **Symptom** (PFL Africa 1: Pretoria, live during this session):
  1. Sankara vs Cruz ended in a No Contest on Tapology (unintentional eye poke, R2 0:18). The fight never surfaced the NC in-app.
  2. Abdelrahman Mohamed vs David Samuel was on Tapology's card (position 5) and was missing entirely from the DB — the daily PFL scraper had skipped it. The live tracker saw it, logged "not found in DB", and moved on.
- **Root cause**:
  1. `tapologyLiveParser.ts` only wrote results when `dbFight.fightStatus !== 'COMPLETED'`. If the lifecycle job had already marked Sankara vs Cruz COMPLETED (with no method), the tracker never backfilled the NC result. It also had no NC-aware winner handling — `result.winner` was undefined for NC fights, so `updateData.winner` was never set, leaving the fight with no clear "no contest" state.
  2. `tapologyLiveParser.ts` logged missing fights to `fightsNotFound` and continued — there was no path to create fighters/fights on the fly.
  3. `normalizeMethod` in `tapologyLiveScraper.ts` checked KO/TKO branches before `"no contest"` and `"draw"`, which could misclassify text like "No Contest, Accidental Knockdown" if the phrasing changed. (In this case the existing order worked, but the ordering was fragile.)
- **Scraper changes** (`tapologyLiveScraper.ts`):
  - `normalizeMethod`: check `"no contest"` and `"draw"` before the KO/TKO branch so neither can be misclassified.
  - In the `<li>` loop, extract Tapology's card position from `#boutCompactNumber{boutId}` (opener = 1, main event = N) and expose it as `boutOrder` on each scraped fight. Used by the parser to assign `orderOnCard` for newly-created fights.
  - Broaden the result-keyword matcher in the uppercase-span scan to catch `No Contest`, standalone `NC`, `Disqualification`, `RTD`, and `Retirement`.
  - Added `boutOrder?: number` to the `TapologyFight` type.
- **Parser changes** (`tapologyLiveParser.ts`):
  - Added `parseFighterName` + `findOrCreateFighter` helpers (same upsert pattern as `ufcLiveParser.ts`'s live-creation path).
  - When a scraped fight has no DB match, the parser now creates both fighters and the fight row in place, using `boutOrder` for `orderOnCard` (falling back to `maxOrder + 1` if that slot is taken). The new fight is pushed into the in-memory `event.fights` array and immediately marked as "matched" so the cancellation-detection sweep won't cancel it the same run.
  - Completion handling is now **idempotent**: the block no longer gates on `dbFight.fightStatus !== 'COMPLETED'`. Instead it compares each result field (winner/method/round/time) against the DB value and writes any difference. This lets us backfill NC results onto a fight that the lifecycle job prematurely marked complete, and lets us land the result on a newly-created fight in the same pass.
  - No Contest and Draw results now set `winner` to the sentinel strings `'nc'` / `'draw'` (per the `Fight.winner` schema comment), instead of leaving `winner` null.
  - `ParseResult` gained a `fightsCreated` counter and the final log line reports it.

### Backend: Three daily Tapology scrapers were silently failing in CI (Golden Boy, Top Rank, Dirty Boxing)
- **Symptom**: `goldenboy-tapology-scraper.yml`, `toprank-tapology-scraper.yml`, and `dirty-boxing-tapology-scraper.yml` workflow runs all exited 1 every day. The scrape step reported success ("found 8 events / 155 fights") but the import step immediately crashed.
- **Golden Boy root cause**: directory name mismatch. `scrapeGoldenBoyTapology.js:257` wrote output to `scraped-data/golden-boy/` (hyphenated) but `goldenBoyDataParser.ts:687` read from `scraped-data/goldenboy/` (no hyphen). The import crashed with ENOENT on `latest-events.json`. Fix: changed scraper output dir to `goldenboy` to match the parser and the rest of the codebase (admin panel key, image dirs, etc. all use `goldenboy`).
- **Top Rank root cause**: identical bug. `scrapeTopRankTapology.js:260` wrote to `scraped-data/top-rank/` but `topRankDataParser.ts:785` read from `scraped-data/toprank/`. Fix: same rename.
- **Dirty Boxing root cause**: not a directory mismatch — both scraper and parser agree on `scraped-data/dirty-boxing/`. Actual bug was in `dirtyBoxingDataParser.ts:264`'s findFirst OR clause:
  ```
  OR: [
    { ufcUrl: eventData.eventUrl },
    { name: eventData.eventName },
    { name: { contains: 'Dirty Boxing' } }   // ← too greedy
  ]
  ```
  Prisma `findFirst` doesn't order results. When importing "DBX 5: Clark vs. Bittencourt", the third clause legitimately matched the unrelated "Dirty Boxing: Premier Event" row. The parser then tried to `update` that row and set its `ufcUrl` to DBX 5's Tapology URL — which the *actual* DBX 5 row already legitimately owned → `P2002 Unique constraint failed on (ufcUrl)`. The crash happened on the first event every run.
- **DB state**: I queried the Event table expecting corrupted rows from past failed runs, but the DB was **completely clean** — exactly 7 Dirty Boxing rows, one per scraped event, each with the correct unique `ufcUrl`. No data fix needed. The bug was purely in the parser's lookup logic, not in persisted state.
- **Dirty Boxing fix** (`dirtyBoxingDataParser.ts:263-310`):
  1. Removed the `name contains 'Dirty Boxing'` clause entirely.
  2. Scoped the name-match clause to the same promotion: `{ promotion: 'Dirty Boxing', name: eventData.eventName }`.
  3. Added a `P2002` fallback on the `update` call: if the unique `ufcUrl` conflicts (legacy duplicate row owns it), retry the update with `ufcUrl` omitted so the rest of the fields still sync. Same pattern as `ufcDataParser.ts:485-517`.
- **Files**:
  - `packages/backend/src/services/scrapeGoldenBoyTapology.js` (output dir rename)
  - `packages/backend/src/services/scrapeTopRankTapology.js` (output dir rename)
  - `packages/backend/src/services/dirtyBoxingDataParser.ts` (findFirst scope + P2002 fallback)

### Mobile: Verified comment-passthrough from fight-card modals to detail screens
- **Question**: When a user types a comment in a fight-card modal and taps "See Comments" (which saves the comment and navigates to the detail screen), does the newly-entered comment show up on first paint, or does the user have to nav away and back? Checked for all three card flows.
- **Flow architectures traced**:
  - **Upcoming**: `UpcomingFightCard` → `UpcomingFightModal` → `POST /fights/:id/pre-fight-comment` → `onSuccess` invalidates `['preFightComments', fightId]` → `router.push('/fight/:id')`. `UpcomingFightDetailScreen` renders the list straight from `useQuery(['preFightComments', fight.id])`, so the invalidation + refetch is what makes it work.
  - **Completed**: `CompletedFightCard` → `CompletedFightModal` → `PUT /fights/:id/user-data` (saves as a **review**, not a pre-fight comment) → `onSuccess` invalidates `['fight', fightId]` → `router.push('/fight/:id')`. The route `app/fight/[id].tsx` fetches via `useQuery(['fight', id, isAuthenticated], { refetchOnMount: 'always' })`, so every mount triggers a fresh fetch regardless of cache state. `CompletedFightDetailScreen` renders the user's own review via `<CommentCard comment={{ content: fight.userReview.content, ... }}>` — reading the **prop** directly, not a local useState — so when the refetch lands the display re-renders with the new content.
  - **Live**: `LiveFightCard` opens the same `CompletedFightModal`, so same save path and same detail screen. Architecturally identical to Completed.
- **Initial suspicion was wrong**: I first guessed `CompletedFightModal`'s `onSuccess` was missing a `['preFightComments', fightId]` invalidation and briefly applied a one-line "fix". Reverted it after reading `CompletedFightDetailScreen.tsx:1896-1921` — the user's own comment is rendered from `fight.userReview.content` (prop), NOT from the `preFightComments` query. That query is a separate "pre-fight hype" comments tab, unrelated to post-fight reviews. Adding a `preFightComments` invalidation on review save would have been noise. No code change shipped.
- **Manual test plan executed** (all three flows, unique markers `U-001`, `C1-002`, `C2-003`, `L-004`):
  - **U (Upcoming)**: Typed hype comment, tapped See Comments → appeared on first paint of `UpcomingFightDetailScreen`. ✅
  - **C1 (Completed, cold — first-ever review on this fight)**: Selected rating + typed comment, tapped See Comments → appeared on first paint of `CompletedFightDetailScreen` as the top "My Review" card. ✅
  - **C2 (Completed, warm — editing an existing review)**: Modified existing comment, tapped See Comments → updated content visible on first paint. ✅
  - **L (Live)**: Same path as Completed (via shared modal), tested on a LIVE event fight card → appeared on first paint. ✅
- **Why it works without any invalidation of local state**: The critical detail is that `CompletedFightDetailScreen` reads the user's own review content from the `fight` PROP (via `<CommentCard content={fight.userReview.content}>`) rather than from a local `useState` that would otherwise shadow prop updates. There IS a local `comment` state at `CompletedFightDetailScreen.tsx:344` (`useState(() => fight.userReview?.content || '')`), but it's only used as the edit buffer for the TextInput when `isEditingComment === true`. On first paint after navigation the edit form is hidden, so the stale useState initializer doesn't matter. Combined with `refetchOnMount: 'always'` on the route's fight query, the prop reliably contains fresh data by the time the user sees the detail screen.
- **Latent issue noted but not fixed**: If a user taps the **Edit** button on their own review immediately after landing from the modal, the TextInput opens prefilled with whatever `fight.userReview?.content` was at the FIRST render of the detail screen — not the latest value. If there was any race where the cached fight prop hadn't updated yet on first render, the edit buffer could be stale. Not observed in today's testing (the `refetchOnMount: 'always'` + awaited mutation in the modal covered it), but structurally the `comment` useState has no sync effect analogous to the rating-sync at `CompletedFightDetailScreen.tsx:391-428`. Leave as-is until it actually reproduces.
- **Result**: All three flows tested and working. Marking as verified. No code changes shipped from this investigation.
- **Files touched**: none (one speculative edit to `CompletedFightModal.tsx` was made and then reverted — file is clean).

### Backend/CI: ONE FC live tracker workflow silently broken for 2 weeks — 8 of 14 fights stuck on ONE Friday Fights 150
- **Symptom**: User reported ONE Friday Fights 150 (April 10) was missing "a whole bunch of fights" in the app and blamed the daily scraper. The event showed as COMPLETED with only 6 of its 14 fights marked COMPLETED/with winners — the other 8 were still UPCOMING, which in a completed-event context makes them effectively invisible or at best confusing.
- **First-level diagnosis — the daily scraper is fine**: Queried `/api/fights?eventId=f8cc1129-4025-4421-a86d-5c8a32edd420&limit=50` and got 14 fights back. Every matchup on the card is in the DB with correct fighters, weightclass, cardType, and orderOnCard. `scrapeAllOneFCData.js` + `oneFCDataParser.importOneFCData()` did their job on the scheduled daily run. The user's framing ("daily scraper missed fights") was wrong — this was a LIVE tracker issue, not a daily scraper issue.
- **Second-level diagnosis — the live tracker has not run successfully in 2+ weeks**: Queried GH API `/repos/mikeprimak/fighting-tomatoes/actions/workflows/onefc-live-tracker.yml/runs`. **Zero runs on April 10**. Latest run was **2026-03-27 16:37Z**, and every run on that day was `failure`. Downloaded the logs from the last failed run (`curl .../actions/runs/{id}/logs -o live_tracker_logs.zip`) and found the real failure point:
  ```
  ##[error]src/services/oneFCLiveScraper.ts(238,25): error TS2584: Cannot find name 'document'. Do you need to change your target library? Try changing the 'lib' compiler option to include 'dom'.
  ##[error]src/utils/fighterMatcher.ts(214,37): error TS2339: Property 'fighterAlias' does not exist on type 'PrismaClient<...>'
  ##[error]src/routes/auth.fastify.ts(33,7): error TS2353: ...'description' does not exist in type 'FastifySchema'
  ##[error]src/services/newsScraperService.ts(5,24): error TS2307: Cannot find module '../utils/prisma' or its corresponding type declarations.
  Error: Cannot find module '.../dist/scripts/runOneFCLiveTracker.js'
  ##[error]Process completed with exit code 1.
  ```
- **Root cause**: `pnpm build` in `packages/backend` runs a combined `prisma generate && tsc --project tsconfig.production.json --noEmitOnError false || true && cp src/services/*.js dist/services/ 2>/dev/null || true`. Despite `--noEmitOnError false` and the trailing `|| true`, tsc was not emitting `dist/scripts/runOneFCLiveTracker.js` on a fresh checkout because of severe enough errors across multiple files:
  1. `src/routes/auth.fastify.ts`, `src/routes/news.ts`, `src/routes/index.ts`, `src/routes/feedback.ts` — hundreds of Fastify schema type errors after an SDK bump (`description` no longer on `FastifySchema`, `any` → `never` assignability).
  2. `src/services/newsScraperService.ts` — `Cannot find module '../utils/prisma'`, a hard module-resolution failure.
  3. `src/utils/fighterMatcher.ts` — references `prisma.fighterAlias` (model removed from Prisma schema, code never cleaned up) and a `WeightClass` string/enum mismatch.
  4. `src/services/oneFCLiveScraper.ts` — `document`, `HTMLAnchorElement`, `HTMLImageElement` not found because `tsconfig.json`'s `lib: ["ES2020"]` doesn't include `DOM`. These are inside `page.evaluate()` callbacks that actually run in the browser context, but TS still type-checks them.
  Hard module-resolution errors (`Cannot find module '../utils/prisma'`) cause tsc to bail out of the whole emit phase, not just the file with the error. So the ONE FC tracker script never made it into `dist/`, and every `node dist/scripts/runOneFCLiveTracker.js` invocation exited with `MODULE_NOT_FOUND`. The daily scraper workflow survived the same build because all ONE FC daily scrapers are plain `.js` files copied via the trailing `cp src/services/*.js dist/services/` step, so they don't depend on tsc emitting anything.
- **Why 6 fights still got COMPLETED despite the broken workflow**: Couldn't fully confirm but most likely via the Hetzner VPS scraper service (`VPS_SCRAPER_URL` path in `eventLifecycle.ts`'s `triggerVPSLiveTracker`) running an older build from its local git clone, OR via manual admin-panel updates. The GH Actions workflow definitely did not run today.
- **Decision — minimal targeted fix, don't try to fix the whole build**: The full `pnpm build` regression has a 2-week backlog of type errors across files unrelated to live tracking. Fixing them all is a separate project. Instead, give the live tracker its own dedicated tsconfig that compiles ONLY the files it needs and ignores everything else.
- **Fix 1 — `packages/backend/src/utils/fighterMatcher.ts`**: Added `// @ts-nocheck` at the top. The file has two layers:
  - Top half: pure string helpers (`stripDiacritics`, `levenshteinDistance`, `similarityScore`, `normalizeName`, `areNameVariations`) — imported by 23 scrapers/parsers and work fine at runtime.
  - Bottom half: DB-writing helpers that call `prisma.fighterAlias.findFirst/create`. This Prisma model was removed from `schema.prisma` at some point but the code was never deleted. It's effectively dead — any code path that hits those calls would throw a Prisma runtime error.
  Rather than delete the dead helpers and churn git blame on the live ones, `@ts-nocheck` lets tsc emit the whole file and the live helpers keep working. Documented the reasoning in a comment at the top of the file.
- **Fix 2 — `packages/backend/tsconfig.tracker.json` (new file)**: Minimal tsconfig extending `tsconfig.json` with:
  - `lib: ["ES2020", "DOM"]` — fixes the `document`/`HTMLAnchorElement` errors inside `page.evaluate()` blocks.
  - `skipLibCheck: true`, `noEmitOnError: false`, all `strict*` flags off — belt-and-suspenders.
  - `incremental: false` — avoid any weird interaction with `.tsbuildinfo` from the main build.
  - Explicit `include` list of exactly 6 files:
    ```
    "src/scripts/runOneFCLiveTracker.ts",
    "src/services/oneFCLiveScraper.ts",
    "src/services/oneFCLiveParser.ts",
    "src/services/notificationService.ts",
    "src/config/liveTrackerConfig.ts",
    "src/utils/fighterMatcher.ts"
    ```
    This is the full transitive dependency graph of the tracker script. `notificationService.ts` is pulled in for the `await import('./notificationService')` dynamic imports in `oneFCLiveParser.ts`. `fighterMatcher.ts` is pulled in for `stripDiacritics`. `liveTrackerConfig.ts` is pulled in for `getEventTrackerType` / `buildTrackerUpdateData`.
  - `exclude`: node_modules, dist, test files.

### Backend: Three compounding ONE FC scraper bugs surfaced by the ONE 150 follow-up
After the initial live-tracker-workflow fix landed 13/14 ONE 150 fights correctly, the user reported two follow-ups: Rittidet Lukjaoporongtom's name was truncated in-app, and fight #2 Kompet vs Attachai Kelasport was still stuck — plus the user noticed Kompet appeared in **two places** on the ONE FC event page (main list + a "current fight" hero section above it). Investigation revealed this single hero-duplicate observation was the root cause of three seemingly unrelated bugs, plus a fourth bug in the sticker detection that had been silently wrong for every ONE FC result ever recorded.

- **Bug 1 — hero duplicate corrupts the scrape**: ONE FC's event pages client-side-render a "Next Event" hero block above the main fight card. On finished events this hero is labeled `.event-live-status.is-finished` and contains a `.status-matchup` with a **fully-populated duplicate `.event-matchup`** of the headline fight (or of whichever fight the site considers featured). The duplicate has the same `.stats` table and the same `a.face.face1` / `a.face.face2` athlete anchors as the real matchup in the main list — so it can't be rejected by content — but it lives inside a `.box-post-event` / `.event-live-status` / `.status-matchup` ancestor chain, while real matchups live inside `.col-12.col-lg-9...` in the main content column. The duplicate matters because `querySelectorAll('.event-matchup')` iterates in DOM order, hits the hero copy first, and both scrapers have a `seenFights` dedup keyed on fighter name signature — so the hero copy wins and the main-list copy is silently skipped. On finished events the hero version displays empty stickers (no `is-win` on either face), so whatever fight is featured there is permanently recorded as `isComplete: false`. That's the exact mechanism behind the Kompet vs Attachai Kelasport "stuck UPCOMING" symptom — Kompet is the main event of ONE 150, so ONE FC's hero featured it, and the hero's zero stickers overwrote the main list's correct result. Diagnosed by writing a one-off puppeteer diagnostic (`debug-kompet.js`) that dumped each matchup's ancestor chain; idx 0 had `DIV.status-matchup → DIV.is-finished → DIV.event-live-status.is-finished → DIV.info → DIV.box-post-event.is-wide`, idx 1+ had `DIV.col-12.col-lg-9.col-xl-8...`. Clear discriminator.
- **Bug 2 — daily scraper off-by-one on orderOnCard**: Same root cause. The daily scraper's `.event-matchup` loop also hit the hero duplicate first, assigned it `order = globalOrder++` (so globalOrder = 1 was consumed by the ghost), and the 14 real fights got orders 2..15. Import-time it created real fighter rows for the ghost (since face anchors looked valid), so it wasn't skipped — it was upserted at `orderOnCard=1` with the same fighter pair as the real matchup. The unique constraint on `(eventId, fighter1Id, fighter2Id)` meant the ghost and the real matchup collided in the upsert, and only one row survived — with whatever `orderOnCard` was written last. The net effect in the DB: 14 fights numbered 2..15, Kompet at 2.
- **Bug 3 — every ONE FC sticker-detection winner was fighter1**: While digging into Kompet's detection failure, I dumped the DB winner for all 13 "working" ONE 150 fights and found **every single one had `winner = fighter1.id`**. Statistically impossible for 13 independent fights. Re-read `oneFCLiveScraper.ts` and found the method-1 bug:
  ```ts
  const fighter1Container = face1.closest('[class*="matchup"]'); // = .event-matchup
  const sticker1 = fighter1Container.querySelector('.sticker.is-win');
  if (sticker1) fighterAWon = true;   // finds ANY is-win sticker in the whole matchup
  // ... same for face2 → both fighterAWon and fighterBWon set to true for every completed fight
  ```
  `.closest('[class*="matchup"]')` resolves to `.event-matchup` itself (not a per-face container), and then the `querySelector('.sticker.is-win')` scan is scoped to the entire matchup. Both face1 and face2 iterations find the same winner sticker somewhere inside `.event-matchup`, so **both `fighterAWon` and `fighterBWon` get set to `true` for every completed fight**. Downstream:
  ```ts
  winner: fight.fighterAWon ? fight.fighterAName : fight.fighterBName,
  winnerSide: fight.fighterAWon ? 'A' : 'B',
  ```
  With `fighterAWon === true`, side A wins unconditionally. Methods 2, 3, 4 are gated on `!fighterAWon && !fighterBWon`, so they never run. The snapshot JSONs in `live-event-data/onefc/` confirmed it — for every "working" fight on ONE 150 the snapshot had `fighterA.isWinner: true, fighterB.isWinner: true`. Every ONE FC event in the DB almost certainly has the same problem (wrong winner on every fight where B actually won).
- **Bug 4 — Rittidet fighter row has two copies**: Stats dump showed fight #11's fighter2 as `firstName=''  lastName='Rittidet'`. `/athletes/rittidet/` is a single-word URL slug, so `parseOneFCFighterName` returns `{ firstName: '', lastName: 'Rittidet' }`. But the ONE FC page's stats table link has the full text `"Rittidet Lukjaoporongtom"`. Worse, querying the DB I found **two** Rittidet rows: the broken `('' , 'Rittidet')` (66,591 ratings via shared fighter statistics — shouldn't be that many, probably a bad query, but at least one real fight reference) and a legitimate `('Rittidet', 'Lukjaoporongtom')` with its own fight reference from a different event. Both were in use. So neither could be renamed in-place without hitting the `firstName_lastName` unique constraint — the fix had to be a merge.

## Notes
- **Pattern**: "all fights CANCELLED" should never be treated as a legitimate event-completion signal. Real events always have at least one fight that reaches COMPLETED. The `allTerminal && hasAnyCompleted` check is the correct invariant for auto-completion.
- **Pattern**: any "detect X by absence from scrape" logic needs a sanity check on the scrape itself first. The DBX 6 incident is structurally identical to yesterday's RAF scraper bug (detecting event completion by absence of a ticket button) — absence-based signals are brittle and need a validity guard before being trusted.
- **Scraper self-check to consider for all live trackers**: before any destructive DB update, require the current scrape to contain at least a threshold number of matching fights vs. the DB. If the scrape can't see what's already in the DB, don't trust it to tell us what changed.
- **Pattern**: CI workflows that chain `build` → `run script` need to fail loudly when the build emits nothing. `pnpm build || true` + `node dist/scripts/foo.js` silently turns a build failure into a `MODULE_NOT_FOUND` that's only visible by digging into the runner logs. Either drop `|| true` so the build step fails the whole workflow visibly, or assert the expected dist file exists immediately after the build step as a separate `test -f dist/scripts/foo.js` guard.
- **Pattern**: when one script in a monorepo needs a different tsc config (e.g., wider `lib`, narrower `include`) from the main build, give it its own `tsconfig.*.json` rather than trying to make one config fit every entry point. The focused tsconfig is both faster (6 files instead of ~200) and less fragile (broken files in unrelated directories can't block compilation). The `tsconfig.tracker.json` added today is the model; same pattern should apply to other per-script entry points like admin tools and one-off migration scripts.
- **Pattern**: check the winner-side distribution as a sanity check on any scraper. If a promotion's DB shows 13 of 13 winners as fighter1, that's not bias — that's a scraper bug defaulting to side A. Real events produce a ~50/50 A/B distribution. Same check works for inverse cases (all side B, or all draws, or all KO). Cheap to run, catches structural detection bugs that manual QA won't.
- **Pattern**: client-side-rendered hero/featured sections on event pages are a silent data-corruption source. Any scraper that uses a broad class selector (`.event-matchup`, `.fight-card`, etc.) will pick up JS-injected duplicates that have the same content but different wrapper classes than the main list. Always anchor scrapers to a specific parent container (e.g. `.col-12.col-lg-9 .event-matchup`) or explicitly reject known hero ancestors (`.closest('.box-post-event, .event-live-status, .status-matchup') === null`). The ONE 150 off-by-one on `orderOnCard` and the Kompet "stuck upcoming" bug were both caused by the same hero duplicate — one bug surfaced in the daily scraper (ordering), the other in the live scraper (winner detection).
- **Pattern**: when a one-off repair script finds that renaming a row would violate a unique constraint, it means there are already TWO rows that should be the same entity — one with stale/broken data, one with correct data. The fix is a merge (reassign FK references from broken to correct, delete broken), not a rename. The `fix-rittidet.js` pattern from today is worth keeping as a template: count references on both sides, reassign via `updateMany`, then delete.
- **Pattern**: "the live tracker is broken" often masquerades as "the daily scraper missed data". When a user reports missing fights on a completed event, check the fight status distribution first (`fightStatus` column on `Fight`) before assuming the scheduling data is wrong. If fights exist with `fightStatus: UPCOMING` on an event with `eventStatus: COMPLETED`, the live tracker is the culprit, not the daily scraper. The daily scraper only writes UPCOMING; the live tracker is the only component that flips fights to COMPLETED with winners.
