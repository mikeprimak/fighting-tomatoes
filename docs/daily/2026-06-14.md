# 2026-06-14

## Live-tracking incident: lifecycle failsafe completed a UFC event mid-card

**Symptom (reported during UFC Freedom 250, live):** event + all 7 fights showed
COMPLETED while only the first 2 had actually finished. Mike corrected it in the
admin panel; the lifecycle re-clobbered the fix within ~5 min (event came back as
`completionMethod: lifecycle-auto`).

**Root cause — `eventLifecycle.ts` Step 3 (LIVE→COMPLETED):** for tracker-backed
events the soft duration estimate is `min(numFights*30min + 60min, 10h)`, anchored
on `getStartTime()` = the *earliest* section start time. UFC Freedom 250 had a
spurious `prelimStartTime` of 21:00Z but **zero prelim fights** (all 7 cardType
"Main Card", real start = `mainStartTime` 00:00Z). So the 4.5h estimate elapsed at
01:30Z — 1.5h into the main card, 2 of 7 fights done — and the failsafe force-
completed the event, overriding the live tracker. Once it fires, the tracker's
no-downgrade rule makes COMPLETED stick.

**Fix (shipped to `main` → Render, commit 77311011):** the soft estimate may no
longer force-complete a *tracker-backed* event that still has UPCOMING/LIVE fights.
The live tracker owns fight-by-fight completion; if the estimate elapses with fights
still pending, defer to the 10h hard cap instead of closing the broadcast. The soft
estimate still promptly flips the event to COMPLETED once all fights are done
(cleanup when the tracker dies after the last fight). Non-tracker events unchanged.

**Live mitigation:** restored event→LIVE + the 5 wrongly-completed fights→UPCOMING
(left the 2 genuine COMPLETED). Nulled the spurious `prelimStartTime` so even the
*old* code's estimate moved out to 04:30Z, covering the deploy window.

**Note:** this is the failsafe misfiring, not the UFC live tracker itself — the
tracker correctly marked the first 2 fights. The general lesson: Step 3 is a
cleanup net for abandoned events, and must never race a live, tracker-backed card.
