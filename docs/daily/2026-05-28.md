## Fighter records backfill (0-0-0 problem) + UI guard

Mike: "many of my fighters show 0-0 / 0-0-0 ... can we get accurate records
today? scrape ufcstats.com?" — yes. Two-part fix, both shipped.

### Scale of the problem (before)

- **9,718** total fighters; **8,241 (84.8%)** at the 0-0-0 default.
- **6,800** of those empties actually appear on ≥1 card (real UX impact).
- By sport: MMA 6,904, Boxing 1,190, Wrestling 71, Muay Thai 62, Kickboxing 14.
- Cause: `wins/losses/draws` are Int columns defaulting to 0; only the active
  UFC roster gets filled by the daily ufc.com scrape. Everyone else (historic
  UFC, opponents, other orgs) stays 0-0-0.

### Part 1 — data backfill from ufcstats.com

- **New gotcha:** ufcstats.com now gates every response behind a lightweight
  Hashcash-style JS proof-of-work ("Checking your browser…"). Page embeds a
  `nonce` + difficulty (2 leading hex zeros); client finds `n` where
  `sha256(nonce:n)` starts with `00`, POSTs `nonce`+`n` to `/__c`, gets a
  clearance cookie. Solved in Node with built-in `crypto` — **no browser**.
  (The old "ufcstats is not protected" note is now stale.)
- `packages/backend/src/services/scrapeUFCStatsFighters.ts` — pulls the 26
  letter pages of the fighter directory (`/statistics/fighters?char=<a>&page=all`),
  parses name + W/L/D. 4,495 fighters total on ufcstats.
- `packages/backend/src/scripts/backfillFighterRecords.ts` — dry-run by default,
  `--apply` to write. Matches our 0-0-0 MMA fighters by normalized full name,
  **conservatively**: skips anything ambiguous on either side, never overwrites
  a populated record. Writes `prisma/fighter-records-backfill.json` audit log.
- **Result: 2,676 MMA records filled → empty share 84.8% → 57.3%** (MMA empties
  6,904 → 4,228). Sample sanity: Court McGee 23-14-0, Wilson Reis 23-10-0,
  CM Punk 0-1-0 — all correct.

**Coverage reality:** ufcstats is UFC/Zuffa-only. The 4,216 unmatched MMA
fighters are genuinely not in ufcstats (regional/non-UFC). The **1,190 boxers**
are entirely out of scope — boxing needs a different source (BoxRec is heavily
bot-protected). Mike: "ufc only is ok" → boxing deferred.

## Web: search loading flash + Main Event section split (commit 8308994)

Two web-app bugs Mike reported. Both shipped (auto-deploy from `main`).

### 1. Search flashes "No results found" while loading

`packages/web/src/app/search/page.tsx` gates the query on `enabled: !authLoading`.
While auth resolves the query is *disabled*, and in React Query v5 a disabled
query reports `isLoading=false` (status pending + fetchStatus idle). The old
`if (isLoading)` guard was skipped, `data` was undefined, so it fell straight
through to the empty state. Fix: show the spinner when
`authLoading || isLoading || (!data && !error)`.

### 2. MVP single "Main Event" split off from the card

`scrapeMVPTapology.js:387` labels the headliner `cardType: 'Main Event'` and the
rest `'Main Card'`. The web event detail keyed sections by raw `cardType`, so the
headliner became its own section *above* "Main Card", with How to Watch wedged
between them (MVP broadcasts are keyed to `MAIN_CARD`). The mobile app never had
this — it canonicalizes to MAIN CARD / PRELIMS / EARLY PRELIMS and treats "Main
Event" as main card. Ported that `canonicalSection()` logic into
`EventDetailClient.tsx` `groupFightsBySection`. Now MVP collapses to one MAIN
CARD section (no stray header since it's single-section), How to Watch sits
above, headliner sorts to top via `orderOnCard`. Also fixes a latent bug where
`SECTION_ORDER.indexOf("Main Card")` failed on mixed-case keys.

`groupFightsBySection` is **duplicated** in `EventCard.tsx` (home/upcoming list)

## Web: reframe "Get the app" CTAs when the account already uses mobile

Mike: "if someone is logged in to web app, it should know if they have the mobile
app or not? and change the messages about 'download the app'." Reasonable — but
done via a **backend signal**, not client-side install detection (unreliable, and
browsers have killed the deep-link-probe tricks).

### Signal
`User.pushToken` (single nullable string, written only by the mobile app's
`POST /api/notifications/register-token`, cleared by the matching DELETE). It's a
**strong positive** ("has app + notifications on right now") but a **weak negative**

to hide/reframe the CTA when present; some app-users will still see it.

### Changes
- `auth.fastify.ts` GET `/profile`: select `pushToken`, return `hasApp: !!pushToken`,
  destructure the raw token out (never ship the credential), declare `hasApp` in the
  response schema (Fastify strips undeclared fields).
- `packages/web/src/lib/auth.tsx`: `hasApp?: boolean` on `User` + new `useHasApp()`
  hook (treats unknown as false → CTA shows by default). Reuse this for the planned
  sitewide download banner.
- `FollowedFightersStrip.tsx` (shared by **profile page** + **homescreen sidebar**)
  and `followed-fighters/page.tsx` (subtitle + the dismissible install card): when
  `hasApp`, reframe to "You'll be notified in the app…" and suppress the install card.

## Web: fight detail didn't show user's rating until reload (commit 6bfc9d3)

Mike (remote control): "Fight detail screen for past fights should show my user
rating, but it doesn't until I reload the page."

**Root cause:** `FightDetailClient.tsx`'s `['fight', fightId]` query supplied SSR
`initialData` but — unlike the sibling `EventDetailClient` — omitted
`initialDataUpdatedAt: 0` and `enabled: !authLoading`. With the global 5-min
`staleTime` (`queryClient.ts:12`), the initialData was treated as *fresh*, so the
authenticated client refetch never ran on mount. The SSR fetch in `page.tsx:39`
is **unauthenticated** (no token forwarded), so `initialFight` never carries
`userRating` — the rating only appeared once a hard reload forced a fresh
authenticated fetch.

**Fix:** added `initialDataUpdatedAt: 0` (treat SSR data as stale → refetch) +
`enabled: !authLoading` (wait for the token so the refetch is authenticated),
mirroring `EventDetailClient`'s fight query. Pulled `isLoading: authLoading` from
`useAuth()`.

Web `tsc --noEmit` clean. Commit `6bfc9d3`, pushed to `main` (auto-deploys via
Vercel). Not yet verified live.

## Repeated-logout fix (auth resilience to backend redeploys)

**Symptom:** Mike and a beta user were repeatedly logged out today.

**Root cause (confirmed, not an auth code change):**
1. `render.yaml` had auto-deploy on with **no path filter** — every push to `main`
   redeployed the backend, including web/docs-only commits. Today had ~40 pushes
   (10:00→22:22), i.e. ~40 backend restarts. Each restart on the starter Docker
   plan is a ~30-90s downtime window.
2. The refresh flow has **zero tolerance for downtime**: 15-min access tokens, and
   `/auth/refresh` rotated the refresh token **in place with no grace period**. A
   refresh landing in a restart window either failed outright (→ client wipes
   tokens → logout) or succeeded server-side but the response dropped mid-restart,
   stranding the client with a now-dead token (→ next refresh 401s → logout).
3. The mobile client treated **any** refresh failure (network/5xx) the same as a
   real token rejection → forced logout.

With ~40 restart windows the odds of someone's 15-min refresh hitting one —
repeatedly — were high, which is why it bit everyone. Normal days have a handful
of pushes so it almost never surfaces.

**Three fixes shipped:**
1. **Render path filter** (`render.yaml` `buildFilter.ignoredPaths`) — web/landing/
   mobile/docs/`*.md`-only commits no longer redeploy the backend. Kills ~90% of
   the spurious restarts.
2. **Grace window on token rotation** (`routes/auth.fastify.ts` `/refresh`) — new
   token goes in a fresh row, old token stays valid 60s, expired rows pruned. A
   dropped refresh response no longer strands the client.
3. **Client distinguishes transient vs. real failures** (`mobile/services/api.ts`)
   — `_doRefreshToken` now returns `'success' | 'invalid' | 'transient'`, retries
   transient failures (3x w/ backoff), and only `'invalid'` (a genuine 4xx
   rejection from the refresh endpoint) clears auth. 5xx/network → `503
   REFRESH_UNAVAILABLE`, session kept.

Backend prod tsc clean; mobile tsc clean.

**Deferred to a later session — biometric login (Face ID / Touch ID / fingerprint):**
The beta user suggested it to simplify re-logging-in. It's a good UX add
(`expo-local-authentication` gating the secure-store credentials so re-auth is one
tap) but it treats the *symptom*, not today's cause — a server-invalidated token
still forces a full login regardless of biometrics. The three fixes above are what
actually stop the logouts. Biometric login to be scoped/built in a separate window.
