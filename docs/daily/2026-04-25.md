## Changes Made

### Admin panel: mobile responsiveness + master-detail UX
File touched: `packages/backend/public/admin.html`

The admin panel was a desktop-only two-pane layout (350px events sidebar + fights main pane) with hover-only fight actions, side-by-side form rows, and a 500px modal — none of which worked on a phone. Mike uses the panel mostly on mobile, so the goal was a phone-first experience without rebuilding the page.

**Approach: pure CSS media queries + a tiny JS toggle for master-detail.**

Added two breakpoints (`max-width: 900px` for tablet/phone, `max-width: 480px` for small phone) that:
- Header wraps onto multiple rows; tab nav becomes a horizontal-scroll strip; promotion select goes full-width.
- The main 2-column grid collapses to 1 column. Form rows (`.form-row`, fighter selector) stack to single column.
- Modals become near-full-screen, top-aligned (so the keyboard doesn't push the dialog off-screen), and footer buttons stack full-width.
- Fight cards switch to vertical stacks. The hover-revealed `.fight-actions` are forced visible (`opacity: 1`) since hover doesn't exist on touch.
- Buttons get bigger touch targets (`min-height: 38px`, status chips 32px square).
- Form inputs use `font-size: 16px` to suppress iOS auto-zoom on focus.
- Operations grids (`.scraper-grid`, `.backup-status`) collapse to a single column; scraper/tracker cards stack their inner flex.
- Feedback cards stack their header and action buttons full-width.
- Toasts span left-to-right at the bottom of the screen instead of being a tiny right-corner notification.

**Mobile master-detail:**
On mobile the user can't see both the events list and the selected event's fights at once. Added a `body.event-selected` class:
- `selectEvent()` adds the class and scrolls to top.
- A new `showEventsList()` function removes the class.
- CSS: when the class is present, hide `.events-panel`; when absent, hide `.fights-panel`.
- Injected a `.mobile-back-btn` (← Events) at the top of the rendered fights pane; it's `display: none` on desktop and `inline-flex` on mobile.

Net effect on mobile: one screen at a time. Tap an event → see its fights with a clear back link. Desktop layout is untouched (the master-detail rules are inside the `@media (max-width: 900px)` block).

Web app (`packages/web`) was not touched in this pass — many `app/*/page.tsx` files don't use Tailwind responsive utilities and would need a separate review. Flagged in chat.

Last commits that did touch Google sign-in code both predate the regression window:
- `207c86c` (Apr 19) — backend: parse `GOOGLE_CLIENT_ID` env var as comma-separated audience list. Bug fix, diff is correct.
- `d40c60c` (Apr 14) — mobile: Sentry capture + try/catch around pre-signin `GoogleSignin.signOut()`.

Most likely real causes (not in code):
1. Render env var `GOOGLE_CLIENT_ID` cleared/malformed (audience falls back to `undefined` → all tokens 401).
2. Mobile mints tokens for two audiences (web client `…pehb7kkuclbv8g4acjba9eeeajprd8j7…` for Android, iOS client `…qpifgfjjg3ve22bnhofhk99purj0aidf…`); Render's `GOOGLE_CLIENT_ID` must include **both** comma-separated. If only one is set, the other platform 401s.
3. Google Cloud Console rotation / SHA-1 fingerprint mismatch.
4. Stale native build after a config change.

Diagnostics worth a glance: Render env vars, Render logs around the failure window, Sentry events tagged `feature: 'google-signin'` (added in `d40c60c`), and on-device `[GoogleAuth]` console lines from `useGoogleAuth.ts`.
