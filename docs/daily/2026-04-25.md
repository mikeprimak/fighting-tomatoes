## Changes Made

### Lifecycle respects manual admin overrides on UPCOMING events

**Bug**: BKFC Clearwater (Apr 25, 2026) was showing as LIVE while still ~24h before its scheduled start. The admin tried to set the status back to UPCOMING via the admin panel, but within 5 minutes the lifecycle flipped it back to LIVE. Repeating the manual fix had no effect.

**Root cause**: Step 1 of `eventLifecycle.ts` (UPCOMING → LIVE) was a pure time-based check: any UPCOMING event whose stored start time is `<= now` gets flipped to LIVE on every 5-min run. For BKFC Clearwater the stored `mainStartTime` (or one of the prelim fields) is in the past — likely because the BKFC daily scraper extracted a wrong time (e.g. a "doors open" countdown or a stale value). The admin endpoint `PUT /admin/events/:id/status` already records `completionMethod: 'manual'` on every manual change, but Step 1 ignored that flag.

**Fix**:
1. `services/eventLifecycle.ts` Step 1 now **skips events with `completionMethod === 'manual'` and `eventStatus === 'UPCOMING'`**. The admin's intent overrides the time-based heuristic.
2. Added diagnostic logging to the UPCOMING → LIVE flip so future bad-data cases are easier to identify (logs which start-time field triggered the flip and what the wall clock said).
3. `services/bkfcLiveParser.ts`: defense-in-depth — the parser also refuses to flip UPCOMING → LIVE on a manual override, in case the live tracker is running against an admin-paused event.

The Step 1 change is in Render's lifecycle service and auto-deploys. The bkfcLiveParser change runs on the Hetzner VPS and won't take effect until a VPS rebuild (`bash /opt/scraper-service/packages/backend/vps-update.sh`); not strictly required since Step 1.5 only dispatches the tracker for LIVE events anyway, but worth the small belt-and-suspenders.

**To unstick the affected event**: After this deploys, set BKFC Clearwater back to UPCOMING in the admin panel — it will stick. When the event actually starts, click "Live" manually (or fix the start time in the DB and clear `completionMethod`).

**Update mid-session**: Mike re-checked and the event was actually genuinely LIVE — the lifecycle's flip was correct. The earlier "tomorrow" framing was a timezone confusion (event is on Sat Apr 25 EDT, was already past `mainStartTime` by the time we looked). Open question for next time: the app *displays* the date as "Sat Apr 25" but `event.date` is likely stored at midnight UTC (= Fri 8pm EDT), so there may still be a date-display formatter bug worth checking — the storage isn't necessarily wrong, but the rendered date may be in UTC instead of the user's TZ. **The lifecycle fix above still stands and is still useful** as a safety valve for any future false-LIVE scenario.

## Investigations (no changes shipped)

### BKFC scraper start-time deep-dive — paused
Investigated `scrapeAllBKFCData.js` to find a scenario where `mainStartTime` could be stored *earlier than* `event.date`. Could not reproduce one purely from the parsing code:
- `parseBKFCDate(dateStr)` correctly returns the UTC date.
- `parseEventStartTime` always uses `America/New_York`, so for any "H:MM PM" string the resulting `mainStartTime` is at least `event.date + 04:00 UTC`.
- The Try-1 selector list does include broad `'p'` which scans ALL `<p>` elements — risky if the page has other "MonthName DD, YYYY H:MM PM TZ" text (e.g. weigh-in date, last event recap), but no concrete instance found.
- Multi-event daily scraper output isn't kept long enough to inspect retroactively.

If this comes up again, the diagnostic log added in commit 8a31519 will print `which=mainStartTime|prelimStartTime|...` and the actual stored time, so root-causing should be much easier next time.
