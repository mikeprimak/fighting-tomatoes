## Background
Adding a new org currently requires hand-editing ~15 different files (admin dropdowns x5, scraper config, live tracker config, workflow map, filter groups, hidden promotions, frontend list, mobile list, logos, GitHub workflow YAML…). Easy to forget a step, easy to typo a string and silently break a filter tab.

This is the first of a 4-PR plan to move every org definition into one registry:
- **PR 1 (this commit)**: backend registry + derived exports + verify script. Zero behavior change.
- **PR 2**: admin.html dropdowns served via API
- **PR 3**: web frontend `orgFilter.tsx` migrated
- **PR 4**: mobile app + per-parser hardcoded `promotion: '...'` strings

## Changes Made

### New file: `packages/backend/src/config/orgs.ts`
Single source of truth. 15 orgs defined as `OrgDefinition` records, each capturing scraperType, dbPromotion, daily-scraper config, live-tracker workflow, Tapology hub config, admin trigger keys, hidden flag, and display names. Exports derived constants/builders: `ALL_SCRAPER_TYPES`, `DERIVED_PRODUCTION_SCRAPERS`, `DERIVED_HIDDEN_PROMOTIONS`, `SHARED_DAILY_SCRAPER_ORG_KEYS`, `ADMIN_TRIGGER_ORG_ORDER`, `BOXING_AGGREGATE_PROMOTIONS`, `buildTapologyPromotionHubs()`, `buildOrgFilterGroups()`.

### Files refactored to derive from the registry
Every file below now imports from `orgs.ts`. Exported names and shapes are unchanged — every consumer keeps working without modification.

- `packages/backend/src/config/liveTrackerConfig.ts` — `ScraperType`, `PRODUCTION_SCRAPERS` derived
- `packages/backend/src/config/hiddenPromotions.ts` — `HIDDEN_PROMOTIONS` derived
- `packages/backend/src/services/dailyAllScrapers.ts` — `SCRAPER_CONFIG` and the `organizations` array derived. Local `IMPORT_FN_BY_NAME` map pairs registry's `importFnName` strings with the actual imported parser functions.
- `packages/backend/src/scraperService.ts` — `TAPOLOGY_PROMOTION_HUBS` derived via `buildTapologyPromotionHubs()`
- `packages/backend/src/routes/admin.ts` — `scraperMap` derived from each org's `adminTriggerKeys`. Zod enum on `scraperType` unchanged (preserves the prior quirk of omitting `'raf'` — flagged for follow-up).
- `packages/backend/src/routes/index.ts` — `ORG_FILTER_GROUPS` derived via `buildOrgFilterGroups()`

- `packages/backend/public/admin.html` 5 hardcoded org dropdowns/lists. PR 2.
- `packages/web/src/lib/orgFilter.tsx`, `packages/mobile/store/OrgFilterContext.tsx`, `packages/mobile/components/PromotionLogo.tsx`. PRs 3 & 4.

### New file: `packages/backend/scripts/verify-org-registry.ts` (+ npm scripts)
Snapshot-style assertion script. Pins every derived export to its pre-registry value via `assert.deepStrictEqual` against inline `EXPECTED_*` snapshots. Run via:

```
pnpm --filter @yourapp/backend verify:orgs       # registry assertions only
pnpm --filter @yourapp/backend verify:orgs:db    # also runs DB coverage check
```

The DB coverage check pulls all distinct `event.promotion` values and asserts each one is either a registered `dbPromotion` or in the `BOXING_AGGREGATE_PROMOTIONS` curated list. Run before deploying any registry change.

### Bug caught by the verify script while writing
Initial implementation derived `HIDDEN_PROMOTIONS` as `[dbPromotion.toUpperCase()]`, which produced `['MATCHROOM BOXING']`. Legacy value was `['MATCHROOM']` (used as a case-insensitive `contains` substring in `prisma.event.findMany`). Substring `'MATCHROOM BOXING'` is narrower than `'MATCHROOM'` — would have allowed events with `promotion: 'MATCHROOM'` (uppercase) to leak through despite Matchroom being hidden. Fixed by adding an optional `hiddenPromotionMatch` field on the org definition that overrides the default.

This is exactly the class of subtle drift the verify script is designed to catch.

## Risk Assessment for Production
Zero behavior change is the goal of PR 1. The registry derives byte-identical values for every legacy export, verified by `verify:orgs`. If snapshot assertions pass, prod behavior is unchanged.

The only file touching shared state is `dailyAllScrapers.ts` — `SCRAPER_CONFIG` is now built at module load time. Same keys, same values, same imports. Should be a no-op.
