## Root Cause
The 4/18 OAuth cleanup flipped mobile `webClientId` from the old project to the new `good-fights-app` project. Android Google Sign-In does a local handshake (Google Play Services confirms the calling app's package + SHA-1 are registered to the OAuth client) **before any network call**. The new project has an Android OAuth client (`FightCrewApp Android`, ID `499367908516-pi84...`) but it was registered with the **upload key** SHA-1 (`DD:D0:C1:CA:E4:F5:F4:F5:93:80:7A:9B:0B:E1:37:EF:A2:1F:F8:23`), not the app signing key. Play Store re-signs every install with the app signing key, so the SHA-1 on the device never matched what the OAuth client knew → `DEVELOPER_ERROR`.

The old project's Android OAuth client was already (correctly) registered with the app signing key SHA-1, which is why sign-in worked pre-4/18.

## Why we didn't fix it on the GCP side
Tried to update the new project's OAuth client to the app signing SHA-1. Got `Save failed — Requested entity already exists` because Google enforces (package_name, SHA-1) uniqueness **globally across all GCP projects** — the combo is already on the old project's client. To put it on the new project we'd have to delete it from the old project first, which would break sign-in for anyone still on a pre-versionCode-36 app version. Pivoting to the JS/OTA fix avoided that risk entirely.

## Why this didn't break iOS
iOS uses `iosClientId` for the actual sign-in handshake; `webClientId` is only used for offline-access server tokens. The iOS URL scheme in `app.json` (`com.googleusercontent.apps.499367908516-j03poule51s7sfvpvdufna0upqa3oseg`) matches the iOS client ID, which is what the iOS Google SDK hard-asserts on. Backend `GOOGLE_CLIENT_ID` env on Render already lists both audiences (old `1082468109842-pehb...` + new `499367908516-j03p...`).

## Unfinished / Next Time
- The cross-project OAuth split (Android on old, iOS on new) is intentional going forward. **Do NOT drop the old `1082468109842-pehb...` audience from Render's `GOOGLE_CLIENT_ID`** — it's now load-bearing for all Android sign-ins, not just legacy users. Memory + docs updated to reflect this.
- Eventual cleanup option: register the app signing SHA-1 on the new project (after deleting from the old project) and re-OTA back to all-new-project. Low priority — current setup works.

## Manual fight-bell race after yesterday's notification N+1 fix

### Symptoms
User reported on Live/Upcoming Events screens: tap an upcoming fight card, tap the bell in the modal, the yellow bell indicator shows on the card. Worked for the first toggle, but tapping a second fight made the first one's bell disappear. Page reload restored all bells. Backend DB was correct — the issue was a refetch race that returned `notificationReasons: null` for fights whose match row hadn't been written yet.

### Root cause
Yesterday's `a71a1d7` (notification N+1 fix) removed the dynamic-evaluation fallback from both the events endpoint and `getNotificationReasonsForFight`. Both now read **only** `fightNotificationMatch`. That's correct in steady state, but `manageManualFightRule` in `services/notificationRuleHelpers.ts` was creating the rule and then firing `syncRuleMatches(newRule.id).catch(...)` **without awaiting it**, returning 200 to the client before the match row existed in the DB.

The frontend's `notifyMutation.onSuccess` immediately invalidates `['upcomingEvents']` → refetch. If the refetch hit the events endpoint before sync finished, the batched `fightNotificationMatch.findMany({ where: { isActive: true } })` came back empty for that fight, the cache replaced the optimistic value with `null`, and the bell disappeared. Before yesterday, the dynamic evaluation pass would have caught this — it re-checked active rules' conditions against each fight, and a manual rule with `conditions = { fightIds: [fightId] }` matches dynamically even when the pre-synced row hasn't landed yet.

### Fix (commit pending)
`packages/backend/src/services/notificationRuleHelpers.ts` — `manageManualFightRule` now writes the rule and the `fightNotificationMatch` row in a single Prisma transaction. No fire-and-forget sync call. Two paths:
- **Existing rule** (re-enable / disable): `$transaction([userNotificationRule.update, fightNotificationMatch.updateMany])`. Both isActive flags flip together.
- **New rule** (first-time enable): `$transaction(async (tx) => { rule.create; match.upsert })`. Uses the callback form because the match needs the freshly-created `rule.id`.

Why this is the right shape architecturally rather than just `await`-ing `syncRuleMatches`: manual fight follows have a known, bounded match set of size 1 (the fight itself). The generic `evaluateRuleAgainstUpcomingFights` evaluator iterates every UPCOMING fight in the DB and runs `evaluateFightAgainstConditions` per fight — completely wasted work for a rule whose only matching fight is the one we already have in hand. The transactional path is one round trip for the create branch, two for the toggle branch. Bell now reflects DB state on the very next refetch.

`manageFighterNotificationRule` and the hype-rule path still go through `syncRuleMatches` because their match set is genuinely unknown at creation (fighter has multiple upcoming fights; hype scores drift). Those paths weren't reported broken and aren't touched.

### What this enables
Future deterministic-match-set rule types (weight class follow, main event follow, championship follow) should follow the same pattern: write rule + matches transactionally. Only rules whose match set drifts after creation need the periodic evaluator.

### Shipped
- `c9c7ec4` — transactional rule + match write
- `d37c1a0` — orphan-recovery upsert on re-toggle (handles users who toggled before the fix and got left with rule-without-match)
- Render auto-deployed; user verified bells now persist across multiple toggles without a reload.

## Comment-like milestone push notifications

### Why
Make commenting feel rewarding so users come back. Notification fires when their pre-fight or post-fight comment gets an upvote from anyone other than themselves, with milestone tiers so a viral comment doesn't spam them: **1, 5, 10, 25, 50, 100**. Tap → opens the relevant fight detail screen (auto-routes to `UpcomingFightDetailScreen` or `CompletedFightDetailScreen` based on event status).

User constraints accepted: default-on (gated by existing master notifications toggle), thresholds as above, no quiet hours for v1.

### Schema (db push, no migration files — repo doesn't use migrations)
Two new columns, both `Int @default(0)`:
- `fight_reviews.lastNotifiedLikeCount`
- `pre_fight_comments.lastNotifiedLikeCount`

Pushed to Render via `npx prisma db push --skip-generate` (auto-loaded `.env` happens to point at the Render External URL).

### Backfill
Ran `packages/backend/src/scripts/backfillLastNotifiedLikeCount.ts` against Render. Sets `lastNotifiedLikeCount` to the highest tier already crossed by **external** upvote count (excludes the author's own auto-upvote on creation), so existing comments never trigger retroactive milestone pings.

Result:
- `fight_reviews`: 816 rows. Distribution: 671 at tier 0, 139 at tier 1, 5 at tier 5, 1 at tier 10.
- `pre_fight_comments`: 67 rows. Distribution: 57 at tier 0, 10 at tier 1.

### New service: `packages/backend/src/services/commentLikeNotifications.ts`
- `LIKE_TIERS = [1, 5, 10, 25, 50, 100]` and tier-specific copy ("Your comment got a like!" → "🚀 Your comment hit 100 likes!").
- `findTierCrossed(prevHighest, newCount)` returns highest tier in range `(prevHighest, newCount]` or `null`. If a comment jumps multiple tiers in one batch (e.g., 4→7), only the highest crossed tier fires once — never multiple notifications for one upvote.
- **External count, not raw `upvotes`**: queries `reviewVote.count({ where: { isUpvote: true, userId: { not: authorId } } })` (and the equivalent for `preFightCommentVote`). Author's auto-upvote is excluded so "1 like" fires on the first vote from someone else, matching user intent.
- **Atomic race guard**: `prisma.fightReview.updateMany({ where: { id, lastNotifiedLikeCount: { lt: tier } }, data: { lastNotifiedLikeCount: tier } })`. Two concurrent voters that both see `external=5` will both attempt to flip 1→5, but only the first UPDATE returns count=1. The loser exits without sending. No `redis-lock` / queue needed.
- **Push payload** includes `{ type: 'comment_liked', tier, commentType, commentId, fightId, eventStatus }` so mobile can deep-link without an extra round trip.
- **Also writes `UserNotification`** row (best-effort `.catch`, never blocks the upvote response). Sets up a future inbox UI without code changes — data starts accumulating now.
- Self-upvote returns early (`if (userId === voterUserId) return`).

### Wiring (`packages/backend/src/routes/fights.ts`)
Three upvote paths, all best-effort (`.catch` on the promise so a notification failure never breaks the upvote response):

| Endpoint | Path | Fires? |
|---|---|---|
| Pre-fight upvote add | `POST /fights/:id/pre-fight-comments/:commentId/upvote` (else branch) | ✅ |
| Pre-fight upvote remove | same endpoint, `if (existingVote)` branch | ❌ |
| Post-fight new upvote | `POST /fights/:fightId/reviews/:reviewId/upvote` (no existing vote) | ✅ |
| Post-fight downvote→upvote | same endpoint, `existingVote && !existingVote.isUpvote` | ✅ |
| Post-fight upvote remove | same endpoint, `existingVote && existingVote.isUpvote` | ❌ |

### Mobile (`packages/mobile/components/NotificationHandler.tsx`)
- Response listener: added `data.type === 'comment_liked' && data.fightId` branch → `router.push('/fight/${data.fightId}')`. The existing `/fight/[id].tsx` route already auto-renders Upcoming or Completed based on fight state, so no new screen logic needed.
- Cold-start (app killed): also added `data.fightId` branch to `handleInitialNotification` so a tap from the killed state lands on the fight detail, not the home screen.

Net effect is fight-start pushes also land on the right screen now (was a pre-existing bug — they went to the live-events tab).

### What this does NOT do (deliberate v1 scope)
- **No scroll-to-comment** on the destination screen. We pass `commentId` in the push data so a follow-up can wire that, but for v1 the user just lands on the fight and finds their comment in the list.
- **No per-rule toggle** in Profile UI. The existing master "Allow Notifications" switch covers it. Adding granular toggles is overkill before we know if users want them.
- **No quiet hours.** Per user.
- **No digest / weekly summary.** Future polish.

## Scraper start time audit + Tapology fix

### Problem
17 upcoming events on the admin panel were missing all three start time fields (`mainStartTime` / `prelimStartTime` / `earlyPrelimStartTime`). Without any start time set, `eventLifecycle.ts` cannot transition them from UPCOMING → LIVE, so they never go live in the app and never get a live tracker.

### Real root cause (the actual bug)
**5 of the 7 Tapology daily scrapers (Top Rank, Karate Combat, Zuffa Boxing, Dirty Boxing, Golden Boy) extracted `eventStartTime` correctly inside `page.evaluate`, then dropped it on the floor when constructing the output `event` object pushed into `latest-events.json`.** The output construction explicitly enumerates fields:

```js
const event = {
  eventName: ..., eventType: 'Regular',
  eventUrl: ..., eventSlug, venue: ...,
  city: ..., state: '', country: ...,
  dateText: ...,
  eventDate: ..., eventImageUrl: ...,
  // ← eventStartTime: was missing here. Page-eval set it on `eventData`,
  //   but it never made it onto `event` and so never reached the JSON file
  //   the parser reads.
  status: ..., fights: eventData.fights
};
```

The parser then read `eventData.eventStartTime === undefined` and silently logged "No start time found". Bug has been latent since the extraction code was added on 2026-04-06 — every CI run for ~2.5 weeks has been silently dropping the field. Gold Star and MVP already had `eventStartTime` in their output object, which is why those two never had the symptom.

Found via diagnostic logging (commit `8b029cc`) which surfaced from CI:
```
✅ Found 7 fights, eventStartTime=03:00 AM
[DIAG] dateTimeFound=true, pageContainsPMET=true
...
[TopRank] ⚠️ No start time found for "Inoue vs. Nakatani"
```
The scraper logged the value seconds before the parser claimed it didn't exist — pointing straight at the boundary between scrape output and parser input.

**Fix in commit `673a57e`:** added `eventStartTime: eventData.eventStartTime || null` to the output object in all 5 affected scrapers. Verified post-deploy: Top Rank CI run now logs `✅ Found 7 fights, start: 03:00 AM` and zero "No start time found" warnings.

### What I thought the bug was, before I found the real one
First diagnosis (commit `0f6b1d1`) was that `document.querySelector('li span.font-bold')` returned the first font-bold span in document order, which on CI sometimes landed on a fight-row label (`[W]`, `[Main Event]`, etc.) before reaching the sidebar Date/Time row. Switched all 7 scrapers to iterate `querySelectorAll(...)` and pick the label matching `/date\s*\/?\s*time/i`.

That fix solves a real edge case but **wasn't the root cause** of today's empty DB column. Diagnostic logs proved the iteration always found the right span on CI (`dateTimeFound=true` for all 7 events). I'm leaving the iteration change in — it's strictly more correct than `querySelector` — but the field-dropoff fix is what actually unblocks the data flow.

Lesson: when scraper logs say "No start time found" and a local probe extracts the value cleanly from the same URL, the failure is somewhere between the page-eval return value and the JSON file the parser reads, not in the page-eval itself. Add a console.log right after the page.evaluate returns, before any field copy/projection, to localize the dropoff.

### One-time backfill
Added `src/scripts/backfillStartTimes.ts`. For every UPCOMING event with all three start-time fields null, it:
1. Uses the event's `ufcUrl` if it already points at a Tapology event.
2. Otherwise falls back to Tapology's site search (`/search?term=...&mainSearchFilter=events`), and only then to per-org promotion hubs (`TAPOLOGY_PROMOTION_HUBS` map duplicated from `backfillTapologyResults.ts`).
3. Pulls the `<li>Date/Time:</li>` value, runs it through `eventTimeToUTC`, writes `mainStartTime`.

Result: **16 of 18 events fixed in one run.** The two skips (`Rodriguez vs. Vargas`, `Zayas vs. Ennis`) aren't fully announced on Tapology yet — Tapology's search returns either an older same-named card or no result. They'll backfill on a future run once Tapology lists them.

### PFL early card
`scrapeAllPFLData.js` already extracted main + prelim. Added `earlyPrelimStartTime` extraction (regex covers `EARLY (PRELIM[INARY]?S?|CARD)` + time + optional ET) and surfaced it through `pflDataParser.ts` so the parser writes it onto the event row alongside main/prelim. ## Duplicate fight on tonight's UFC card (Davey Grant vs Juan / Adrian Luna Martinetti)

### Symptoms
Tonight's UFC Fight Night Sterling vs. Zalal showed two fights with Davey Grant: vs `Juan Martinetti` AND vs `Adrian Luna Martinetti`. Both UPCOMING, both `cardType=Main Card / orderOnCard=4`, both with the same `ufcFightId=12698`. Same opponent, real name is **Adrian Luna Martinetti** — UFC.com originally listed him as "Juan Martinetti" then corrected.

### Root cause
Two compounding bugs:
- **Bug A — daily scraper forks a new fighter on rename.** `Fighter` upsert in `ufcDataParser.ts:298` keys on the `firstName_lastName` composite. When UFC corrected the name on 2026-04-16, the parser created a brand-new `Adrian Luna Martinetti` row instead of updating the existing one, then created a new fight (different fighter2Id → different upsert key).

But `ufcLiveParser.ts:641` decided `fightIsInScrapedData` purely from `ufcFightId`, and both DB rows shared `ufcFightId=12698`, so the live tracker un-cancelled the orphan on every tick during the live event.

### Shipped today
- **DB cleanup** (prod): orphan fight `e17ca690-7ede-4d67-92b8-9be62bb447c5` flipped to `CANCELLED` and `ufcFightId` nulled out so the live tracker can't re-resurrect it. Phantom `Juan Martinetti` fighter row kept (still referenced by the cancelled fight; deleting would orphan that historical reference). Zero ratings/predictions on either fight, so the cleanup was lossless.
- **Bug B fix** — commit `06efa7c` on `main`. `ufcLiveParser.ts` un-cancellation now requires both ufcFightId AND fighter-name signature to match; a duplicate row sharing only the id no longer counts as "in scraped data". Render auto-deployed.
