### 1. Deprecated edge-to-edge / window APIs (Android 15)

Play reports these deprecated APIs in use:
- `android.view.Window.setStatusBarColor` / `setNavigationBarColor` / `getStatusBarColor` / `getNavigationBarColor`
- `LAYOUT_IN_DISPLAY_CUTOUT_MODE_SHORT_EDGES`, `LAYOUT_IN_DISPLAY_CUTOUT_MODE_DEFAULT`

Origins are all **third-party/framework code**, not our app source:
- `com.facebook.react.views.view.WindowUtilKt.enableEdgeToEdge` / `statusBarHide` / `statusBarShow` (React Native core)
- `com.google.android.material.bottomsheet.BottomSheetDialog.onCreate`, `EdgeToEdgeUtils.applyEdgeToEdge`, `sidesheet.SheetDialog.onCreate` (Material)
- `com.swmansion.rnscreens.ScreenWindowTraits.setColor` / `setNavigationBarColor` (react-native-screens)
- `expo.modules.devlauncher...DevLauncherExpoActivityConfigurator.setColor` (dev-launcher, dev-only)
- `expo.modules.imagepicker.ExpoCropImageUtils.applyWindowTheming` (expo-image-picker)

**Assessment:** these resolve as RN/Expo/Material upgrade away from the deprecated
window-color APIs. Not actionable in our code today ‚Äî fix is to bump Expo SDK / RN /
react-native-screens once their releases drop the deprecated calls. Revisit on next
SDK upgrade.

### 2. Resizability / orientation restrictions (Android 16)

From Android 16, large-screen devices (foldables, tablets) will **ignore** orientation
restrictions. Play detected portrait locks on:
- `com.google.mlkit.vision.codescanner.internal.GmsBarcodeScanningDelegateActivity` (PORTRAIT) ‚Äî third-party (ML Kit)
- `com.fightcrewapp.mobile.MainActivity` (PORTRAIT) ‚Äî **ours**, from `app.json` `"orientation": "portrait"`

**Assessment:** the app is designed portrait-only (`supportsTablet: false` on iOS too),
so this is intentional, but on Android 16 tablets/foldables it'll render in landscape
regardless and may show layout issues. If we ever care about large-screen Android,
test landscape layouts and relax the lock. Low priority ‚Äî combat-sports phone app,
no tablet story today.

### Root cause
The per-fight cancellation pass treats a missing fight uniformly under the two-strike rule
(`CANCELLATION_STRIKE_THRESHOLD = 2`): a fight must be absent on **two consecutive** scrapes
before it's CANCELLED. That guard exists to survive transient UFC.com render glitches.

But a **rebooking is not a transient glitch** ‚Äî when the same fighter reappears in a
*different* matchup in the same scrape, the old bout is definitively dead (a fighter can't be
on two bouts of one card). The code already computed `fighter1Rebooked`/`fighter2Rebooked` and
labelled the reason `'fighter rebooked'`, but still applied the two-strike delay. When UFC.com
flapped between old and new matchups, the strike counter kept resetting and the stale bout
never reached two consecutive misses ‚Üí both stayed UPCOMING. Observed strike state confirmed
flapping: old Mitchell/Luna had `strike=0` (reset by a re-show) while new Mitchell/Henry had
`strike=1`.

### Fix
When a fighter is rebooked (present in the scrape but in a different matchup), bypass the
two-strike wait and cancel immediately:

```js
const rebooked = fighter1Rebooked || fighter2Rebooked;
const { newCount, shouldCancel: strikeReached } = decideStrike(dbFight.missingScrapeCount);
const shouldCancel = strikeReached || rebooked;
```

Still runs inside the existing sanity gates (`scrapeIsSane`, `!eventInProgress`,
`scrapeLooksComplete`), so a broken/partial scrape can't trigger it. The un-cancel-on-reappear
path is unchanged, so a genuine source flap self-corrects to whichever matchup the source
currently shows.

**Applied to all 16 daily parsers**, across two parser shapes:
- **10 name-based** (`ufc, bkfc, goldenBoy, goldStar, matchroom, oktagon, oneFC, pfl, rizin,
  topRank`) already computed `fighter1Rebooked`/`fighter2Rebooked` from a `scrapedFighterNames`
  set ‚Äî just OR'd it into the cancel decision.
- **6 ID-signature** (`gamebred, dirtyBoxing, zuffaBoxing, mvp, raf, karateCombat`) had no
  rebooking detection at all. Added a parallel `scrapedFighterIds` set next to
  `scrapedFightSignatures` (populated in the same import loop) and detect rebooking via
  `scrapedFighterIds.has(dbFight.fighter1Id/fighter2Id)`.

## Repeated mobile logouts on heavy-deploy days ‚Äî the 2026-05-28 fix was incomplete

**Symptom (recurring):** Mike kept getting logged out of the **mobile app** on big work
days ‚Äî the same thing we thought we fixed on 2026-05-28.

**Root cause:** the 2026-05-28 fix (`lesson_render_redeploy_token_rotation_logouts`) only
hardened `services/api.ts` ‚Äî the **request path**. But `store/AuthContext.tsx` has a
**second, separate** refresh/validation path that was never touched, and it's the one that
actually fires on a heavy-deploy day:

- `initializeAuth()` ‚Äî runs on every **cold-start**. A 502/503 on the startup `/auth/profile`
  call (not a 401) fell through to the `else` branch and **wiped tokens**; an expired token +
  a refresh landing in a restart window also wiped.
- `refreshUserDataInternal()` ‚Äî runs on every **foreground** (AppState listener). Refresh
  hitting 502/503 ‚Üí **wiped tokens**.
- `refreshUserData()` (profile mount, after rating, filter changes) and `refreshToken()`
  (called `logout()` on any failure) ‚Äî same class of bug.

A restarting Render container surfaces as a **502/503 from the proxy**, which is an HTTP
*response*, not a network throw ‚Äî so the existing try/catch fallbacks didn't catch it, and the
app treated "server briefly restarting" as "your token is bad ‚Üí log out." Many restarts per
heavy push day = repeated logouts.

**Fix (`packages/mobile/store/AuthContext.tsx`):** added one hardened `tryRefreshTokens()`
helper (mirrors `api.ts`: retries 5xx/network 3√ó with backoff, classifies
`success` / `invalid` / `transient`) and routed all four paths through it. Rule everywhere now:
**5xx/network during a redeploy ‚Üí transient ‚Üí keep the session** (fall back to cached user
data); **only a genuine 4xx rejection from `/auth/refresh` ‚Üí wipe + logout.** Confirmed the
backend returns 200 on success / 401 on genuine rejection / 5xx-or-connection-fail while
restarting, so the classification is correct. The 60s backend grace window + `api.ts` retry
were already in place ‚Äî this closes the last uncovered path.

**Status:** one file changed, **type-check clean** on it. **Not committed/pushed/OTA'd by this
window** ‚Äî multiple Claude windows were running concurrently; per Mike, a single designated
window does the commit + push + `eas update --branch production` last, so its one OTA bundles
every window's work (an OTA ships the whole JS tree, not a per-window diff). Behavioral fix ‚Äî
true confirmation requires foregrounding/cold-starting the app during a backend restart and
staying logged in, post-OTA.

**Still open (flagged, not touched):** the **web app** has the same class of bug ‚Äî
`packages/web/src/app/api/auth/refresh/route.ts` deletes the `refreshToken` cookie on *any*
non-OK backend response, including 502/503 during a redeploy (permanent logout until
re-login), and `packages/web/src/lib/api.ts` wipes the access token on any refresh failure.
Web was left as-is this session; mirror the transient-vs-invalid fix there when picked up.
