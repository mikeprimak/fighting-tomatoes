### 2. Resizability / orientation restrictions (Android 16)

From Android 16, large-screen devices (foldables, tablets) will **ignore** orientation
restrictions. Play detected portrait locks on:
- `com.google.mlkit.vision.codescanner.internal.GmsBarcodeScanningDelegateActivity` (PORTRAIT) — third-party (ML Kit)
- `com.fightcrewapp.mobile.MainActivity` (PORTRAIT) — **ours**, from `app.json` `"orientation": "portrait"`

**Assessment:** the app is designed portrait-only (`supportsTablet: false` on iOS too),
so this is intentional, but on Android 16 tablets/foldables it'll render in landscape
regardless and may show layout issues. If we ever care about large-screen Android,
test landscape layouts and relax the lock. Low priority — combat-sports phone app,
no tablet story today.

## Rebooked-fighter cancellation bypass (daily scraper parsers)

### Problem
UFC.com cancelled two bouts on **UFC Fight Night Muhammad vs. Bonfim (2026-06-06)** and
replaced them with new opponents. The daily UFC scraper imported the new fights but left the
old ones live, so the card showed both old and new as UPCOMING:

- Bryce Mitchell vs **Santiago Luna** (out) + Bryce Mitchell vs **Victor Henry** (in)
- Iwo Baraniewski vs **Billy Elekana** (out) + Iwo Baraniewski vs **Junior Tafa** (in)

### Root cause
The per-fight cancellation pass treats a missing fight uniformly under the two-strike rule
(`CANCELLATION_STRIKE_THRESHOLD = 2`): a fight must be absent on **two consecutive** scrapes
before it's CANCELLED. That guard exists to survive transient UFC.com render glitches.

But a **rebooking is not a transient glitch** — when the same fighter reappears in a
*different* matchup in the same scrape, the old bout is definitively dead (a fighter can't be
on two bouts of one card). The code already computed `fighter1Rebooked`/`fighter2Rebooked` and
labelled the reason `'fighter rebooked'`, but still applied the two-strike delay. When UFC.com
flapped between old and new matchups, the strike counter kept resetting and the stale bout
never reached two consecutive misses → both stayed UPCOMING. Observed strike state confirmed
flapping: old Mitchell/Luna had `strike=0` (reset by a re-show) while new Mitchell/Henry had
`strike=1`.

### Fix

## Repeated mobile logouts on heavy-deploy days — the 2026-05-28 fix was incomplete

**Symptom (recurring):** Mike kept getting logged out of the **mobile app** on big work
days — the same thing we thought we fixed on 2026-05-28.

**Root cause:** the 2026-05-28 fix (`lesson_render_redeploy_token_rotation_logouts`) only
hardened `services/api.ts` — the **request path**. But `store/AuthContext.tsx` has a
**second, separate** refresh/validation path that was never touched, and it's the one that
actually fires on a heavy-deploy day:

- `initializeAuth()` — runs on every **cold-start**. A 502/503 on the startup `/auth/profile`
  call (not a 401) fell through to the `else` branch and **wiped tokens**; an expired token +
  a refresh landing in a restart window also wiped.
- `refreshUserDataInternal()` — runs on every **foreground** (AppState listener). Refresh
  hitting 502/503 → **wiped tokens**.
- `refreshUserData()` (profile mount, after rating, filter changes) and `refreshToken()`
  (called `logout()` on any failure) — same class of bug.

A restarting Render container surfaces as a **502/503 from the proxy**, which is an HTTP
*response*, not a network throw — so the existing try/catch fallbacks didn't catch it, and the
app treated "server briefly restarting" as "your token is bad → log out." Many restarts per
heavy push day = repeated logouts.

**Fix (`packages/mobile/store/AuthContext.tsx`):** added one hardened `tryRefreshTokens()`
helper (mirrors `api.ts`: retries 5xx/network 3× with backoff, classifies
`success` / `invalid` / `transient`) and routed all four paths through it. Rule everywhere now:
**5xx/network during a redeploy → transient → keep the session** (fall back to cached user
data); **only a genuine 4xx rejection from `/auth/refresh` → wipe + logout.** Confirmed the
backend returns 200 on success / 401 on genuine rejection / 5xx-or-connection-fail while
restarting, so the classification is correct. The 60s backend grace window + `api.ts` retry
were already in place — this closes the last uncovered path.

**Status:** one file changed, **type-check clean** on it. **Not committed/pushed/OTA'd by this
window** — multiple Claude windows were running concurrently; per Mike, a single designated
window does the commit + push + `eas update --branch production` last, so its one OTA bundles
every window's work (an OTA ships the whole JS tree, not a per-window diff). Behavioral fix —
true confirmation requires foregrounding/cold-starting the app during a backend restart and
staying logged in, post-OTA.

**Still open (flagged, not touched):** the **web app** has the same class of bug —
`packages/web/src/app/api/auth/refresh/route.ts` deletes the `refreshToken` cookie on *any*
non-OK backend response, including 502/503 during a redeploy (permanent logout until
re-login), and `packages/web/src/lib/api.ts` wipes the access token on any refresh failure.
Web was left as-is this session; mirror the transient-vs-invalid fix there when picked up.
