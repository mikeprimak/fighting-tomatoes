## Disk cleanup — 43 GB project folder → 3.4 GB

User noticed the `fight-mobile-app` folder was 43 GB. Investigated with PowerShell `Get-ChildItem`.

### Root cause
`packages/backend/live-event-data/oktagon/` held **39.52 GB** in 934 JSON files, all from one Oktagon live-tracker run on Jan 17, 2026. File sizes climbed from 0.1 MB → 86.6 MB because `save()` wrote the entire accumulated `snapshots` array on every call, and each call produced a NEW file (filename had `Date.now()` baked in). So the last file alone contained every snapshot the prior 933 files contained.

Folder is in `.gitignore:133` — never was tracked.

### Fix
Changed three live-scraper `save()` implementations so the filename is stable per run instead of changing every save. Each class now gets a `private readonly runTimestamp: number = Date.now()` field at instance construction, and `save()` uses `this.runTimestamp` instead of calling `Date.now()` inline. Result: repeated `save()` calls overwrite the same file instead of creating a new one each time.

| File | Change |
|---|---|
| `packages/backend/src/services/oktagonLiveScraper.ts` | Added `runTimestamp` field; `save()` uses it in filename |
| `packages/backend/src/services/matchroomLiveScraper.ts` | Same fix (had same bug pattern) |
| `packages/backend/src/services/oneFCLiveScraper.ts` | Same fix (had same bug pattern) |

### Cleanup
- Deleted `packages/backend/live-event-data/oktagon/` (39.5 GB freed).
- Total project size: **43 GB → 3.39 GB**.
- No other similar bug sources found (UFC/BKFC/Tapology live trackers don't use this snapshot-array pattern).

## iOS Google Sign-In crash — root-caused + new builds shipped

### The bug
Multiple users reported the app crashes instantly when they tap "Sign in with Google". One confirmed iOS user on latest iOS. Another user reported it "a few days ago", which prompted the Apr 14 Sentry install (commit `d40c60c`).

### Sentry was dark
Checked Sentry API (org `good-fights`, project `react-native`). **Only 1 event ever captured** — the verification test user ran from a Pixel 6a on 2026-04-15. Zero real events in 3+ days.

Reason: `@sentry/react-native` native crash handler requires a native binary rebuild. The Apr 14 install only reached users via OTA (JS-only) for iOS, and the Android versionCode with Sentry had never been uploaded to Play Store. So **no production user had the native Sentry SDK** — native-level crashes (which "instant crash on tap" strongly indicates) wouldn't be captured even on the updated users.

### Root cause
Google Sign-In iOS SDK requires the reversed iOS OAuth client ID to be registered as a `CFBundleURLTypes` URL scheme in Info.plist. On recent iOS versions, a mismatch causes a native `fatalError()` that JS can't catch. Three Google Cloud references were mismatched across two projects:

| Location | Value | Google Cloud Project |
|---|---|---|
| `hooks/useGoogleAuth.ts:8` `GOOGLE_CLIENT_ID_WEB` | `1082468109842-pehb7kk...` | fight-app-ba5cd |
| `hooks/useGoogleAuth.ts:9` `GOOGLE_CLIENT_ID_IOS` | `1082468109842-qpifgfj...` | fight-app-ba5cd |
| `app.json` iOS `CFBundleURLSchemes` | `com.googleusercontent.apps.499367908516-j03poule...` | **good-fights-app** |

Verified in Google Cloud Console: both projects have iOS OAuth clients for `com.fightcrewapp.mobile`, but only `good-fights-app`'s client has a Team ID set (`K7TB3SVUMP`) and was last used Jan 7, 2026. The `fight-app-ba5cd` iOS client (created Jan 12) was never used beyond its creation day — it was an accidental duplicate created when someone set up Firebase/FCM separately. The `499367908516` URL scheme in Info.plist was correct; the code was wrong.

### Why Android worked anyway
User's Android install had Google Sign-In working despite the code pointing at the wrong project. On Android, Google Sign-In validates package name + SHA-1 fingerprint against an Android OAuth client; if either project has a matching Android client, sign-in succeeds. The backend `GOOGLE_CLIENT_ID` env var was unset on Render, so `OAuth2Client.verifyIdToken({ audience: undefined })` skipped audience validation — it was working by luck.

### Fix

**Mobile code** (`packages/mobile/hooks/useGoogleAuth.ts`): both client IDs now point at good-fights-app:
- `GOOGLE_CLIENT_ID_WEB` → `499367908516-f5qu2rjeot6iqnhld7o3tg71tqdqlngk.apps.googleusercontent.com`
- `GOOGLE_CLIENT_ID_IOS` → `499367908516-j03poule51s7sfvpvdufna0upqa3oseg.apps.googleusercontent.com`

**Backend** (`packages/backend/src/controllers/authController.ts` + `src/routes/auth.fastify.ts`): added `getGoogleAudiences()` helper that parses `GOOGLE_CLIENT_ID` as an **optional comma-separated list of accepted audiences**. Lets old + new client IDs coexist during the rollout window. Also switched `new OAuth2Client(process.env.GOOGLE_CLIENT_ID)` → `new OAuth2Client()` since the ctor arg is just the default audience — the per-call `audience` option overrides it.

**Render env var** `GOOGLE_CLIENT_ID` set to (new first, old second, no spaces):
```
499367908516-f5qu2rjeot6iqnhld7o3tg71tqdqlngk.apps.googleusercontent.com,1082468109842-pehb7kkuclbv8g4acjba9eeeajprd8j7.apps.googleusercontent.com
```

Backend audience verification is now ON (it wasn't before). Drop the legacy client from the env var once shipped-app adoption of the new build is high.
