## Fan DNA personality "types" — SHIPPED

Picked up the top forward path from yesterday's handoff: aggregate the 5+ batch-computed trait values into a single shareable identity label. Highest user-visible payoff per the area-doc ranking because it turns N independent trait cards into one synthesized hero.

### Architecture

New module `packages/backend/src/services/fanDNA/personalityType.ts` lives **outside** the trait registry. Keeping it separate preserves trait independence — each `Trait` stays a single-signal plugin, and the type aggregator reads `TraitValue` rows after the engine has computed them.

Public surface is one function:

```ts
computeUserType(prisma, userId): Promise<UserPersonalityType | null>
```

Wired into `GET /api/fan-dna/profile` — returns `{ personalityType, cards, count }` alongside the existing trait cards. The `/profile` endpoint's staleness pass already lazily recomputes TraitValue rows before the type pass runs, so the type always reflects current data.

### Rule design

Priority-ordered ladder of 11 rules, first match wins:

| Rank | Type | Trigger |
|---|---|---|
| 1 | Hot Take Artist | `hotTakeCount ≥ 3 AND totalHypedFights ≥ 10` |
| 2 | Trailblazer | `firstHypeCount ≥ 5` |
| 3 | The Oracle | `accuracyPct ≥ 60 AND totalHypedFights ≥ 15` |
| 4 | The Skeptic | `rating-bias.n ≥ 15 AND avgDelta ≤ −0.7` |
| 5 | The Generous Critic | `rating-bias.n ≥ 15 AND avgDelta ≥ +0.7` |
| 6 | The Hype Believer | `hype-bias.n ≥ 15 AND avgDelta ≥ +0.7` |
| 7 | The Doubter | `hype-bias.n ≥ 15 AND avgDelta ≤ −0.7` |
| 8 | {Org} Loyalist | non-UFC dominant ≥ 60% AND total ≥ 15 |
| 9 | UFC Mainstay | UFC dominant ≥ 60% AND total ≥ 15 |
| 10 | The Globetrotter | ≥ 4 promotions AND no org ≥ 60% share |
| 11 | The Calibrator | both biases within ±0.3 with n ≥ 15 each |

Returns `null` when no rule clears its floor — hero card stays hidden, trait cards still appear if they fired.

### Two corrections during verification

First probe of top-5 raters showed 4/5 collapsing to "Trailblazer" because legacy ratings were bulk-imported and `firstRateCount` is dishonest signal for legacy users (cookmeyer had 709 firsts out of 1635 ratings, qegjkirljzy had 901/2384 — both clearly migration artifacts). This is the existing `lesson_dataset_aggregates_dishonest` memory in action.

**Fix 1:** Trailblazer rule now gates on `firstHypeCount` only. Hypes happen pre-fight and weren't migrated en masse, so they're a truthful "you saw it coming" signal. Renamed primary stat from "firsts" to "first hypes" so the copy matches the metric.

Second issue: Globetrotter rule was guarded by `if (ratingDominant) return null`, but org-affinity sets `ratingDominant` once any org passes 40%. A 48%-UFC user with 10 promotions is a Globetrotter, not "ufc-but-not-quite-dominant" with no type at all. Tested case: cry.hagen@mailbox.org (10 orgs, UFC at 48%).

**Fix 2:** Globetrotter now checks `dominantPct < 60%` instead of `!dominant`. The 40-59% band gets the diverse-fan identity.

Also added `totalHypedFights ≥ 10` to Hot Take Artist so thin-data users can't backflip into it by 3 lucky hot takes among 5 hypes.

### Subtle decisions logged

- **Personality type lives outside the trait registry.** Traits are independent single-signal plugins; type is a cross-trait synthesizer. Keeping it in a separate module preserves the engine invariants (one trait can't break another, registry auto-discovery, etc.).
- **First match wins, no scoring.** The priority order encodes specificity (Hot Take Artist before generic UFC Mainstay). A scoring system would let multiple types blend, but the requested UX is one identity per user.
- **No secondary type yet.** First-cut ships the primary label only. "With shades of {secondary}" copy line is a future iteration once Mike sees the v1 land.
- **`hasFloor` gates rule access.** The `get<T>` helper in `personalityType.ts` returns `null` for any trait whose `TraitValue.hasFloor` is false. Rules can't accidentally read thin-data values — they have to opt in by checking explicit thresholds.
- **Trailblazer rule body uses "first to hype" not "first to weigh in".** Yesterday's commit `8d48e6f` already moved to plain English ("first to rate", "first to hype"); the type body copy matches that vocabulary instead of the older `secondaryStat: 'first-ever'`.
