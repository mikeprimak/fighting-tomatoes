# 2026-04-30

## ONE FC scraper cadence: 1×/day → every 6h

User reported ONE Friday Fights 152 (Inner Circle) showing the full card on onefc.com but 0 fights in the app. ONE 152 starts 2026-05-01T11:30 UTC (2026-05-01T18:30 ICT).

### Investigation

DB showed the event row existed (`scraperType=onefc`, status=UPCOMING, fights=0) but empty. Same pattern on 153, 156, 157, 158, 159, 160. Last night's scheduled scrape ran 2026-04-30T23:45 UTC and reported `Scraped 0 fights, start: 2026-05-01T18:30:00+07:00` for 152.

Initial hypothesis was that the hero-duplicate filter from `0031664` (rejecting `.event-matchup` inside `.box-post-event`/`.event-live-status`/`.status-matchup`) had become over-broad on a new template. Pulled fresh HTML for 152, 153, 154, 155, 156, 157, 160, and Fight Night 43, parsed offline with cheerio:

| Event | DOM matchups | passes filter | JSON-LD performers |
|---|---|---|---|
| ONE 152 | 13 | 13 | 13 |
| ONE 153 | 0 | 0 | 0 |
| ONE 154 | 3 | 3 | 3 |
| ONE 155 | 2 | 2 | 2 |
| ONE 156 | 0 | 0 | 0 |
| ONE 157 | 0 | 0 | 0 |
| ONE 160 | 0 | 0 | 0 |
| FN 43 | 6 | 6 | 6 |

Filter cleared. The seven empty-in-DB events split cleanly: 152 has a fully-populated card on the live site; the others (153, 156, 157, 158, 159, 160) genuinely don't have cards published yet. The 152 card was published in the ~2h gap between the scheduled scrape and the curl fetch.

### Root cause

`onefc-scraper.yml` cron was `0 23 * * *` — once daily. ONE Friday Fights cards are routinely published <24h before the event (existing comment in `scrapeAllOneFCData.js` already calls this out for late "Inner Circle" additions). Once a card lands after the 23:00 UTC daily run, the next scheduled run lands after the event has already started. The live tracker (`oneFCLiveScraper.ts`) only updates fights it already has rows for — it doesn't populate the card from scratch.

### Fix

Bumped cron to `0 */6 * * *` (every 6 hours). 4 runs/day, ~5 min each, ~20 min/day total — well within free-tier budget. The same hero-duplicate filter and JSON-LD performer-name fallback all stay intact. No selector or parser changes needed.

Manual `workflow_dispatch` triggered at 2026-05-01T02:09 UTC to confirm and immediately backfill: `Scraped 13 fights, start: 2026-05-01T18:30:00+07:00` → `Imported 13/13 fights` for ONE Friday Fights 152. ONE 153/156/157/158/159/160 stayed empty, as expected.
