## Changes Made

### Pull-to-refresh on all main tabs (commit `e3a5417`)
Added a `RefreshControl` to every main tab screen. Each refresh re-runs the tab's primary data query; React Query's `refetch()` is awaited so the spinner stays visible until the network round-trip completes.

Files touched:
- `packages/mobile/app/(tabs)/live-events.tsx` — FlatList, wraps the `['upcomingEvents', ...]` infinite query
- `packages/mobile/app/(tabs)/events/index.tsx` — FlatList, same `['upcomingEvents', ...]` query
- `packages/mobile/app/(tabs)/past-events/index.tsx` — FlatList, `['pastEvents', ...]` query
- `packages/mobile/app/(tabs)/top-fights/index.tsx` — FlatList, `['topRecentFights', ...]` query
- `packages/mobile/app/(tabs)/profile.tsx` — ScrollView; refresh fans out to `refreshUserData`, `getPredictionAccuracyByEvent`, `getGlobalStanding`, `getMyTopReviews`, `getMyTopPreflightComments`

All use the same shape: `const [isRefreshing, setIsRefreshing] = useState(false)` + an async `onRefresh` callback that awaits the refetch. `tintColor`/`colors` both set to `colors.primary` so the spinner matches the brand yellow.

### Live events periodic self-update
Live Events already polled every 30s via `refetchInterval: 30000` on the `['upcomingEvents', ...]` infinite query — it was working before. Added `refetchIntervalInBackground: false` explicitly so we don't burn requests when the app is backgrounded (default behavior anyway, but now it's intentional).

### Event screens load-time fixes (commits `a7d2d8e`, `462b178`)
User reported Upcoming/Live Events screens taking ~10s to load on app start, target ~3s. Investigated mobile + backend; root causes were compounding.

Mobile-side quick wins (commit `a7d2d8e`):
- `packages/mobile/app/(tabs)/events/index.tsx` — swapped `queryClient.invalidateQueries` for `refetchQueries({ type: 'active', stale: true })` on tab focus. Quick tab switches within the 30s staleTime now reuse the cache instead of hitting the network.
- Same file — `refetchOnMount: 'always'` → `refetchOnMount: true`. Mount now respects staleTime too.
- `packages/mobile/app/(tabs)/live-events.tsx` — removed the auto-fetch-page-2 `useEffect` that doubled the round trips on every startup. Page 2 now only loads via pull-to-refresh or scroll.

Backend cold-start fix (commit `462b178`):
- Render free tier spins down after ~15 min idle, adding 3–5s to the first load. Added `.github/workflows/keep-backend-warm.yml` — pings `/health` every 10 min via GH Actions cron. The `/health` endpoint runs `SELECT 1` so the DB connection pool stays warm in the same request.
- Manual test run confirmed HTTP 200 in <1s with `database: connected`.
- Note: the first commit was rejected by GitHub because the local PAT lacked the `workflow` scope. User updated the token, then the push succeeded.

Not fixed yet (documented for later):
- Backend runs 2 sequential Prisma queries + in-memory sort in `packages/backend/src/routes/index.ts:406-442` (the upcoming-events endpoint). Adds 300–500ms.
- Per-fight notification rule lookup loop in `packages/backend/src/routes/index.ts:574-586`. ~500–1000ms for a typical 2-event page.
- React Query cache is in-memory only — first launch of the app still has no cache to reuse. Could add the AsyncStorage persister later for instant cold-start loads.
