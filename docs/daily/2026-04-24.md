### Notification-reasons N+1 fix (commit `a71a1d7`)
After the mobile + warm-up fixes, startup was down to ~5s from ~10s. The remaining 1–2s on the events endpoint was the per-fight notification lookup — `Promise.all(fightIds.map(getNotificationReasonsForFight))`. Each call did ~4–6 DB round trips (fight lookup + pre-synced matches + active-rules fetch + dynamic eval per unmatched rule), so a 10-fight page was firing ~60 queries in parallel and hammering the Render pool.

Investigation revealed the dynamic-evaluation pass only existed for rule types whose match set can change after rule creation — specifically **hype-based rules** (hype scores drift as predictions roll in). Per user, hype and fighter-follow *notifications* are currently turned off; manual per-fight bells are the only firing notification type. Manual follows pre-sync into `fightNotificationMatch` up-front with an immutable `fightIds` condition, so a plain read of that table is authoritative today.

Changes:
- `packages/backend/src/routes/index.ts:574-612` — events endpoint now does a single `fightNotificationMatch.findMany({ userId, fightId: { in: allFightIds }, isActive: true })` and groups results by `fightId`. Also fixed a subtle regression from the earlier `a7d2d8e` edit: reason `type` is classified by rule-name prefix (`'Manual Fight Follow:'` → manual, `'Fighter Follow:'` → fighter, else rule) the way the original code did, not hardcoded to `'manual'`.
- `packages/backend/src/services/notificationRuleEngine.ts:getNotificationReasonsForFight` — simplified to the same one-query form. Callers in `fights.ts`, `search.ts`, and `notificationRules.ts` get faster for free; output shape preserved.

**Recoverability (user will reintegrate hype/fighter notifications later)**: nothing deleted. `evaluateFightAgainstConditions`, `evaluateUserRulesForFight`, `evaluateRuleAgainstUpcomingFights`, `syncRuleMatches`, `syncAllUserRuleMatches`, all of `notificationRuleHelpers.ts` (`manageFighterNotificationRule`, `managePreEventReportRule`, etc.), and `routes/notificationRules.ts` remain intact. The fighter-follow endpoints in `routes/index.ts:1475,1542,1707` still create Fighter Follow rules on follow. A clear restoration-path comment on `getNotificationReasonsForFight` explains how to re-add the dynamic eval block when hype rules come back.

Why simplification is safe even though creation code still runs: pre-synced matches from fighter follows, pre-event reports, etc. still land in `fightNotificationMatch` the way they always did, so the one-query read surfaces them identically to the old code. The only behavior the simplification drops is dynamic re-evaluation of stale hype-threshold conditions, which are disabled from firing anyway.

Type-check of the two touched files: clean (pre-existing errors in admin UI / old controllers are unrelated).

### BKFC Clearwater start time fix
User reported the Apr 24 BKFC Clearwater event displaying as 12:00 AM in the mobile app; bkfc.com lists 8:00 PM EDT.

Root cause: `event.mainStartTime` in prod was `2026-04-25T04:00:00Z` (= April 25 00:00 EDT, i.e. midnight). The BKFC scraper had picked up "12:00 AM" from a `[data-countdown-date]` rollover element on the event detail page during an earlier production scrape. (Local Mar-27 scrape correctly produced "8:00 PM" — the bug is intermittent and depends on which selector wins.)

Fix:
- One-time DB update on Render: `mainStartTime` set to `2026-04-25T00:00:00Z` (= April 24 8:00 PM EDT) for the BKFC Clearwater event. `date` was already correct.
- Hardened scraper + parser to discard `"12:00 AM"` as a known countdown-rollover artifact:
  - `packages/backend/src/services/scrapeAllBKFCData.js` — both `scrapeEventsList` and `scrapeEventPage` now null out `eventStartTime` when it equals "12:00 AM", letting later strategies / events-list-page values win in the merge.
  - `packages/backend/src/services/bkfcDataParser.ts:parseEventStartTime` — same guard at the parser level. Returns `null` for "12:00 AM" so the upsert passes `mainStartTime: undefined` and preserves the existing correct value rather than overwriting it.

The next 19:00 UTC daily scrape will run with the new code; if BKFC's page state regresses, the existing DB value won't be clobbered.
