## Web: two-column layout held to md (768px)

## Notifications: follow-fighter morning-of/3-day fired a day early (US users)

**Report:** Friday 9am Mike got 5 "X fights today" pushes (Matcharashvili, Holm,
Steveson, Snyder, Tsarukyan) for fighters on Saturday's card — a full day early.

**Cause:** `followFighterNotifications.computeFightDay()` reinterpreted
`Event.date` in the *user's* timezone and rolled the day back when the local hour
was < noon (the rollback was meant for genuine 3am overnight starts). But
`Event.date` is a **date marker stored at a fixed UTC hour, not a real kickoff
instant** — `ufcDataParser.ts:258` stores UTC noon on the event day; Tapology
parsers store 00:00 UTC. The UFC noon-UTC marker reads as 5-8am local across the
*entire* US (verified NY/Chi/Den/LA), so the rollback always fired →
`fightDay` came out as the previous calendar day → morning-of AND 3-day-warn
fired one day early for every US user. UK/Europe (UTC+) were unaffected. This was
systematic, not an edge case; surfaced now because Saturday's card was the first
with followed fighters whose morning-of window opened.

**Fix:** `packages/backend/src/services/followFighterNotifications.ts` —
`computeFightDay` now reads the event day straight off `Event.date`'s UTC
components (no per-user TZ reinterpretation, no rollback). Both placeholder
formats (UFC noon-UTC, Tapology midnight-UTC) carry the intended day as their UTC
date. The 9am/10am *send time* still localizes to `user.timezone` via
`localWallClockToUTC`. Type-checks clean on strict + production tsconfigs.

Commit `7e06133`, pushed to `main` (Render auto-deploy).

**Residual:** users who already received the early push have `morningOfSentAt`
set, so they won't get a corrected one for Saturday — they just got it a day
early this once. Not-yet-sent matches fire correctly after deploy.

**Future:** to restore true overnight handling (e.g. genuine 3am ONE FC starts),
feed the rollback a *real* timestamp from `Event.mainStartTime`/`prelimStartTime`
rather than the `Event.date` marker. Deferred — not needed for the current bug.

## Results backfill: Tapology completed-event results never reaching the app

**Report (Mike):** Several recent past cards showed no winner/method — Usyk vs.
Rico (Matchroom), BKFC Palm Desert, Top Rank Davis vs. Albright II, Zuffa Boxing 6,
Rizin 53. "I thought we had a system that goes back after fights to fill in outcome
data if the live tracker missed it — why aren't these populated?"

**The system exists and is built.** Two daily orchestrators:
- Phase 1 `backfillTapologyResults.ts` (`.github/workflows/tapology-backfill.yml`,
  09:00 UTC, 7-day window) — all `scraperType='tapology'` orgs.
- Phase 2 `backfillResults.ts` (`results-backfill.yml`, 10:00 UTC, 14-day window) —
  native orgs (ufc/bkfc/onefc/oktagon/matchroom/pfl/raf); **explicitly skips tapology**.

Both run daily and report "success." But the Tapology side has been a **silent no-op**,
and the root cause was three layered bugs:

1. **GitHub Actions gets HTTP 403 from Tapology.** Tapology blocks datacenter/GH-Actions
   IP ranges. From a residential IP both Node `fetch` and `curl` return 200 (verified).
   This is the *inverse* of ufc.com (which allow-lists GH and rate-limits home IPs). So
   the daily Phase-1 job fetches nothing — every event errors with 403 and the orchestrator
   catches each error and still exits 0 ("success"). The Tapology **live tracker** works only
   because it runs on the Hetzner VPS (unblocked IP); the **backfill** was put on GH Actions.

2. **Even on a successful scrape, results land only in shadow fields.** The scraper extracts
   every winner/method/round correctly, but `buildTrackerUpdateData` (liveTrackerConfig.ts)
   writes the real `winner`/`method` columns only when `shouldAutoPublish(scraperType)`.
   `production_scrapers` = `["ufc","oktagon","bkfc","onefc","pfl"]` — **tapology + raf are
   deliberately opted out** (review gate for the unreliable live tracker). So Tapology results
   go to `trackerWinner`/`trackerMethod` only and require a manual admin Publish that nobody runs.

3. **The admin publish-all endpoint would skip them anyway** — it gates on `trackerFightStatus`,
   which is null for fights the lifecycle had already marked COMPLETED before backfill ran.

**Fix shipped (commit `36046f8`, branch `home-screen`):** `backfillTapologyResults.ts` now
runs a post-parse `publishCompletedShadowResults()` that promotes shadow → published for the
event's completed fights. Null-only on `winner` (never clobbers a manual fix or a real live
result), does **not** gate on `trackerFightStatus`. **LIVE tracking stays gated/review-only** —
unchanged. This is sound because the backfill is the trusted days-later reconciliation: by then
the source has settled.

**Backlog cleared (ran locally from home IP, Render DB, 90-day window):** ~153 results published
across Matchroom, Top Rank, Zuffa Boxing, Rizin, Karate Combat, Dirty Boxing, MVP, Gold Star,
Gamebred, PFL MENA. Named events now complete: Usyk 10/11, Top Rank Davis/Albright 6/6, Zuffa 6
8/8, Rizin 53 15/15, Rizin 52 12/12. (Tapology rate-limits the home IP after ~7-8 sequential
scrapes; took two passes.) Also repointed Usyk's `ufcUrl` from matchroomboxing.com → the Tapology
event URL (discovery had failed because the Matchroom hub `slugFilter:['matchroom']` rejects
fighter-named event slugs like `usyk-vs-verhoven`).

**Automated (corrected approach — commits `d9ed6ca` + `5d57c36`, main):** First attempt assumed
the Hetzner VPS had an unblocked residential IP and the fix was a VPS cron. **That was wrong on
both counts.** The VPS test run 403'd on the *first* request — but a DB probe found 53 Tapology
shadow writes predating today (e.g. Rousey vs. Carano polled 8× through the night of May 17),
proving the VPS live tracker reaches Tapology fine. So the 403 wasn't an IP block; it was a
**rate-limit cooldown** — Tapology throttles an IP after sustained scraping, and the VPS already
runs the live tracker + 8 daily Tapology-org scrapers, so the backfill landed mid-penalty. (Home
hits the same throttle after ~7-8 sequential scrapes; it's volume-based, not datacenter-specific.)
Real root cause: `TapologyLiveScraper` had **zero retry logic** — one 403 was fatal. Fix:
  1. **403/429 retry+backoff** in `TapologyLiveScraper.fetchHtmlWithRetry()` (30/60/90s, 4 attempts)
     — benefits live tracking *and* the backfill.
  2. **Self-scheduled daily backfill inside `scraperService.ts`** — the always-on service spawns
     `backfillTapologyResults.js` at 11:00 UTC (quiet ET overnight), then every 24h. No cron to
     install (the buggy `install-tapology-cron.sh` was deleted; `run-tapology-backfill.sh` kept as
     a manual trigger). Deploy = `bash /opt/scraper-service/packages/backend/vps-update.sh` on the
     VPS (git pull + build + `systemctl restart scraper-service`). The GH Actions schedule stays
     disabled (it genuinely 403s — datacenter IP, no live-tracker history to keep it warm).
- **BKFC was a separate problem — now FIXED (see section below).** `bkfc` *is* in
  `production_scrapers` (auto-publishes), so its zero winners weren't the shadow-field bug. Turned out
  the data *is* on bkfc.com — the scraper just read the wrong element for the winner. Fixed in
  PR #5.
- **Duplicate/cancelled residue.** A few events still show null rows that are phantom-duplicate or
  cancelled artifacts (e.g. Usyk #9 "Mabrouk Yehya v Sserunkuma Kalyalya", Rizin 52 dup orderOnCard
  pairs) — pre-existing Tapology fight-bleed + legacy-import dupes, tracked under the residual
  cleanup project. Not caused by this work.

## BKFC results: winners now extracted (the "0 winners" problem) — PR #5

**Handoff:** `docs/HANDOFF-bkfc-results-2026-05-29.md` framed this as every BKFC event
COMPLETED with "0 winners / 0 methods" and a likely source-layer gap (bkfc.com maybe not
publishing results), with two candidate fixes (native scraper vs. Tapology fallback).

**The handoff's framing was wrong in a useful way.** A DB audit showed methods *were*
populated — only **winners** were null:

| Event | fights | winners | methods |
|---|---|---|---|
| Palm Desert | 17 | 0 | 10 |
| Australia 2 | 10 | 0 | 10 |
| Hawaii | 16 | 0 | 13 |

If results were truly absent, methods would be null too. They weren't — so the data is on the
page; the scraper just couldn't read the winner.

**Root cause (`services/scrapeBKFCLiveEvent.js`):** results extraction pulled
`WinMethod`/`RoundEnded`/`RoundEndedTime` from `[data-render]` (method/round/time worked), but
resolved the *winner* from (a) `RedResult`/`BlueResult` `data-render` fields that **don't exist**
on current BKFC pages, and (b) a `.fight-card_list-title` color/opacity heuristic — those rows are
stat headers (`W - L - D`, `HEIGHT`, `FIST SIZE`), not results. So `result.winner` was always null;
the parser wrote method and left winner null. Same Webflow-data-attribute trap as the earlier date
bug (`lesson_bkfc_data_event_date_local_lies`).

**Where the winner actually lives:** inside each fighter's `a.fight-card_headshot` link are four
`.fight-card_win-label` badges (win / lose / draw / no contest); only the real outcome is rendered
`display:block` (+ class `is--winner` / `is--draw`), the rest `display:none`. Each carries
`data-cond-value="win|lose|draw|no contest"`.

**Fix:** read the visible `.fight-card_win-label` badge per bout and map the winning badge to its
corner via the fighter slug. Verified against Palm Desert live DOM: **10/10** bouts resolve the
correct winner; full scraper run emits winner+method+round for all 10.

**Backlog cleared (ran `results-backfill` locally, `BACKFILL_ORGS=bkfc`, null-only, Render DB):**
**80 winners filled across all 8 completed BKFC events.** Post-fix audit: **0** events with
method-but-no-winner. (Daily `results-backfill.yml` would have caught Palm Desert via its 14-day
window, but older events were out of window — hence the one-time wide pass.)

**Shipped:** PR #5 `fix/bkfc-winner-extraction` → `main` (one-file change to
`scrapeBKFCLiveEvent.js`). Data is already corrected in prod; merging deploys the code so future
live-tracking + the daily backfill resolve winners going forward.

**Residual (not this bug):** several BKFC events have fights with neither method nor winner
(e.g. Palm Desert 17 fights but only 10 with stats) — undercard/free-fight bouts the bkfc.com stats
page doesn't render. Separate prelim-coverage gap.

## BKFC stale/superseded matchup cleanup (the "prelim gap")

Picked up `docs/HANDOFF-bkfc-stale-matchups-2026-05-29.md`. Residual after the
PR #5 winner fix: phantom fight rows on BKFC's COMPLETED events with no
winner/method — early provisional bookings whose opponents later changed, leaving
a stale row beside the real (correctly scored) bout. Not a scraping gap; the data
is on bkfc.com.

**New read-only auditor:** `backend/src/scripts/auditBKFCStaleMatchups.ts`.
Re-scrapes every COMPLETED BKFC event and diffs each non-cancelled DB row against
authoritative current pairings (mirrors `bkfcLiveParser.findFightByFighters`).
Prints every stale row with ratings/reviews/predictions/hype/comment counts so
cancel-vs-merge is decided per row. Modeled on `detectTapologyFightBleed.ts`.

**Findings (verified, Render DB):** 7 phantoms initially — exactly the handoff's
list. 3 zero-engagement (BKFC Hawaii); 4 with prediction-only engagement
(Newcastle 2) — **no ratings/reviews on any of them.** Predictions were for
matchups that never happened (unscoreable), and `FightPrediction` has
`@@unique(userId,fightId)` so a merge would collide with predictions on the real
bout. Mike's call: **cancel all.**

**8th phantom found via a blind spot.** The audit passed BKFC Clearwater as
"clean," but it had TWO "Heckert vs Walters" rows pointing at **duplicate fighter
records** — real scored bout = Heckert vs *Justi* Walters (COMPLETED); phantom
UPCOMING row = *Justin* Walters (split record, same person). Last-name equality
made the phantom's pairing look present in the scrape. (Surfaced while chasing why
`backfillResults` returned 0 for Clearwater — it matched the already-scored real
row first and never reached the duplicate.) Cancelled it too (zero engagement).

**Cleanup script:** `backend/src/scripts/cleanupBKFCStaleMatchups.ts` — cancels the
8 confirmed phantom IDs (sets `fightStatus=CANCELLED`, `completionMethod=
'stale-matchup-cleanup-2026-05-29'`). Fail-closed on `winner=null` +
`scraperType=bkfc`; predictions/comments stay attached, just stop rendering.

**Audit hardened** to catch the duplicate-fighter class: when >1 non-cancelled DB
row maps to the same scraped pairing, the unscored sibling is flagged as a
phantom. Final re-audit: **all 8 BKFC events clean, 0 stale rows.**

Commits `baabea4` (auditor), then cleanup + harden, all pushed to `home-screen`.

**Residual / follow-up:** the duplicate Justi/Justin Walters fighter records still
