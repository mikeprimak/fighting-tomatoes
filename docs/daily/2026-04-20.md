### Handled account deletion request (thecodynelson@gmail.com)
User emailed asking to delete any account/info associated with that address. Account existed — displayName was literally "delete my account", created 2026-02-14, 34 fight ratings.

Followed the in-app deletion protocol from `AuthController.deleteAccount` (`packages/backend/src/controllers/authController.ts:818`): strip PII and auth identifiers, keep rating/review content attributed to anonymous "Deleted User".

**What was stripped:** email (→ `deleted_<ts>_<rand>@deleted.local`), password, firstName, lastName, displayName (→ "Deleted User"), avatar, googleId, appleId, email/password reset tokens, pushToken, media fields. `isActive=false`, `wantsEmails=false`, `isEmailVerified=false`. All refresh tokens deleted. Notification rules/matches/notifications wiped.

**What was preserved (anonymized):** 34 fight ratings. No reviews, predictions, or pre-fight comments.

### New script: `packages/backend/src/scripts/deleteUserByEmail.ts`
Reusable CLI for future deletion requests. Mirrors the controller exactly so in-app and manual deletions behave identically.

```bash
cd packages/backend && npx tsx src/scripts/deleteUserByEmail.ts <email>
```

Runs against whatever `DATABASE_URL` is in `.env` — currently the Render External URL.

See `docs/areas/support-tooling.md` § Account Deletion for the standing process.

### Marketing demo: pin UFC 328 at top of Upcoming for `avocadomike@hotmail.com`
Needed a clean screen-capture for a promo video where UFC 328: Chimaev vs. Strickland (2026-05-09) appears as the first event on the Upcoming tab, ahead of sooner-dated cards. Scoped the override to one account so no other user is affected.

**Change:** `packages/backend/src/routes/index.ts`, inside the `type === 'upcoming'` branch of `/api/events`. When the authenticated request's `user.email === 'avocadomike@hotmail.com'`, the hardcoded event ID `b992560c-fe62-417b-8953-e3323cdf7b2a` is moved to position 0 of the sorted lite list **before** pagination, so it lands on page 1 of the infinite-scroll feed. The pin silently no-ops if the event isn't in the result set (e.g., after it completes or is hidden).

**Why pin by ID instead of name:** name matching would break if scrapers rename the event (e.g., to "UFC 328: Chimaev vs. Someone Else" after a card change). Event ID is stable.
