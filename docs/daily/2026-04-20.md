### Fix "index" header flash on app launch
Noticed the word "index" briefly appearing at the top of the screen on app open before the home screen loaded. Root cause: `app/(tabs)/index.tsx` is a redirect-only route that renders an `ActivityIndicator` while `useHasLiveEventWithLoading` resolves, then `<Redirect>`s to `/live-events` or `/events`. The tabs navigator rendered its header during that loading window, and because the `Tabs.Screen` for `index` set `href: null` but no `headerTitle`, the header defaulted to the route name — literally `"index"`.

**Change:** `packages/mobile/components/TabBar.tsx:163` — added `headerShown: false` to the `index` screen options.

JS-only, so it can ship via `eas update --branch production` without a new native build.

### Index-flash fix, round 2 + OTA ship (session 2)
User reported the flash was still happening. Two things:

1. The `headerShown: false` change in `81b7940` was **committed but never deployed** — the installed production app was still on the pre-fix bundle. My own follow-up to run `eas update` never got checked off.
2. Even with `headerShown: false`, the fix relied on `Tabs.Screen` options being applied before the first render frame. If anything short-circuited that (cold-start timing, a transition frame) the route name `"index"` could still leak through as a fallback title. Added belt-and-suspenders.

**Changes (commit `578e3f2`):**
- `packages/mobile/app/(tabs)/index.tsx` — return `null` while `useHasLiveEventWithLoading` resolves, instead of mounting a full-screen `<View>` with an `ActivityIndicator`. The redirect-only route now never draws its own body.
- `packages/mobile/components/TabBar.tsx:163` — on the `index` `Tabs.Screen`, in addition to `headerShown: false` also set `title: ''`, `headerTitle: ''`, and `tabBarStyle: { display: 'none' }`. Every surface that could fall back to the route name now renders empty.

**Shipped via OTA:**

