## What I Worked On
- [x] Admin panel Events filter overhaul: replace Current/Upcoming/All with Live/Upcoming/Past matching the app's filter style (area: backend)

## Changes Made
### Backend: `/admin/events` filter swap (`status` query param)
- `packages/backend/src/routes/admin.ts` — replaced the `upcoming` + `current` boolean params with a single `status` enum: `live` | `upcoming` | `past`. Filters on `eventStatus` directly (LIVE/UPCOMING/COMPLETED) and applies the right sort:
  - `live` → `date` desc (most recently started first)
  - `upcoming` → `date` asc (next to start first)
  - `past` → `updatedAt` desc (most recently finished first — `updatedAt` reflects the lifecycle's flip to COMPLETED)

### Frontend: admin panel Events filter dropdown
- `packages/backend/public/admin.html`
  - Filter dropdown now `Live / Upcoming / Past`. Default selection is **Upcoming**.
  - `loadEvents()` sends `status=<value>` instead of the old `upcoming=true` / `current=true`.
  - `checkMissingStartTimes()` updated to query `status=upcoming`.

## Why
Old filters were date-window based ("yesterday → 2 days from now") and didn't always line up with what's actually live or what the user is looking for. Switching to the app's existing eventStatus-driven Live/Upcoming/Past taxonomy gives a single mental model across the user-facing app and admin tool, and the new sort orders ("most recently started" / "next to start" / "most recently finished") make it easy to grab the relevant event without scrolling.
