### Live parser audit — none have the Bug B pattern

Bug B (UFC live tracker un-cancelling a duplicate by `ufcFightId` alone) was UFC-specific. The other live parsers were flagged for an audit because their cancellation/un-cancellation logic was untouched by the 2026-04-25 fix. Read each:

### Bug A — fighter-rename forking (PR #1)

Schema migration + backfill, on branch `fix/ufc-fighter-slug-upsert`. PR: https://github.com/mikeprimak/fighting-tomatoes/pull/1

**Root cause recap:** `Fighter` upsert was keyed on the `firstName_lastName` composite. When UFC corrected "Juan Martinetti" → "Adrian Luna Martinetti" on 2026-04-16, the parser created a brand-new fighter row instead of updating the original, then forked a new fight (different `fighter2Id`).

**Fix:**
- Add `Fighter.ufcAthleteSlug String? @unique`. Nullable so non-UFC fighters and rows without a scrape URL are unaffected; multiple NULLs allowed under PG `@unique`.
- `importFighters` now upserts by slug when `athlete.url` yields one (with `firstName`/`lastName` in the `update` block so renames propagate). Falls back to the existing name composite when no URL.
- Exported `extractUfcAthleteSlug` and `parseFighterName` so the backfill can reuse them.
- New `src/scripts/backfillFighterSlug.ts` — reads `latest-athletes.json`, matches existing rows by current `(firstName, lastName)`, sets `ufcAthleteSlug` when null. Idempotent. Skips collisions with a warning.

**Trade-off (worth knowing):** the slug isn't strictly more stable than the name — UFC's athlete URL is name-derived, so a rename could in theory rename the URL too. But in practice, slug collisions on rename are rare (the rename site would also need to coincide with a CMS slug change), and even when both change at the same time the orphan flow only happens once per fighter. This is the right move per the existing decision in the daily doc; the alternative (use a stable internal UFC athlete ID) would require a deeper scraper change to surface that ID.

## Backend: results-backfill orchestrator + UFC wrapper shipped (Phase 2 first cut)

Resumed from the Phase 2 plan section above. Shipped the orchestrator, the UFC wrapper, and the daily workflow. Commit on `main`, pushed.

### What landed (commit `371559a`)

- `packages/backend/src/scripts/backfillResults.ts` — daily orchestrator. Queries events with `eventStatus='COMPLETED'`, `date` within `BACKFILL_WINDOW_DAYS` (default 14), at least one fight with `winner=NULL` and `fightStatus IN (COMPLETED, UPCOMING, LIVE)`, and dispatches by `scraperType`. Optional `BACKFILL_ORGS` env var for CSV filtering. Per-org tally summary at the end. Non-zero exit on any per-event failure so the workflow's failure step alerts.
- `packages/backend/src/services/backfillUFCResults.ts` — UFC wrapper. Reuses `scrapeLiveEvent.js` + `parseLiveEventData` exclusively (single source of truth per org). Counts NULL winners before/after for the touched event so it can report `filledWinners` count.
- `packages/backend/src/services/ufcLiveParser.ts` — added `ParseLiveEventOptions` parameter (third arg, default `{}`). Four opt-in flags, all defaulting to existing live-tracker behavior:
  - `nullOnlyResults` — gates winner/method/round/time writes on the DB value being NULL. Honors the safety contract ("never overwrite manual fixes or live-tracker results").
  - `skipCancellationCheck` — bypasses the CANCELLED↔UPCOMING reconciliation pass. Backfill runs days after the event; the source page may have shifted and we don't want to retroactively cancel real fights.
  - `skipNotifications` — suppresses the next-fight push notification when a fight flips to COMPLETED. Users shouldn't be paged about a 2-week-old fight.
  - `completionMethodOverride` — when set (e.g. `'backfill-ufc'`), stamps it onto `Fight.completionMethod` plus `Fight.completedAt = now()` for any fight whose status flips to COMPLETED on this run. Audit trail.
- `.github/workflows/results-backfill.yml` — daily cron at 10:00 UTC (1hr offset from `tapology-backfill.yml` at 09:00 UTC, so the two jobs don't fight for Render DB connections or Puppeteer memory). `workflow_dispatch` inputs for `window_days` and `orgs` filter.

### Design notes worth keeping

- **Why modify the parser instead of writing a parallel one**: the architecture spec says "single source of truth per org" — duplicating the UFC fighter-matching/winner-resolution logic into a separate backfill parser would be the wrong move. Adding four opt-in flags with defaults that preserve existing live-tracker behavior keeps the live tracker hot path untouched while letting backfill honor the safety contract.
- **Why audit trail lives at the fight level, not the event level**: `Event.completionMethod` records *how the event was marked complete* (`scraper`, `lifecycle-no-tracker`, etc.); the event was already COMPLETED before backfill ran, so overwriting it would be misleading. `Fight.completionMethod` is the right place to record where each fight's result data came from.
- **Why orchestrator does its own NULL-winners count (not relying on parser return value)**: `parseLiveEventData` returns `void` in the live tracker for historical reasons. Counting before/after via Prisma is cheap (~ms per event), gives the orchestrator a number to surface in logs, and avoids changing the parser's return contract.
- **Window default of 14 days vs Tapology's 7**: Tapology posts results within a day or two; UFC.com sometimes takes longer to fully stabilize event archive pages. 14 covers the long tail without much extra load (most events will have all-NULL or all-filled fights and be a no-op past a certain point).

Switching to the app's existing eventStatus-driven Live/Upcoming/Past taxonomy gives a single mental model across the user-facing app and admin tool, and the new sort orders ("most recently started" / "next to start" / "most recently finished") make it easy to grab the relevant event without scrolling.
