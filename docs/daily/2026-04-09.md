### Scrapers: RAF scraper wrongly marked future events COMPLETED
- **Symptom**: RAF 09 (May 30, 2026) showed up on the "past events" screen in the mobile app despite being ~7 weeks in the future. Database showed `eventStatus: COMPLETED, completionMethod: null`. RAF 08 (Apr 18, 2026) was correctly UPCOMING, so the bug was event-specific.
- **Root cause**: `scrapeAllRAFData.js:123` determined status from the gallery card buttons: `hasBuyTickets ? 'Upcoming' : 'Complete'`. RAF 09 tickets aren't on sale yet, so no "buy tickets" button exists, and the scraper reported it as Complete. The event-page scraper does extract a more reliable `isPastEvent` signal from `div.past-event-tag` (the site's own CMS flag), but line 448 only used it to upgrade Upcomingâ†’Complete, never to correct Completeâ†’Upcoming. And `rafDataParser.ts:225` leaves `eventStatus` untouched on the update path, so once an event was imported as COMPLETED it stayed that way forever.
- **Fix**: Dropped the button heuristic. Gallery scrape now defaults to `'Upcoming'`. Event-page scrape trusts `isPastEvent` as authoritative in both directions. Added a `fallbackStatusFromDate()` helper that marks an event Complete only if `eventDate + 24h < now` â€” used only when the event page scrape fails (previously the fallback was the meaningless `galleryEvent.status`). The 24-hour buffer avoids the midnight-date problem where `eventDate` is stored at 00:00 UTC and would otherwise flip to Complete before the event starts.
- **DB cleanup**: Manually reset RAF 09 to `UPCOMING` / `completionMethod: null` via Prisma. Queried all promotions for other future-dated events marked COMPLETED â€” none found, RAF 09 was the only stale record.

## Notes
- Individual PFL event pages (`/event/{slug}`) drop the year entirely (`FRI APR 10`, `SAT MAR 28`) â€” not useful as a fallback for date verification. The list page is the only place with canonical dates.
- The GitHub Actions workflow chains `node scrapeAllPFLData.js` with `node -e "...importPFLData()..."` on the same line. Because bash stops after the first command fails, a scraper crash also prevents the import â€” even if events were successfully scraped and saved.
- **RAF status detection lesson**: "Absence of a ticket-purchase button" is not a reliable signal of event completion. Three states can produce no button: already happened, not yet on sale, sold out. For any scraper that determines event state from page elements, prefer (a) an explicit CMS flag from the source, (b) date comparison with a buffer, in that order â€” never absence-of-button.
- **`rafDataParser.ts` update path is write-once on status**: line 225 deliberately logs "status unchanged" and doesn't touch `eventStatus` on subsequent scrapes. This means data fixes for wrongly-imported events have to happen at the DB level, not by re-running the scraper. Worth keeping in mind if similar bugs surface in other parsers.
