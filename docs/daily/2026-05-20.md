## BKFC enrichment fix — shipped

Mike asked to fix the BKFC 0% gap immediately after the audit, so picked it up.

### Root cause

`enrichOneEvent.ts:90-112` dispatch only branched on `ufc.com` or `tapology.com` URLs. Every BKFC event in the DB has `ufcUrl` pointing to `bkfc.com/events/<slug>`, which matched neither branch — so BKFC events ran editorial-only. The editorial layer's allowlist was MMA-flavored (mmafighting, mmajunkie, bloodyelbow, sherdog, espn, mmamania, mmaweekly, cbssports, bjpenn), none of which reliably cover BKFC cards. Result: every BKFC run produced 0/N narrative.

### Changes

**`packages/backend/src/services/aiEnrichment/fetchBkfcEventPreview.ts` (new)** — mirrors `fetchTapologyEventPreview.ts` pattern. Plain fetch + cheerio (bkfc.com is Webflow-hosted, no JA3 protection — probed with curl, HTTP 200 in 330ms). Extracts `main.main-wrapper` then strips noise (merchandise, newsletter blocks). Smoke-tested against BKFC 90 Birmingham + BKFC Palm Desert: ~5800 chars of usable content per event including full card with weight classes / championship flags / fighter records and the "event spotlight" editorial blurb.

**`packages/backend/src/services/aiEnrichment/enrichOneEvent.ts`** — added `isBkfcCom` branch alongside `isUfcCom` / `isTapology`. Same shape as the Tapology branch (no Puppeteer needed).

**`packages/backend/src/services/aiEnrichment/fetchEditorialPreviews.ts`** — clarified the role of the allowlist (third-party perspective, NOT promotional copy — bkfc.com / ufc.com are pulled as the backbone, not via Brave editorial). Mike's call after seeing the bkfc.com text: it's promoter marketing voice and double-counting it via editorial would amplify hype-flavored language. Added objective outlets more likely to cover BKFC and other low-coverage promotions: `lowkickmma.com`, `boxingscene.com`, `talksport.com` (UK angle for Darren Till crossover stories), `dailymail.co.uk` (UK mainstream), `bareknuckle.tv` (third-party niche). Dropped `bkfc.com` from the editorial allowlist (still pulled as backbone).

## Follow-fighter notification lanes — shipped

While the other window was working AI enrichment, this session built the three follow-fighter notification lanes the walkout dispatcher couldn't reach: **Booked** (when a followed fighter's fight is freshly scraped onto the schedule), **3-day-warn** (10am user-local, 3 days before fight-day), and **morning-of** (9am user-local on fight-day with overnight rollback). Walkout now gated by the same toggle. Mike confirmed two design calls up-front: per-user local timezone (with overnight rollback for 3am ONE FC / RIZIN fights) and ship engagement tracking alongside (doc-mandated per workstream §4).

### Backend
- `User.timezone` (IANA, default `America/New_York`), captured from `Intl.DateTimeFormat().resolvedOptions().timeZone` inside `notificationService.registerPushToken()` — once per session, only PUTs when changed.
- 4 user toggles + 3 per-lane sent timestamps on `FightNotificationMatch` for cross-tick idempotency.
- New `FollowNotificationEvent` table — one row per dispatch (`matchId`, `userId`, `fightId`, `fighterId`, `lane`, `dispatchedAt`, `openedAt?`, `clickedAt?`). Open/click attribution is mobile telemetry callback work, not built yet.
- New service `services/followFighterNotifications.ts`:
  - `computeFightDay(utc, tz)` — overnight rollback: fight starts before noon local → previous local day. Handles 3am ONE FC, daytime UFC prelims, RIZIN.
  - `localWallClockToUTC(y,m,d,h,tz)` — two-step Intl correction. Accurate at 9am/10am (well clear of DST transitions).
  - `dispatchBookedNotification(...)` called from the scrape-time sync.
  - `runFollowFighterCron()` registered in `notificationScheduler.ts` on `*/15 * * * *`. Scans pending matches, computes per-user triggers, fires within a 30-min window.
- `syncFighterFollowMatchesForFight` now distinguishes new-row creation from re-sync. Booked dispatches only when `Fight.createdAt > UserFighterFollow.createdAt` — retroactive follows do NOT get bombarded with "booked!" notifications for the fighter's existing upcoming fights. Clean semantics.
- `notifyFightStartViaRules` + `notifyEventSectionStart` gated by `notifyFollowedWalkout`. If the user's only reason to be notified is a Fighter Follow rule AND walkout is off, skip the push but still mark the match row sent so the moment isn't replayed if they later flip it on. Manual / hype / pre-event matches bypass the toggle.
