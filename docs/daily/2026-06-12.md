## Verification soft-cap SHIPPED (same day, Mike-approved 50/50)

Unverified accounts may make up to 50 ratings and 50 hype predictions, then
verification is forced — capture behavior first, obstacle after investment.

- **`requireVerifiedOrUnderCap(kind)`** in `middleware/auth.ts`: verified →
  pass (zero queries); unverified with an existing row on THIS fight → pass
  (updates/removals never blocked); else count rows, ≥cap → 403
  **`VERIFICATION_CAP_REACHED`** + `details {kind, cap, count}` (distinct
  from EMAIL_NOT_VERIFIED). `UNVERIFIED_ACTION_CAP` env override for tests.
- **Applied to:** POST `/fights/:id/rate` (rating), POST
  `/fights/:id/prediction` (hype — this UNGATES hype, previously a hard
  wall), PUT `/fights/:id/user-data` **split gate**: rating-only body →
  rating cap (the rate modal's commit path); body with review/tags → hard
  EMAIL_NOT_VERIFIED stays (public content).
- **Mobile:** RateFightModal + PredictionModal onError show "Verify your
  email" + the server's message on the cap code; UpcomingFightModal's
  otherwise-silent flame-tap hype mutation now Alert.alerts on the cap (the
  hype would just vanish otherwise).
- **Verified live (cap=3 via env, fresh unverified tester):** ratings 1-3 →
  201, 4th → VERIFICATION_CAP_REACHED, re-rate at cap → 201; identical
  matrix for hype; user-data rating-update at cap → 200, new-fight → 403,
  review body → EMAIL_NOT_VERIFIED; verified account uncapped. Tester reset
  after (reset script now also deletes the account's hype predictions —
  they count against the cap; raw delete is safe, Fight stores no hype
  aggregates).
- Gates: backend production tsc 0; mobile 0 new.
