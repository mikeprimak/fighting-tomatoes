# 2026-04-29

## Backend: daily UFC scraper unblocked (slug-upsert P2002)

Daily UFC scraper had been failing for two consecutive days (Apr 28 18:11 UTC and Apr 29 18:06 UTC) on a `P2002` against `(firstName, lastName)`. First failed run was the very first scheduled run after `47c74a6` (slug-based upsert) shipped on Apr 28. Today's run halted at the first imported athlete (Jack Della Maddalena) with the whole import aborting from a single `prisma.fighter.upsert` throw.

### Root cause

`47c74a6`'s upsert pattern in `importFighters`:
- `where: { ufcAthleteSlug: slug }` — if no row has that slug, fall through to `create`.
- `create: { ufcAthleteSlug, firstName, lastName, ... }` — bombs with `P2002` when a row with that `(firstName, lastName)` already exists in DB without (or with a different) slug.

The Apr 28 backfill (`backfillFighterSlug.ts`) tagged 149/160 of the latest scrape's athletes, but anyone NOT in that scrape's snapshot stayed slug-less — and any future scrape that surfaces a fighter under a slug we hadn't tagged that row with takes the `create` path and collides with the existing name-keyed row.

There's a second collision mode the original code also couldn't handle: a slug-tagged row whose name no longer matches the fighter at that URL (the Mitch Raposo → `aleks-oliveyra-2` pattern noted in 04-28's daily log). In that case the upsert would `update`, but renaming the row collides with an actual `Aleks Oliveyra` row. We didn't trip this today, but it's a latent crash on the same code path.

### Fix (commit `6a44445`)

Replaced the upsert with an explicit find-first flow in `importFighters`:

1. **Find by slug.** If found → `update` by id with new fields.
2. **Else find by `firstName_lastName`.** If found → `update` by id and **claim the slug onto it** (adopt). This is the case the original upsert couldn't express.
3. **Else create.**

Both update branches catch `P2002`:
- Rename via slug-tagged row collides with an existing name → log warning, redo update without the rename (refresh non-name fields only), import continues.
- Slug claim onto a name-tagged row collides because another row already owns that slug → log warning, redo update without the slug claim, import continues.

Net effect: one bad fighter no longer halts the whole batch; warnings surface in workflow logs so we can audit name/slug drift later without losing a daily run.
