## BACKLOG §3 + §4 — Profile images (durable R2 + mobile picker + web persist)

Branch `backlog/round-3-profile-images` → merged to `main` (merge in `d478af67`).

**Root cause of the web "broken image" bug.** `POST /upload/profile-image` wrote to
Render's **ephemeral local disk** (`public/uploads/profiles/`) and returned a **relative**
URL (`/uploads/profiles/<file>`). The browser resolved that against goodfights.app → 404 →
broken image. Two further problems: the file vanished on the next Render redeploy, and the
web edit page never **persisted** the avatar (only `setUser(...)` in memory).

**Fixes (one root fix + two clients):**
- **Backend** (`routes/upload.ts`) — profile uploads now go to **R2** via
  `uploadLocalFileToR2` and return the **absolute** public URL (durable, renders as-is on
  every client). Local-disk path kept as a dev-only fallback, now built absolute from the
  request host. `/upload/crew-image` left untouched (out of scope; its mobile client
  re-absolutizes the relative URL, so it'd double-prepend an absolute one).
- **Web** (`lib/api.ts` + `app/profile/edit/page.tsx`) — `updateProfile` accepts `avatar`;
  the edit page now persists the uploaded avatar to the user record + shows an uploading
  spinner + resets the file input so re-selecting the same file works.
- **Mobile** (`app/edit-profile.tsx`) — added the **missing** avatar UI: tap-to-pick
  (`expo-image-picker`, already shipped in the native build via `crew/info/[id].tsx` →
  **OTA-safe**), upload → `updateProfile({avatar})` → `refreshUserData()`, spinner state,
  initial-letter fallback. Screen retitled "Edit Display Name" → "Edit Profile".

Avatar URLs are now absolute everywhere — consistent with existing Google sign-in avatars
(which were already absolute `picture` URLs), and confirmed no client re-absolutizes
`avatar`, so nothing breaks.

**Verification:** backend prod tsconfig exit 0; web `tsc` exit 0; mobile 82 errors (under
the 84 baseline, none in the touched file). Not yet exercised live.

## BACKLOG §3 — Android notification tray icon (the "switched back" mystery solved)

Mike recalled fixing this before and it inexplicably reverting. **Root cause:** the
bare `android/` dir ships **committed** `android/app/src/main/res/drawable-*/notification_icon.png`
(5 densities). Those override `app.json`'s `expo-notifications.icon` — prebuild regenerates
them from the managed asset, but a bare-`android` `eas build` **skips prebuild**, so whatever
drawables are committed win. A past prebuild set them correctly; a later commit/build carried
the old ones back. Both the drawables and the managed `assets/notification-icon.png` were a
227KB full-color raster, which Android masks to a solid white blob.

**Fix** (`f112c5c1`): regenerated all 5 density drawables (24/36/48/72/96px) **and** the 96px
managed asset from Mike's new `THICKER-GOOD-FIGHTS-LOGO-WHITE-ALPHA.png` (1513² white-on-
transparent glove) using `sharp` (resolved from the backend dep) — `.trim()` to drop the
source's transparent margin, then scaled to ~86% fill and centered on a transparent square.
Fixing **both** paths means it won't "switch back" whether a future build uses the committed
drawables or a clean prebuild. Verified on a dark-bg composite: clean white glove filling the
frame (vs. the old blob).

**⚠️ Needs a new Android build to reach users** — native drawables are NOT OTA-able, and the
vc39/2.1.2 build already in Play Console review predates this. Either fold it into the next
Android build or cut a vc40 when convenient (don't burn a build on the icon alone).

## BACKLOG §6 — UFC scraper now captures betting odds

Mike: ufc.com posts odds next to fighter names (e.g. the June 27 card — Fiziev -120,
Torres +100) → wire them into the daily scraper. **Root finding: the scraper was already
capturing odds, the importer was throwing them away.**

- `scrapeAllUFCData.js` reads `.c-listing-fight__odds-amount` into
  `fighterA.odds`/`fighterB.odds` (both card structures, lines ~452 + ~571) and writes them
  to `scraped-data/*.json`. Confirmed in a real JSON (Jan 2026): values like `+100`/`-120`,
  plus a `"-"` **no-line placeholder** (92 of them) for fights without posted odds.
- `ufcDataParser.ts` builds the `upsertFightSwapAware` create/update payloads but **never
  included `fighter1Odds`/`fighter2Odds`** — so they stayed null in the DB.

**Fix** (`ufcDataParser.ts`): added `normalizeOdds()` (accepts only `^[+-]\d+$`; turns the
`"-"` placeholder / empty / junk into null) and spread an `oddsData` object into both the
create and update payloads. Two deliberate choices:
- **Only write odds when present** — a later scrape that shows `"-"` (lines often pulled
  close to fight night) must not wipe a previously captured line.
- **Corner alignment is safe** — odds are keyed to the *current scrape's* A/B corners, and
  `upsertFightSwapAware` canonicalizes `fighter1Id`/`fighter2Id` to that same scrape's order
  in the same `update`, so `fighter1Odds` always stays paired with `fighter1Id`.

Confirmed via WebFetch that the linked June 27 page still exposes odds (Fiziev -120 /
Torres +100, etc.). Backend prod `tsc` exit 0.

**Scope note:** UFC only this pass. Other daily scrapers (BKFC / the Tapology-backed orgs)
may also expose odds on their source pages — a follow-up. The old "wire The Odds API
consensus" plan is now optional (only for sources that lack odds, or to add a consensus line
on top of UFC's). No odds-on-card UI yet — `fighter1Odds`/`fighter2Odds` populate silently
for now.
