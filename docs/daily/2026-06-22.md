## BACKLOG §3 — Android notification tray icon (the "switched back" mystery solved)

Mike recalled fixing this before and it inexplicably reverting. **Root cause:** the
bare `android/` dir ships **committed** `android/app/src/main/res/drawable-*/notification_icon.png`
(5 densities). Those override `app.json`'s `expo-notifications.icon` — prebuild regenerates
them from the managed asset, but a bare-`android` `eas build` **skips prebuild**, so whatever
drawables are committed win. A past prebuild set them correctly; a later commit/build carried
the old ones back. Both the drawables and the managed `assets/notification-icon.png` were a
227KB full-color raster, which Android masks to a solid white blob.

**Fix** (`f112c5c1`): regenerated all 5 density drawables (24/36/48/72/96px) **and** the 96px
managed asset from Mike's new `THICKER-GOOD-FIGHTS-LOGO-WHITE-ALPHA.png` (1513² white-on-
transparent glove) using `sharp` (resolved from the backend dep) — `.trim()` to drop the
source's transparent margin, then scaled to ~86% fill and centered on a transparent square.
Fixing **both** paths means it won't "switch back" whether a future build uses the committed
drawables or a clean prebuild. Verified on a dark-bg composite: clean white glove filling the
frame (vs. the old blob).

**⚠️ Needs a new Android build to reach users** — native drawables are NOT OTA-able, and the
vc39/2.1.2 build already in Play Console review predates this. Either fold it into the next
Android build or cut a vc40 when convenient (don't burn a build on the icon alone).

## BACKLOG §6 — UFC scraper now captures betting odds

Mike: ufc.com posts odds next to fighter names (e.g. the June 27 card — Fiziev -120,
Torres +100) → wire them into the daily scraper. **Root finding: the scraper was already
capturing odds, the importer was throwing them away.**

- `scrapeAllUFCData.js` reads `.c-listing-fight__odds-amount` into
  `fighterA.odds`/`fighterB.odds` (both card structures, lines ~452 + ~571) and writes them
  to `scraped-data/*.json`. Confirmed in a real JSON (Jan 2026): values like `+100`/`-120`,
  plus a `"-"` **no-line placeholder** (92 of them) for fights without posted odds.
- `ufcDataParser.ts` builds the `upsertFightSwapAware` create/update payloads but **never
  included `fighter1Odds`/`fighter2Odds`** — so they stayed null in the DB.

**Fix** (`ufcDataParser.ts`): added `normalizeOdds()` (accepts only `^[+-]\d+$`; turns the
`"-"` placeholder / empty / junk into null) and spread an `oddsData` object into both the
create and update payloads. Two deliberate choices:
- **Only write odds when present** — a later scrape that shows `"-"` (lines often pulled
  close to fight night) must not wipe a previously captured line.
- **Corner alignment is safe** — odds are keyed to the *current scrape's* A/B corners, and
  `upsertFightSwapAware` canonicalizes `fighter1Id`/`fighter2Id` to that same scrape's order
  in the same `update`, so `fighter1Odds` always stays paired with `fighter1Id`.

Confirmed via WebFetch that the linked June 27 page still exposes odds (Fiziev -120 /
Torres +100, etc.). Backend prod `tsc` exit 0.

**Scope note:** UFC only this pass. Other daily scrapers (BKFC / the Tapology-backed orgs)
may also expose odds on their source pages — a follow-up. The old "wire The Odds API
consensus" plan is now optional (only for sources that lack odds, or to add a consensus line
on top of UFC's). No odds-on-card UI yet — `fighter1Odds`/`fighter2Odds` populate silently
for now.

**Verified live end-to-end:** pushed to `main`, dispatched `ufc-scraper.yml` (run
27986186414, success), then queried prod. **UFC Fight Night Fiziev vs. Torres (Jun 27)** now

## Fighter headshot auto-refresh on belt/status changes (BACKLOG §line 71) — UFC

**Problem (Mike):** headshots don't update automatically. After UFC Freedom 250 (Pereira/Topuria
lost belts; Gane/others gained status) ufc.com swapped in new portraits, but the app kept the
old ones. Root cause: `backfillUFCHeadshots.ts` is **null-only** — it never touches a fighter
who already has a `profileImage`, so a fighter whose photo *changed* is invisible to it.

**Fix — a separate overwrite-capable daily refresh scoped to who just fought:**
- `packages/backend/src/scripts/refreshRecentUFCHeadshots.ts`
- `.github/workflows/ufc-headshot-refresh.yml` (cron `0 14 * * *`, separated from the 17:00 UTC
  daily UFC scrape so we don't hammer ufc.com's CDN concurrently; also `workflow_dispatch` with
  window/limit/dry_run inputs).

**Design:**
- **No schema migration** (prod-DB rule). "Recently fought" is derived: fighters joined to an
  event with `eventStatus='COMPLETED'` + `scraperType='ufc'` + `COALESCE(mainStartTime,date)`
  within a rolling window (`REFRESH_HEADSHOT_WINDOW_DAYS`, default 10).
- **Overwrite on change, for free.** `imageStorage` keys each R2 object on `md5(sourceUrl)`.
  When UFC publishes a new portrait it lives at a new asset path → `uploadFighterImage()` yields
  a **new** R2 url → differs from the stored one → we write. Unchanged photo → `imageExists`
  short-circuits → identical url → skip. So daily re-runs over the same window are cheap: one
  page fetch per fighter, an R2 download *only* when the photo actually changed.
- **Safety rails:** aborts if R2 unconfigured (else `uploadFighterImage` falls back to the raw
  ufc.com url and would *downgrade* a hosted image — strictly worse); never replaces a real
  photo with a silhouette/placeholder; keeps the og:title trust-check (ufc.com redirects unknown
  slugs to other fighters' pages); optimistic-concurrency `updateMany WHERE profileImage = <read
  value>` so a concurrent backfill/admin edit isn't clobbered; P2002 slug-collision retry
  (image-only) like the backfill.

**Verified end-to-end (dry-run, LIMIT=2, window 14d, R2 configured locally):**
- Candidate query returns 43 fighters over 14d — incl. **Ciryl Gane**, Sean O'Malley, Michael
  Chandler (the exact belt/status-change population).
- **Derrick Lewis**: stored `derrick-lewis-08cfacca.png` → ufc.com now serves
  `LEWIS_DERRICK_06-14.png` → new key `derrick-lewis-bc0e7dc1.png` → correctly flagged **changed**.
  This is precisely the case the null-only backfill skips.
- **Bia Mesquita**: same — `cbb0f2af` → `6447c83c`. Both "updated", 0 errors, trust-check passed,
  slugs resolved from the existing column.
- Returned R2 urls use the same `pub-*.r2.dev` domain as the stored values, so unchanged photos
  compare equal (no accidental mass-rewrite). Backend prod `tsc` exit 0 (0 errors).

**Scope:** UFC only — it's the only org with a headshot fetcher (`scrapeUFCAthleteHeadshot.ts`).
Other orgs deferred. The workflow is scheduled daily; it can also be run once now via
`workflow_dispatch` to immediately refresh the current window if desired.

## BACKLOG §6 — Harden the other 12 unguarded importers

Follow-up to the DBC fix (`cf661d63`, 2026-06-21), which guarded one importer and flagged
12 others sharing the same unguarded `fs.readFile(eventsFilePath)`. When a scraper writes no
events file, that read throws `ENOENT` → the importer `exit(1)`s → the GH Actions run fails →
the admin gets paged. For a promotion that *legitimately* has zero upcoming events that's a
**false alarm**; for a major that always has a full slate, zero events is a **real signal**
(scrape broke). So per the BACKLOG note this was decided **per-tier**, not a blanket sweep.

**Tiering applied (branch `backlog/round-6-harden-importers` → merged `c0ee3bef`):**
- **SKIP tier — graceful return** (log + `return`, matching the existing
  karateCombat/raf/dirtyBoxing idiom): `mvp`, `gamebred`, `matchroom`, `oktagon`,
  `goldenBoy`, `goldStar`, `rizin`, `topRank`, and **`pfl`**. These are small / intermittent /
  shelved promos, or (PFL) **seasonal** with genuine off-season gaps, so an empty upcoming
  slate is normal and must not page.
- **ALERT tier — clear, actionable thrown error** (replaces the raw `ENOENT`; still fails the
  run + pages, *intended*): **`ufc`, `bkfc`, `oneFC`**. These run year-round at high frequency
  and should never legitimately have an empty slate, so a missing events file almost always
  means the **scrape failed** (bot-block, Cloudflare, selector drift). The new message names
  the org + path and says "this almost always means the scrape FAILED, not an empty slate —

`tsc --project tsconfig.production.json --noEmit` exit 0. Backend-only → Render auto-deploys

## BACKLOG §8 — Legacy event-order inversion: read-only audit (phase 1)

Many legacy (`scraperType=null`) events have **inverted `orderOnCard`** (main event at the
HIGHEST order instead of 1) so they render upside-down. Scope is NOT uniform — a blanket
flip would break the already-correct ones (memory `lesson_legacy_event_order_inversion`).
The backlog calls for a **read-only audit first**, then fix only the genuinely-inverted.

**Built** `packages/backend/scripts/audit-legacy-event-order.ts` (read-only; writes nothing
to the DB). The key improvement over a naive rating heuristic: an **authoritative name-match
signal** — most UFC cards are *named after their main event* ("FN Holm vs Shevchenko"), so if
the event name contains the highest-order fight's fighters but not the lowest-order fight's,
the card is inverted. Combined with `isTitle` (authoritative but sparse in legacy rows) and a
weak rating tiebreak. Dropped a `rounds` signal — legacy fights all default to
`scheduledRounds=3`, so it carries no info.

**Results** (1241 legacy COMPLETED events; 1130 with ≥3 fights scored), report in
`docs/audits/legacy-order-audit.{md,csv}`:
- **211 `INVERTED-HIGH`** (name + rating agree) + **14 `INVERTED-STRUCTURAL`** → **225 safe-to-fix**
- **325 `INVERTED-RATING-ONLY`** → weak signal only; need ufcstats/Wikipedia before any flip
- **30 `SENTINEL`** (out-of-range orders like 99/115) → manual
- **341 already-correct** (212 structural + 129 rating) ≈ the memory's "~328 correct"
- 209 `UNCERTAIN`

**Validated** by querying 3 `INVERTED-HIGH` events directly: in each the named main event sits
at the highest order (Kattar vs Chikadze @10, Dos Anjos vs Fiziev @11, Blaydes vs Aspinall @14)
with order 1 an early prelim. Logic confirmed.

**Phase 2 (the fix) deliberately NOT applied.** Reordering 225 historic user-facing events on
prod is a large, outward-facing mutation — holding for Mike's explicit go-ahead. The transform
is self-inverse (`newOrder = (minOrder + maxOrder) − oldOrder`); after applying, re-running the
audit should flip the targeted events to `CORRECT`. The 325 rating-only + 30 sentinel want an
authoritative external source, not the heuristic. Backlog §8 first item marked `[~]` (phase 1 done).

### §8 — Legacy event-order fix applied (phase 2)

Mike gave the go-ahead, so applied the fix to the 225 safe-to-fix events.
`packages/backend/scripts/fix-legacy-event-order.ts` (dry-run by default; `--apply` to write)
re-runs the audit classification, filters to `INVERTED-HIGH` + `INVERTED-STRUCTURAL`, and
applies the self-inverse transform `newOrder = (minOrder + maxOrder) − oldOrder` as a **single
set-based SQL UPDATE** over the target IDs. Safe because `orderOnCard` is in no unique key (the
Fight unique is `(eventId, fighter1Id, fighter2Id)`) and the transform is a bijection over each
event's order set.

Two snags before it took: (1) a transient P1017 "server closed the connection" on the first
findMany (Render PG cold connection) — failed before any write, retried; (2) P2010 42P01
`relation "Fight" does not exist` — the table is `@@map("fights")`, so the raw SQL needed
`"fights"` (columns stay camelCase `"eventId"`/`"orderOnCard"` — no column @map).

**Result:** 2491 fight rows across 225 events updated. Post-fix verification (re-classify the
touched events): **CORRECT=225, still-inverted=0.** Spot-checked Kattar vs Chikadze / Dos Anjos
vs Fiziev / Blaydes vs Aspinall — main event now at order 1, early prelims pushed to the top
orders. These 225 historic event pages now render right-side-up. The fix script is
re-run-safe (re-classifies live; the now-correct 225 won't be re-targeted).

**Still open** (not the rating heuristic — needs ufcstats/Wikipedia): 325 `INVERTED-RATING-ONLY`
+ 30 `SENTINEL`.
