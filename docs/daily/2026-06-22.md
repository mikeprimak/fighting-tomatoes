## BACKLOG ¬ß3 + ¬ß4 ‚ÄĒ Profile images (durable R2 + mobile picker + web persist)

Branch `backlog/round-3-profile-images` ‚Üí merged to `main` (merge in `d478af67`).

**Root cause of the web "broken image" bug.** `POST /upload/profile-image` wrote to
Render's **ephemeral local disk** (`public/uploads/profiles/`) and returned a **relative**
URL (`/uploads/profiles/<file>`). The browser resolved that against goodfights.app ‚Üí 404 ‚Üí
broken image. Two further problems: the file vanished on the next Render redeploy, and the
web edit page never **persisted** the avatar (only `setUser(...)` in memory).

**Fixes (one root fix + two clients):**
- **Backend** (`routes/upload.ts`) ‚ÄĒ profile uploads now go to **R2** via
  `uploadLocalFileToR2` and return the **absolute** public URL (durable, renders as-is on
  every client). Local-disk path kept as a dev-only fallback, now built absolute from the
  request host. `/upload/crew-image` left untouched (out of scope; its mobile client
  re-absolutizes the relative URL, so it'd double-prepend an absolute one).
- **Web** (`lib/api.ts` + `app/profile/edit/page.tsx`) ‚ÄĒ `updateProfile` accepts `avatar`;
  the edit page now persists the uploaded avatar to the user record + shows an uploading
  spinner + resets the file input so re-selecting the same file works.
- **Mobile** (`app/edit-profile.tsx`) ‚ÄĒ added the **missing** avatar UI: tap-to-pick
  (`expo-image-picker`, already shipped in the native build via `crew/info/[id].tsx` ‚Üí
  **OTA-safe**), upload ‚Üí `updateProfile({avatar})` ‚Üí `refreshUserData()`, spinner state,
  initial-letter fallback. Screen retitled "Edit Display Name" ‚Üí "Edit Profile".

Avatar URLs are now absolute everywhere ‚ÄĒ consistent with existing Google sign-in avatars
(which were already absolute `picture` URLs), and confirmed no client re-absolutizes
`avatar`, so nothing breaks.

**Verification:** backend prod tsconfig exit 0; web `tsc` exit 0; mobile 82 errors (under
the 84 baseline, none in the touched file). Not yet exercised live.

## BACKLOG ¬ß4 ‚ÄĒ Web Live-tab auto-update

`events/live/page.tsx` showed live event cards (each rendering `FightSectionList` with
live/up-next/completed results) but polled `['events','live']` at a flat **60s**, so results
lagged vs mobile. Changed `refetchInterval` to a function (RQ v5 `(query) => ...`): **15s
while any event is live** (matches `EventDetailClient`'s live cadence) and **60s otherwise**
to avoid hammering the backend when nothing's on. Cards re-render automatically since
`EventCard`/`FightSectionList` render from the polled `data.events`. Web `tsc` exit 0.
Web-only ‚ÄĒ deploys on the `main` push. (Kept the existing `['events','live']` query key
distinct from the red-dot poll per [[lesson_web_rq_key_collision_live_tab]].)
